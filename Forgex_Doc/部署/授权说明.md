# Forgex 授权说明

> 更新时间：**2026-04-16**

本文档用于说明 Forgex 在 `dev`、`test`、`yanshi`、`prod` 四类环境中的授权方式，重点回答两个问题：

- 开发环境现在如何授权
- 各个环境上线时应如何申请、导入、验证授权

## 一、先理解当前授权机制

Forgex 当前授权链路如下：

1. 系统启动后读取 `forgex.deployment.license-dir`
2. 如果目录下没有 `request-info.json`，系统会自动生成
3. 交付现场或开发环境把 `request-info.json` 提交给授权签发端
4. 授权签发端使用 `FxLicenseGenerator` 生成 `license.lic`
5. 现场将 `license.lic` 放回授权目录
6. 网关在访问受保护接口前校验授权状态
7. 未授权、失效、过期或不匹配时，登录页会提示“系统未授权，请先导入有效授权”

当前默认配置来自后端 `application.yml`：

- `forgex.deployment.instance-code`
- `forgex.deployment.home`
- `forgex.deployment.license-dir`
- `forgex.license.enabled`
- `forgex.license.public-key`
- `forgex.license.file-name`
- `forgex.license.request-info-file-name`

默认授权文件名：

- 授权文件：`license.lic`
- 请求文件：`request-info.json`
- 授权历史：`activation-history.json`

默认授权目录：

- 如果未显式配置：`./forgex/license`

## 二、开发环境如何授权

你现在是开发环境，建议按下面步骤操作。

### 1. 确认开发环境实例配置

开发环境建议至少显式配置下面几个环境变量：

```bash
FORGEX_PROFILE=dev
FORGEX_INSTANCE_CODE=DEV_LOCAL
FORGEX_HOME=D:/mine_product/forgex/runtime/dev
FORGEX_LICENSE_DIR=D:/mine_product/forgex/runtime/dev/license
FORGEX_LICENSE_ENABLED=true
FORGEX_LICENSE_PUBLIC_KEY=这里填写授权公钥
```

说明：

- `FORGEX_INSTANCE_CODE` 会参与生成 `customerCode`
- `FORGEX_LICENSE_DIR` 必须固定，否则换目录后机器码请求文件容易混淆
- `FORGEX_LICENSE_PUBLIC_KEY` 不能为空，否则即使拿到 `license.lic` 也无法通过签名校验
- 开发环境现在支持从 `license-dir/public-key.base64` 读取公钥，便于脚本分发后的本地兜底

### 2. 启动系统生成请求授权文件

先启动后端，至少保证网关或系统服务启动一次。系统会在授权目录自动生成：

```text
request-info.json
```

开发环境下如果你使用默认相对目录，文件一般会出现在：

```text
./forgex/license/request-info.json
```

更推荐直接固定到：

```text
D:/mine_product/forgex/runtime/dev/license/request-info.json
```

### 3. 获取开发环境请求信息

可以通过两种方式确认请求信息：

方式一：直接查看文件

- 打开 `request-info.json`
- 确认里面至少包含：
  - `product`
  - `edition`
  - `customerCode`
  - `machineCode`
  - `osType`
  - `hostname`
  - `generatedAt`

方式二：调用接口

- `GET /api/sys/license/request-info`
- 该接口已在网关授权拦截中放行，可在未登录时访问

### 4. 提交给授权签发端

把开发机生成的 `request-info.json` 提交给授权签发端，授权签发端使用：

- `Forgex_build/license-tools/issuer/FxLicenseGenerator/`

签发时必须保证以下信息一致：

- `customerCode` 与请求文件一致
- `machineCode` 与请求文件一致
- 产品是 `Forgex`
- 授权版本与模块范围符合开发环境用途

### 5. 导入开发环境授权

拿到 `license.lic` 后，放入开发环境授权目录：

```text
D:/mine_product/forgex/runtime/dev/license/license.lic
```

如果你使用的是请求授权客户端，也可以通过：

- `Forgex_build/license-tools/request-client/FxLicenseRequest/`

执行导入，把 `license.lic` 写入授权目录。

### 6. 刷新授权状态

导入后建议执行以下任一动作：

- 重启网关和系统服务
- 调用 `POST /api/sys/license/refresh`

### 7. 验证开发环境是否授权成功

按以下顺序检查：

1. 调用 `GET /api/sys/license/status`
2. 确认返回中的 `valid=true`
3. 确认 `status` 为以下之一：
   - `VALID`
   - `EXPIRING_SOON`
   - `GRACE`
4. 登录页面不再提示“系统未授权，请先导入有效授权”

如果还是未授权，重点检查：

- `FORGEX_LICENSE_PUBLIC_KEY` 是否已配置
- `license.lic` 是否放到了正确的 `license-dir`
- `license.lic` 的 `customerCode` 是否和 `request-info.json` 一致
- `license.lic` 的 `machineCode` 是否和当前机器一致
- 当前授权是否已过期
- `license.lic` 是否被文本编辑器或脚本写入了 UTF-8 BOM
  - 如果后端报错 `授权解析失败: Illegal base64 character 3f`
  - 优先检查 `license.lic` 文件头是否包含 BOM
  - 当前授权工具已调整为无 BOM UTF-8 输出，如遇到旧文件，重新执行 `activate-dev-license.bat` 生成即可
- `public-key.base64` 是否存在于当前 `license-dir`
  - 如果后端报错 `未配置 Ed25519 公钥`
  - 先确认授权目录下存在 `public-key.base64`
  - 再重启 Gateway 和 Sys 服务，让进程重新加载授权目录

### 8. 开发公司统一分发 bat 的推荐做法

为了避免开发环境一台一台手工申请授权，当前已经提供开发环境专用脚本：

```text
Forgex_build/license-tools/activate-dev-license.bat
```

推荐分发方式：

1. 把整个 `Forgex_build/license-tools/` 目录发给开发公司
2. 要求开发机先安装 `.NET SDK 9`
3. 要求开发机启动过一次 Forgex 后端，先让系统生成本机 `request-info.json`
4. 要求开发机确认环境变量 `FORGEX_PROFILE=dev`
5. 直接双击或命令行执行 `activate-dev-license.bat`

脚本行为说明：

- 只在 `FORGEX_PROFILE=dev` 时执行
- 不支持通过命令行参数伪装成 `dev`
- 如果本机已经存在 `request-info.json`，脚本会优先复用，避免重新生成后出现机器码不一致
- 脚本会自动完成以下动作：
  - 编译独立授权工具
  - 首次生成本地开发密钥
  - 生成或复用 `request-info.json`
  - 签发开发专用 `license.lic`
  - 导入到当前开发机授权目录
  - 复制 `public-key.base64` 到当前开发机授权目录
  - 写入当前用户环境变量：
    - `FORGEX_PROFILE=dev`
    - `FORGEX_INSTANCE_CODE`
    - `FORGEX_LICENSE_ENABLED=true`
    - `FORGEX_LICENSE_DIR`
    - `FORGEX_LICENSE_PUBLIC_KEY`
    - `FORGEX_LICENSE_PUBLIC_KEY_FILE_NAME=public-key.base64`

开发专用限制：

- 该脚本生成的是 `edition=dev` 的开发授权
- 授权备注固定为 `DEV_ONLY_LOCAL_LICENSE`
- 后端运行时会再次校验：
  - 只有 `FORGEX_PROFILE=dev` 时，该开发授权才视为有效
  - 如果把这个授权文件拷贝到 `test`、`yanshi`、`prod`，系统会判定为无效授权

因此，开发环境可以通过统一分发 bat 一键授权，但测试、演示、生产环境仍然必须走正式授权流程，不能复用开发授权。

## 三、测试环境授权步骤

测试环境对应 `FORGEX_PROFILE=test`，建议步骤如下：

1. 固定测试环境实例编码，例如 `TEST_A`、`TEST_UAT`
2. 固定授权目录为绝对路径
3. 首次启动生成 `request-info.json`
4. 将 `request-info.json` 提交给授权签发端
5. 签发测试环境专用 `license.lic`
6. 放入测试环境 `license-dir`
7. 调用 `POST /api/sys/license/refresh`
8. 通过 `GET /api/sys/license/status` 验证

建议测试环境配置：

```bash
FORGEX_PROFILE=test
FORGEX_INSTANCE_CODE=TEST_UAT
FORGEX_HOME=/opt/forgex/test_uat
FORGEX_LICENSE_DIR=/opt/forgex/test_uat/license
FORGEX_LICENSE_ENABLED=true
FORGEX_LICENSE_PUBLIC_KEY=这里填写授权公钥
```

测试环境注意事项：

- 不要直接复用开发环境授权
- 不要复制别的测试机上的 `request-info.json`
- 同一个测试实例改机器后，要重新申请授权

## 四、演示环境授权步骤

演示环境对应 `FORGEX_PROFILE=yanshi`，建议单独申请演示授权，不要混用生产授权。

步骤：

1. 设置 `FORGEX_PROFILE=yanshi`
2. 设置单独的实例编码，例如 `DEMO_SZ`、`YANSHI_001`
3. 启动系统生成 `request-info.json`
4. 提交到授权签发端生成演示授权
5. 将 `license.lic` 放入演示环境授权目录
6. 刷新授权并验证状态

建议演示环境重点关注：

- 到期时间是否覆盖演示周期
- 是否只开放演示所需模块
- 机器更换后是否需要重新签发

## 五、生产环境授权步骤

生产环境对应 `FORGEX_PROFILE=prod`，这是最严格的一类环境，建议按正式交付流程执行。

步骤：

1. 先确定生产实例编码
2. 固定安装目录和授权目录
3. 首次部署后启动服务生成 `request-info.json`
4. 将请求文件提交到授权签发端
5. 使用 `FxLicenseGenerator` 生成正式生产授权
6. 将 `license.lic` 放入生产授权目录
7. 调用 `POST /api/sys/license/refresh` 或重启服务
8. 使用 `GET /api/sys/license/status` 验证
9. 验证登录、菜单、业务接口是否恢复正常

建议生产环境配置：

```bash
FORGEX_PROFILE=prod
FORGEX_INSTANCE_CODE=ACME_PROD
FORGEX_HOME=/opt/Forgex_ACME_PROD
FORGEX_LICENSE_DIR=/opt/Forgex_ACME_PROD/license
FORGEX_LICENSE_ENABLED=true
FORGEX_LICENSE_PUBLIC_KEY=这里填写授权公钥
```

生产环境注意事项：

- 生产授权不要和演示、测试环境混用
- 生产机器迁移后必须重新申请授权
- 变更 `instanceCode` 后建议重新申请授权
- 授权目录应纳入备份，但不要把私钥工具下发到生产机

## 六、各环境推荐做法对比

| 环境 | Profile | 是否必须授权 | 推荐目录 | 是否允许临时调试 |
|---|---|---|---|---|
| 开发 | `dev` | 建议授权 | 本地固定绝对路径 | 可以，但不建议长期关闭授权 |
| 测试 | `test` | 必须授权 | 独立测试目录 | 不建议 |
| 演示 | `yanshi` | 必须授权 | 独立演示目录 | 不建议 |
| 生产 | `prod` | 必须授权 | 正式安装目录 | 不允许 |

## 七、现场排查顺序

当系统提示“系统未授权，请先导入有效授权”时，按下面顺序排查：

1. 调 `GET /api/sys/license/status`
2. 看 `status` 和 `message`
3. 看授权目录里是否存在 `request-info.json`
4. 看授权目录里是否存在 `license.lic`
5. 对比 `customerCode` 是否一致
6. 对比 `machineCode` 是否一致
7. 确认 `FORGEX_LICENSE_PUBLIC_KEY` 是否配置
8. 执行 `POST /api/sys/license/refresh`
9. 仍失败时重启网关与系统服务

## 八、与交付目录的对应关系

`Forgex_build` 已按用途拆分，现场使用时请对应取用：

- Windows 安装软件：
  - `Forgex_build/delivery/windows/installer/`
- Linux 安装脚本：
  - `Forgex_build/delivery/linux/scripts/`
- 授权签发软件：
  - `Forgex_build/license-tools/issuer/`
- 请求授权客户端软件：
  - `Forgex_build/license-tools/request-client/`

这样区分后：

- 交付现场拿 Windows 或 Linux 的部署材料
- 授权中心拿 `issuer`
- 客户现场申请授权拿 `request-client`

## 九、当前结论

你现在的开发环境如果要授权，最稳妥的做法是：

1. 固定 `FORGEX_INSTANCE_CODE=DEV_LOCAL`
2. 固定 `FORGEX_LICENSE_DIR`
3. 启动一次系统生成 `request-info.json`
4. 用该文件申请开发授权
5. 把返回的 `license.lic` 放到授权目录
6. 调 `POST /api/sys/license/refresh`
7. 用 `GET /api/sys/license/status` 验证

这样处理后，登录页的未授权提示就会消失。
