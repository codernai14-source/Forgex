# 部署软件实现逻辑与二开说明

> 本文说明 Forgex 部署交付软件的编译工具、技术栈、当前实现逻辑和二次开发入口。本文只描述部署交付工程，不替代业务前端、后端功能文档。

## 一、部署软件范围

Forgex 部署软件由 `Forgex_Build` 统一维护，覆盖以下交付能力：

| 软件/脚本 | 位置 | 作用 |
|---|---|---|
| Windows 安装器 | `Forgex_Build/delivery/windows/installer/ForgexSetup.iss` | 生成客户可双击安装的 `Forgex-Setup-{Version}.exe` |
| Windows 控制中心 | `Forgex_Build/tools/control-center/ForgexControlCenter` | 图形化授权申请/导入、服务启停、前端启动、日志与异常查看 |
| 授权请求客户端 | `Forgex_Build/license-tools/request-client/FxLicenseRequest` | 现场生成 `request-info.json` 并导入 `license.lic` |
| 授权签发端 | `Forgex_Build/license-tools/issuer/FxLicenseIssuer` | 授权签发人员使用的图形化 `license.lic` 生成工具 |
| 授权核心库 | `Forgex_Build/license-tools/FxLicenseCore` | 机器指纹、请求文件、授权签发、授权导入、密钥处理的公共实现 |
| Windows 安装脚本 | `Forgex_Build/delivery/windows/installer/scripts` | 安装初始化、Nacos 导入、数据库导入、运行配置修复、卸载清理 |
| WinSW 服务模板 | `Forgex_Build/delivery/windows/installer/winsw/winsw-service.xml.template` | 生成 Windows 服务包装 XML |
| Linux 部署脚本 | `Forgex_Build/delivery/linux/scripts` | Linux 离线包安装、升级、回滚、备份、恢复骨架 |
| 构建收集脚本 | `Forgex_Build/collect-artifacts.ps1`、`build-*.ps1` | 收集前端、后端、JRE、Nacos、SQL、工具并输出交付包 |

## 二、编译工具与技术栈

| 部署软件 | 编译/打包工具 | 主要技术 | 输出产物 |
|---|---|---|---|
| Windows 安装器 | Inno Setup 6，命令行编译器 `ISCC.exe` | Inno Setup Script、Pascal Script、PowerShell 调用 | `dist/windows/Forgex-Setup-{Version}.exe` |
| Windows 控制中心 | .NET SDK 9，`dotnet publish` | C#、.NET 9、Windows Forms、`System.ServiceProcess.ServiceController` | `tools/control-center/ForgexControlCenter.exe` |
| 授权签发端 | .NET SDK 9，`dotnet publish` | C#、.NET 9、Windows Forms、`FxLicenseCore` | `license-tools/issuer/publish/win-x64/FxLicenseIssuer.exe` |
| 授权请求客户端 | .NET SDK 9，`dotnet publish` | C#、.NET 9 Console、`FxLicenseCore` | Windows/Linux 请求授权客户端 |
| 授权核心库 | .NET SDK 9 | C#、`BouncyCastle.Cryptography`、JSON 文件读写 | 被控制中心、请求客户端、签发端引用 |
| Windows 安装/导入/修复脚本 | Windows PowerShell | 文件复制、JSON/YAML 文本更新、Nacos REST、`mysql.exe` 调用、WinSW XML 修复 | 安装目录 `scripts/*.ps1` / `*.bat` |
| Windows 服务包装 | WinSW | XML 服务描述、JVM 参数、环境变量、日志路径 | `services/wrappers/*.exe` / `*.xml` |
| Windows 前端托管 | Nginx 或控制中心内置静态 Web 服务 | Windows Nginx、C# `HttpListener` fallback | `nginx/forgex.conf`、前端访问端口 |
| Linux 包 | PowerShell、Python `tarfile` | Shell、Docker Compose 模板、tar.gz 权限处理 | `dist/linux/forgex-linux-bundle-{Version}.tar.gz` |

构建环境通常需要：

1. `.NET SDK 9`：编译控制中心、授权签发端、授权请求客户端。
2. `Inno Setup 6`：编译 Windows 安装器，命令行入口是 `ISCC.exe`。
3. `Node.js/npm`：构建 `Forgex_MOM/Forgex_Fronted` 前端静态资源。
4. `JDK/Maven`：构建 `Forgex_MOM/Forgex_Backend` 后端服务 JAR。
5. `Windows JRE/JDK`：作为客户 Windows 包内置 JRE 复制到 `app/jre/`。
6. `Python`：生成 Linux `tar.gz` 时保留可执行权限。
7. `mysql.exe`：服务器现场执行 `import-database.bat` 时需要 MySQL 客户端。

## 三、构建命令

后端 JAR 需要先在后端工程中生成：

```powershell
cd D:\mine_product\forgex\Forgex_MOM\Forgex_Backend
mvn -DskipTests package
```

全量收集、打包 Windows/Linux 交付物：

```powershell
cd D:\mine_product\forgex\Forgex_Build
powershell -ExecutionPolicy Bypass -File build-all.ps1 -Version 1.0.0 -WindowsJreHome "C:\Program Files\Java\jdk-17" -CompileWindowsInstaller
```

仅重新编译 Windows 安装器：

```powershell
& "D:\APP\Inno Setup 6\ISCC.exe" "D:\mine_product\forgex\Forgex_Build\delivery\windows\installer\ForgexSetup.iss"
```

仅发布控制中心：

```powershell
dotnet publish D:\mine_product\forgex\Forgex_Build\tools\control-center\ForgexControlCenter\ForgexControlCenter.csproj -c Release -r win-x64 --self-contained true -p:PublishSingleFile=true -p:IncludeNativeLibrariesForSelfExtract=true
```

仅发布授权签发端：

```powershell
powershell -ExecutionPolicy Bypass -File D:\mine_product\forgex\Forgex_Build\license-tools\publish-license-issuer.ps1
```

## 四、当前实现逻辑

### 4.1 交付物收集

`collect-artifacts.ps1` 是统一收集入口，执行逻辑如下：

1. 清空并重建 `staging/windows`、`staging/linux`。
2. 在前端工程执行 `npm ci`（必要时）和 `npm run build`，将 `dist/` 复制到 `frontend/`。
3. 从后端各模块 `target/` 目录收集最新业务 JAR 到 `services/`，模块包括 `Gateway`、`Auth`、`Sys`、`Basic`、`Job`、`Integration`、`Workflow`、`Report`。
4. 复制公共配置模板、授权公钥、Nacos 配置、数据库初始化 SQL、Nginx 模板和 Windows Nginx 运行时。
5. 解析本机 Java/JRE 目录，将 Windows JRE 复制到 `staging/windows/app/jre`。
6. 发布授权请求客户端到 Windows/Linux 交付目录。
7. 发布 Windows 控制中心到 `tools/control-center`。
8. 复制 Windows 安装脚本、WinSW 模板和 Linux 部署脚本。

如果后端 JAR 不完整，脚本默认失败；只有显式使用 `-AllowDistFallback` 时才会从历史交付包复用 JAR。

### 4.2 Windows 包与安装器

`build-windows.ps1` 负责把 `staging/windows` 组装成最终 Windows 交付包：

1. 生成 `dist/windows/Forgex-Windows-Package-{Version}/`。
2. 复制 staging 内容，并覆盖最新 `scripts/`、`winsw/`、Nginx 运行时和 Inno Setup 脚本。
3. 将安装器脚本中的本地路径改写为交付包相对路径。
4. 生成 `Forgex-Windows-Package-{Version}.zip`。
5. 如果传入 `-CompileInstaller`，调用 `ISCC.exe` 编译 `Forgex-Setup-{Version}.exe`。

`ForgexSetup.iss` 的安装逻辑：

1. 显示实例编码、部署环境、运行目录、中间件地址等安装页面。
2. 默认部署环境为 `yanshi`，Nacos namespace 默认跟随部署环境。
3. 安装前执行临时 `uninstall-cleanup.ps1`，尝试停止当前安装目录关联进程并清理旧 `app/jre`。
4. 复制交付文件到 `{app}`。
5. 执行 `scripts/install.ps1`，生成目录结构、`config/forgex-control.json`、`config/install-config.yml`、`nginx/forgex.conf` 和服务配置。
6. 如果安装目录中存在 WinSW 可执行文件和模板，则注册 Windows 服务；否则控制中心仍可使用进程模式启动后端 JAR。
7. 卸载时执行 `scripts/uninstall-cleanup.ps1`，停止服务、卸载 WinSW、删除服务残留并清理 `app/jre`。

### 4.3 控制中心

控制中心是 C#/.NET 9 Windows Forms 单文件程序，主实现集中在 `Program.cs`：

| 类/模块 | 职责 |
|---|---|
| `MainForm` | 主界面、按钮事件、语言切换、授权申请/导入、服务启停入口 |
| `ForgexServiceManager` | 后端服务状态检测、Windows 服务/进程模式启动停止、启动前目录与 XML 修复 |
| `FrontendWebServer` | 前端服务入口，优先使用 Nginx，不满足条件时回退到内置静态服务 |
| `NginxFrontendServer` | 启动/停止安装目录中的 `nginx.exe` 并加载 `nginx/forgex.conf` |
| `ForegroundStaticWebServer` | 内置静态 Web 服务 fallback |
| `ControlCenterDiagnostics` | 控制中心自身异常写入 `logs/control-center/control-center-error.log` |
| `ForgexControlConfig` | 读取安装生成的 `config/forgex-control.json` |

服务启动时，控制中心会先创建日志目录，再修复旧 WinSW XML 中的关键参数，包括：

- `--spring.profiles.active`
- `-Dforgex.deployment.log-dir`
- `-DLOG_DIR`
- `-Dlogging.file.path`
- `FORGEX_PROFILE`
- `FORGEX_NACOS_NAMESPACE`
- `FORGEX_DATASOURCE_CONFIG`

这样旧安装器残留的 `prod` 或 `D:\forgex\log` 不会继续污染服务启动。

### 4.4 数据库和 Nacos 一键导入

`import-database.bat` 调用 `import-database.ps1`：

1. 默认读取安装目录 `database-init/*.sql`。
2. 优先从 `config/forgex-control.json` 和 Nacos 数据源配置推断 MySQL 地址、账号、密码。
3. 按 SQL 文件名创建数据库，例如 `forgex_admin.sql` 对应 `forgex_admin`。
4. 默认只在目标库为空时导入；如需重建库，显式追加 `-ResetDatabase`。

`import-nacos-config.bat` 调用 `import-nacos-config.ps1`：

1. 默认读取安装配置中的 Nacos 地址、namespace、group。
2. 不存在 namespace 时自动创建。
3. 将 `nacos/DEFAULT_GROUP` 下的 `.yml`、`.yaml`、`.properties` 文件通过 Nacos REST API 发布为配置。

### 4.5 Linux 交付包

Linux 包当前是离线部署骨架：

1. `collect-artifacts.ps1` 将前端、后端 JAR、Nacos 配置、数据库脚本、授权请求客户端和 Linux 脚本收集到 `staging/linux`。
2. `build-linux.ps1` 生成 zip，并使用 Python `tarfile` 生成 `tar.gz`。
3. Python 打包时显式设置 `install.sh`、`upgrade.sh`、`rollback.sh`、`backup.sh`、`restore.sh` 和 Linux 授权请求客户端为可执行权限。
4. `delivery/linux/scripts/docker-compose.yml.template` 和 `.env.template` 是 Linux 容器化部署的主要模板。

## 六、二开后的验证清单

部署软件二开后至少验证：

1. `dotnet publish` 控制中心成功。
2. `dotnet publish` 授权请求客户端/签发端成功。
3. `mvn -DskipTests package` 后端 JAR 成功。
4. `npm run build` 前端成功。
5. `collect-artifacts.ps1` 收集成功。
6. `build-windows.ps1 -CompileInstaller` 或 `ISCC.exe ForgexSetup.iss` 成功。
7. Windows 安装器可完成安装，生成 `config/forgex-control.json`。
8. `repair-runtime-config.bat` 可修复 profile、namespace、日志目录和 WinSW XML。
9. `import-database.bat`、`import-nacos-config.bat` 可在测试环境成功导入。
10. 控制中心启动服务失败时不会退出，并能在日志中看到失败原因。

## 七、注意事项

1. 文档版本字段不因普通部署脚本修复而自动变更，除非明确进行文档版本发布。
2. 生产环境密码、授权私钥不得写入交付文档或客户包。
3. 客户 Windows 安装器只开放 `prod` 和 `yanshi`，`dev`、`test` 属于内部环境。
4. 日志目录不能依赖数据库配置，必须通过安装配置、环境变量或 JVM 参数传入，因为日志系统在数据库连接前初始化。
5. 安装器、控制中心、修复脚本对运行参数有重叠兜底，二开时要同步维护，避免只改其中一处导致现场残留配置继续生效。

## 八、控制中心单服务运维能力

Windows 控制中心支持在服务列表中选中某一个后端服务或 `web` 前端服务，并执行“启动选中服务”“停止选中服务”“重启选中服务”“打开选中日志目录”。这用于现场只替换某一个服务 JAR 后的局部更新流程，避免每次都停止整套系统。

单服务更新建议流程：

1. 在控制中心服务列表选中目标服务。
2. 点击“停止选中服务”。
3. 替换 `services/` 目录中对应服务 JAR。
4. 点击“启动选中服务”或“重启选中服务”。
5. 点击“打开选中日志目录”查看该服务的 `stdout.log`、`stderr.log` 或业务日志。

控制中心命令行也支持单服务操作：

```powershell
ForgexControlCenter.exe --install-root C:\Forgex_ACME_PROD --start-service basic
ForgexControlCenter.exe --install-root C:\Forgex_ACME_PROD --stop-service basic
ForgexControlCenter.exe --install-root C:\Forgex_ACME_PROD --restart-service basic
```

`serviceId` 使用安装配置中的服务标识，例如 `auth`、`sys`、`basic`、`job`、`workflow`、`integration`、`report`、`gateway`，前端服务使用 `web`。
## 九、前后端轻量升级程序

Windows 交付包包含 `scripts/upgrade.ps1` 和 `scripts/upgrade.bat`，用于只升级前端静态资源和后端 JAR，不重新安装整套程序，不重新导入数据库或 Nacos 配置。

推荐现场流程：

```powershell
powershell -NoProfile -ExecutionPolicy Bypass -File C:\Forgex_ACME_PROD\scripts\upgrade.ps1 -InstallRoot C:\Forgex_ACME_PROD -PackageRoot D:\Forgex-Windows-Package-1.0.1
```

升级包根目录只需要包含需要替换的目录，例如 `services/`、`frontend/`，也可以附带 `tools/control-center/`、`nginx/`、`winsw/`、`scripts/`。脚本会先调用控制中心停止全部服务，备份当前 `services` 和 `frontend` 到 `backup/upgrade-yyyyMMddHHmmss`，复制新文件，更新 `config/forgex-control.json` 里的 JAR 路径和 WinSW XML 中的 `-jar` 参数，然后再启动全部服务。

常用参数：

- `-SkipStop`：不自动停止服务，适合人工已停服务的场景。
- `-SkipStart`：升级后不自动启动，适合升级后还要检查配置的场景。
- `-NoBackup`：不生成升级备份，仅在已有外部备份时使用。

## 十、服务日志命名规则

后端各服务的业务日志统一写入安装日志根目录下的模块子目录：

- `logs/auth/auth-info.yyyy-MM-dd.i.log`、`logs/auth/auth-error.yyyy-MM-dd.i.log`
- `logs/sys/sys-info.yyyy-MM-dd.i.log`、`logs/sys/sys-error.yyyy-MM-dd.i.log`
- 其他模块同理：`basic`、`job`、`workflow`、`integration`、`report`、`gateway`、`common`

`info` 文件只记录 `INFO/WARN` 等非错误日志，`error` 文件只记录 `ERROR`。单个业务日志文件最大 `2GB`，按日期和序号切分。WinSW 或控制中心进程模式产生的 `stdout.log`、`stderr.log` 仍保留在对应模块目录中，用于排查启动器和标准输出问题。
