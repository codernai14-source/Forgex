# 多语言输入实现逻辑

> 分类：前端 / 国际化与布局  
> 版本：**V0.6.0**  
> 更新时间：**2026-04-17**

本文档说明 Forgex 前端多语言输入相关公共组件的实现方式，重点覆盖：

- `I18nInput.vue`
- `I18nJsonEditor.vue`

## 功能定位

### I18nInput

`I18nInput.vue` 是主力组件，负责把一个多语言 JSON 字符串做成可编辑控件。

它支持：

- 简单输入模式
- 表格模式
- 弹窗编辑模式
- 文本框与文本域
- 占位符工具栏
- 动态加载启用语言列表

### I18nJsonEditor

`I18nJsonEditor.vue` 是更轻量的双语 JSON 编辑器，主要适合：

- 简单中英文字段
- 调试场景
- 不需要动态语言列表的简单配置页

## I18nInput 的核心设计

它对外绑定的是一个字符串：

```ts
modelValue?: string
```

字符串内容通常长这样：

```json
{
  "zh-CN": "菜单名称",
  "en-US": "Menu Name"
}
```

也就是说，页面里保存的不是对象，而是 JSON 字符串。

这样设计的好处是：

- 可以直接和后端字段对齐
- 表单提交时不需要额外再序列化
- 与数据库里常见的 `*_i18n_json` 字段模型一致

## I18nInput 的内部数据流

内部核心流程如下：

```text
父组件传入 modelValue(JSON 字符串)
  -> parseI18nJson()
  -> 根据启用语言生成 tableData
  -> 用户编辑某一行或默认语言
  -> generateI18nJson()
  -> emit('update:modelValue', jsonString)
```

这意味着：

- 页面永远拿到的是最新 JSON 字符串
- 组件内部才维护可编辑的结构化行数据

## 动态语言加载逻辑

`I18nInput` 不是写死只支持中英文。  
它挂载后会调用系统启用语言接口，拿到当前系统启用语言列表，再动态生成编辑表格。

因此它更适合真正的国际化后台，而不是简单的双语表单。

## 模式设计

`I18nInput` 主要支持三种模式：

### simple

页面默认看到一个输入框或文本域，只展示默认语言值。  
点击地球图标后弹出完整多语言编辑弹窗。

适合：

- 菜单名称
- 按钮文案
- 普通模板标题

### table

页面直接展示所有语言的表格编辑区。

适合：

- 专门的多语言维护页面
- 需要一次性同时编辑多种语言的场景

### modal

当前代码主要还是通过 `simple + modal` 的组合来使用，核心还是弹窗编辑完整语言内容。

## 默认语言同步逻辑

在 `simple` 模式下，组件表面上只展示默认语言值。  
用户直接在输入框里输入时，组件会同步更新：

- `defaultValue`
- `tableData` 中默认语言那一行
- 最终输出 JSON

因此简单输入模式不是“只改一份文本”，而是在改完整多语言数据中的默认语言。

## 占位符工具栏逻辑

`I18nInput` 支持消息模板类场景常见的占位符插入，例如：

- `${userName}`
- `${userAccount}`
- `${tenantName}`
- `${currentTime}`

实现方式是：

- 内部维护 `builtinPlaceholders`
- 在文本域或弹窗侧栏中点击占位符按钮
- 将占位符插入当前光标位置

这套能力主要为消息模板正文配置服务。

## I18nJsonEditor 的实现思路

`I18nJsonEditor.vue` 走的是更轻量的方案。

它的特点是：

- 固定只编辑 `zh-CN` 和 `en-US`
- 实时生成 JSON 预览
- 结构简单，适合低复杂度表单

如果页面不需要动态语言列表，也不需要弹窗编辑能力，用它会更轻。

## 实际页面中的使用方式

### 菜单页面

菜单名称使用 `I18nInput`，原因是菜单天然需要国际化，而且编辑时只希望看到一个简洁输入框。

### 国际化消息页面

消息内容本身就是国际化对象，适合用 `I18nInput` 或 `I18nJsonEditor` 直接维护完整 JSON。

### 消息模板页面

模板标题和正文需要：

- 多语言
- 占位符
- 文本域

所以这里更适合 `I18nInput`。

## 为什么这样设计

### 1. 与后端字段模型直接对齐

后端很多字段本来就是 `xxxI18nJson` 字符串，前端直接围绕这个模型工作，成本最低。

### 2. 兼顾简单输入与完整编辑

大多数页面不希望一直展开一整张多语言表，所以 `simple + modal` 是更平衡的方案。

### 3. 支持语言动态扩展

以后系统启用更多语言时，`I18nInput` 不需要大改页面代码。

## 常见排查

### 页面显示为空但后端有值

优先检查：

1. `modelValue` 是否真的是合法 JSON 字符串
2. 默认语言键是否存在

### 编辑后提交的是空字符串

优先检查：

1. 是否所有语言值都为空
2. 外层表单是否把 `update:modelValue` 的结果覆盖掉了

### 占位符按钮没效果

优先检查：

1. 是否开启了 `showPlaceholders`
2. 当前是否是支持插入的文本域模式
