# 公共弹窗实现逻辑

> 分类：前端 / 请求与反馈  
> 版本：**V0.6.0**  
> 更新时间：**2026-04-17**

本文档说明通用表单弹层 `BaseFormDialog.vue` 的实现思路、代码位置和事件模型。

## 功能定位

`BaseFormDialog.vue` 是 Forgex 前端统一表单弹层容器，目标是把“新增 / 编辑 / 详情”这类高频弹层场景的外壳统一掉。

它不处理具体业务字段，只处理这些通用能力：

- `Modal` / `Drawer` 切换
- 标题、宽度、关闭方式
- 提交按钮和取消按钮
- 提交 loading
- 关闭销毁
- 表单模式统一

## 核心实现结构

组件内部支持两种显示模式：

```text
Modal 模式
Drawer 模式
```

实现上通过：

```ts
const finalMode = computed(() => props.mode || appStore.formMode)
```

这表示模式选择顺序是：

1. 页面显式传入的 `props.mode`
2. 全局 `appStore.formMode`

## 组件职责边界

### 它负责什么

- 渲染弹层壳子
- 统一按钮区
- 统一触发 `submit` / `cancel`
- 统一处理 `v-model:open`

### 它不负责什么

- 不负责表单校验
- 不负责接口调用
- 不负责组装业务参数
- 不负责刷新列表

这些都交给业务页面或业务弹窗组件自己处理。

## 事件模型

组件对外抛出的主要事件：

| 事件 | 作用 |
|---|---|
| `update:open` | 控制打开关闭 |
| `submit` | 点击确认时触发 |
| `cancel` | 点击取消或关闭时触发 |
| `ok` | 兼容部分旧组件监听模式 |

这里的设计重点是“容器只告诉外部发生了提交动作”，而不直接侵入业务逻辑。

## 关闭与销毁

组件支持：

```ts
destroyOnClose
```

默认开启关闭销毁，原因是：

- 弹窗再次打开时表单状态更干净
- 避免旧页面数据残留
- 复杂表单切换场景更稳定

组件还支持：

```ts
maskClosable
```

默认值为 `true`，表示点击遮罩会关闭弹窗。该默认值用于统一管理页新增、编辑类弹窗的交互体验。若某个页面存在长流程或不可误关场景，应在业务调用处传入 `:mask-closable="false"` 显式关闭遮罩点击关闭能力。

## 为什么这样设计

### 1. 页面交互统一

所有管理页的弹层壳子、按钮区、关闭方式保持一致。

### 2. 减少重复开发

各个业务模块不再重复实现一套自己的 Modal / Drawer 外壳。

### 3. 支持全局 UI 策略

项目可通过 `appStore.formMode` 统一切换表单偏好为 `modal` 或 `drawer`。

## 典型调用链路

```text
列表页面
  -> 打开业务弹窗组件
  -> 业务弹窗内部使用 BaseFormDialog
  -> 用户点击确定
  -> BaseFormDialog 抛出 submit
  -> 业务弹窗执行表单校验和保存
  -> 页面关闭弹窗并刷新列表
```

## 常见排查

| 现象 | 优先检查 |
|---|---|
| 点击确定没有反应 | 是否监听了 `submit` |
| 弹窗关闭后数据残留 | `destroyOnClose` 是否被关闭 |
| Drawer / Modal 模式不对 | `props.mode` 和 `appStore.formMode` 谁生效 |
