# 公共弹窗

> 分类：前端 / 请求与反馈  
> 版本：**V0.5.0**  
> 更新时间：**2026-04-13**

## 一、概述

Forgex 前端统一表单弹窗组件位于：

- `Forgex_MOM/Forgex_Fronted/src/components/common/BaseFormDialog.vue`

它负责把系统内大量“新增 / 编辑 / 详情”弹窗统一成同一套交互模型，避免每个页面重复维护：

- `Modal` 与 `Drawer` 的切换逻辑
- 确认 / 取消按钮
- 提交 loading
- 关闭时销毁内容
- 国际化按钮文案

---

## 二、组件定位

`BaseFormDialog` 本质上是一个**容器组件**：

1. 自己不关心具体表单字段；
2. 负责统一外层弹窗形态；
3. 通过插槽承载内部表单内容；
4. 通过事件把提交、取消动作抛给业务页面。

常见使用位置：

- 用户编辑弹窗
- 角色编辑弹窗
- 租户配置弹窗
- 工作流任务配置弹窗
- 通用管理页的新增 / 编辑表单

---

## 三、支持的显示模式

### 3.1 `modal`

使用 `a-modal` 渲染，适合：

- 字段较少的标准表单
- 需要聚焦操作的小型弹层
- 不希望占满右侧空间的场景

### 3.2 `drawer`

使用 `a-drawer` 渲染，适合：

- 表单字段较多
- 需要更大编辑空间
- 需要保留页面上下文的场景

### 3.3 模式选择规则

组件最终模式的计算方式是：

```text
props.mode 优先
  ↓
未传时回退到 appStore.formMode
```

因此系统可以做到：

- 某些页面显式指定总是弹 `modal`
- 某些页面跟随全局偏好切换 `modal/drawer`

## 四、Props 说明

| 属性 | 类型 | 默认值 | 说明 |
|---|---|---|---|
| `open` | `boolean` | `false` | 是否打开弹窗 |
| `title` | `string` | `''` | 弹窗标题 |
| `mode` | `'modal' \| 'drawer'` | `undefined` | 显式指定显示模式 |
| `width` | `number \| string` | `600` | 弹窗宽度 |
| `loading` | `boolean` | `false` | 提交中状态 |
| `destroyOnClose` | `boolean` | `true` | 关闭时销毁内部内容 |

### 4.1 重点说明

- `loading` 只控制确认按钮 loading，不会自动提交；
- `destroyOnClose: true` 适合大多数表单，可避免旧状态残留；
- 详情页或需要保留未提交内容时，可按需关闭销毁。

---

## 五、事件说明

| 事件 | 说明 |
|---|---|
| `update:open` | 更新显示状态，通常用于 `v-model:open` |
| `submit` | 点击确认按钮时触发 |
| `cancel` | 点击取消 / 关闭时触发 |
| `ok` | 与 `submit` 同时触发，兼容部分旧写法 |

### 5.1 提交逻辑

组件本身不会校验表单，也不会自动调用接口。标准流程是：

```text
BaseFormDialog 点击确认
  → emit('submit') / emit('ok')
  → 业务组件校验表单
  → 调接口保存
  → 成功后关闭弹窗
```

### 5.2 取消逻辑

点击取消时，组件会先：

1. 触发 `update:open = false`
2. 再触发 `cancel`

因此业务层一般只需要维护：

```vue
<BaseFormDialog v-model:open="open" @submit="handleSubmit" />
```

## 六、推荐使用方式

### 6.1 标准写法

```vue
<BaseFormDialog
  v-model:open="open"
  :title="form.id ? '编辑用户' : '新增用户'"
  :loading="submitting"
  width="720"
  @submit="handleSubmit"
>
  <a-form :model="form">
	<!-- 表单内容 -->
  </a-form>
</BaseFormDialog>
```

### 6.2 脚本侧建议

```ts
async function handleSubmit() {
  await formRef.value?.validate()
  submitting.value = true
  try {
	await saveUser(form.value)
	open.value = false
	await reload()
  } finally {
	submitting.value = false
  }
}
```

这里推荐搭配统一 HTTP 层使用，让成功提示由后端 + `http.ts` 自动托管。

## 七、适用场景建议

| 场景 | 建议 |
|---|---|
| 标准管理页新增/编辑 | 优先使用 `BaseFormDialog` |
| 表单字段较少 | `modal` |
| 表单复杂、字段多 | `drawer` |
| 需要全局切换交互风格 | 不传 `mode`，跟随 `appStore.formMode` |
| 页面需要完全自定义 footer | 再单独写自定义弹层 |

## 八、与其他公共能力的组合

### 8.1 与 `http.ts`

推荐组合：

- 提交时调用 `http` / `httpSuccess`
- 成功消息交给后端返回 `message`
- 页面只处理表单校验、关闭弹窗、刷新列表

### 8.2 与 `FxDynamicTable`

标准管理页常见组合是：

```text
FxDynamicTable 列表页
  → 工具栏点击新增 / 编辑
  → BaseFormDialog 打开
  → 保存成功后 reload 表格
```

### 8.3 与 `I18nInput`

需要录入多语言标题、名称、模板内容时，可直接在弹窗表单内嵌：

- `I18nInput`
- `I18nJsonEditor`

---

## 九、最佳实践

1. **优先让弹窗只负责容器职责，不把接口逻辑塞进公共组件**；
2. **业务表单校验放在页面或具体表单组件中**；
3. **提交成功后统一关闭弹窗并刷新列表**；
4. **尽量复用 `v-model:open`，减少手工维护显隐状态**；
5. **统一使用后端 message + HTTP 公共层提示，不要在每个弹窗里重复写提示文案**。
