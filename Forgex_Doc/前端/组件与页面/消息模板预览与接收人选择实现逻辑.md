# 消息模板预览与接收人选择实现逻辑

> 分类：前端 / 组件与页面
> 版本：**V0.6.5**
> 更新时间：**2026-04-17**

本文档说明消息模板相关的两个公共组件如何实现：

- `TemplatePreview.vue`
- `ReceiverSelector.vue`

这两个组件都服务于消息模板配置页面，但职责不同。

## 功能定位

### TemplatePreview

负责把多平台消息内容做成“可视化预览弹窗”，帮助用户在保存前看到：

- 不同平台的标题与正文
- 占位符替换后的示例效果
- 链接展示效果

### ReceiverSelector

负责统一选择接收人配置，支持：

- 用户
- 角色
- 部门
- 岗位
- 自定义

它的本质不是发消息，而是生成接收人配置模型。

## TemplatePreview 实现逻辑

### 输入模型

`TemplatePreview` 主要接收：

- `visible`
- `contents`
- `messageType`

其中 `contents` 是模板内容数组，每个元素对应一个平台。

### 内部处理流程

组件打开后会：

1. 根据 `contents` 决定默认激活的 tab
2. 根据平台类型显示不同平台名称和图标
3. 通过 `getI18nValue` 解析多语言标题和正文
4. 使用内置示例数据替换占位符
5. 生成一个仅用于预览的消息卡片

这意味着它不是把真实消息发出去，而是做“本地渲染模拟”。

### 占位符模拟逻辑

组件内部维护了一套示例数据，例如：

- `userName`
- `userAccount`
- `tenantName`
- `currentTime`

当正文里出现 `${userName}` 这类占位符时，会被替换成示例文本。

这样做的价值是：

- 配置人员不需要实际触发消息就能看到效果
- 模板里的占位符错误更容易提前发现

### 平台差异处理

预览组件会根据 `platform` 渲染不同的平台头部与样式，例如：

- 站内消息
- 企业微信
- 短信
- 邮件

虽然底层都是同一个组件，但视觉上会给出平台区分。

## ReceiverSelector 实现逻辑

### 输入输出模型

`ReceiverSelector` 使用：

```vue
v-model="receiver"
```

值结构大致为：

```ts
{
  receiverType?: string
  receiverIds: string[]
}
```

### 类型切换逻辑

用户切换接收类型后，组件会：

1. 清空当前 `receiverIds`
2. 如果不是 `CUSTOM`，重新加载候选项
3. 把新的值回传给父组件

这保证了“类型变化”和“候选项变化”始终一致。

### 候选项加载逻辑

组件根据 `receiverType` 动态调用不同接口：

- `USER` -> `getUserList`
- `ROLE` -> `getRoleList`
- `DEPT` -> `listDepartments`
- `POSITION` -> `listPositions`

这说明它的本质是一个“统一加载器”，而不是固定写死某一类选择器。

### CUSTOM 类型处理

当类型是 `CUSTOM` 时，组件不再显示多选框，而是显示说明提示。

原因是这类接收人不是页面直接选出来的，而是由后端运行时动态决定。

## 在消息模板页面中的调用链路

典型链路如下：

```text
消息模板页面
  -> receiver tab 中渲染 ReceiverSelector
  -> content tab 中渲染 I18nInput
  -> 点击预览按钮
  -> 打开 TemplatePreview
  -> 读取当前 formData.contents 做模拟展示
```

对应页面：

```text
Forgex_MOM/Forgex_Fronted/src/views/system/messageTemplate/index.vue
```

## 为什么拆成两个公共组件

### 1. 接收人与内容预览是两种完全不同的职责

一个负责“收给谁”，一个负责“发什么样子”。

### 2. 后续可复用

如果以后其他通知、审批、告警模块也需要类似能力，这两个组件可以直接复用。

### 3. 页面复杂度可控

消息模板页面本身已经很重，如果把这些逻辑都塞在页面文件里，会更难维护。

## 常见排查

### 预览弹窗打开但没有内容

优先检查：

1. `contents` 是否真的有值
2. `contentBodyI18nJson` 是否为空

### 占位符没有被替换

优先检查：

1. 模板里是否使用了 `${key}` 这种标准写法
2. 是否写成了系统未支持的占位符名

### 接收人选项加载失败

优先检查：

1. 当前 `receiverType` 对应的接口是否可用
2. 接口返回结构是否能被组件正确映射为 `{ label, value }`

### 切换接收类型后旧值残留

正常情况下组件会清空旧值。
如果页面仍然看到旧值，通常是外层表单数据被手动覆盖回去了。
