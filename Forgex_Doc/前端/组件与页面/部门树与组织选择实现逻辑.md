# 部门树与组织选择实现逻辑

> 分类：前端 / 组件与页面  
> 版本：**V0.6.0**  
> 更新时间：**2026-04-17**

本文档说明部门树组件在 Forgex 中的真实实现方式，包含组件定位、数据来源、事件模型、页面联动方式以及代码位置。

## 功能定位

`DeptTree.vue` 不是一个“组织架构管理系统”，它更准确的定位是“组织选择面板”。

它主要服务于以下场景：

- 左树右表页面
- 部门筛选用户、岗位、角色
- 组织节点选择
- 在详情页中作为统一的组织导航入口

它不负责：

- 完整的组织编辑逻辑
- 组织新增、修改、删除接口封装
- 右侧业务数据的查询

这些能力都应该交给业务页面处理。

## 组件对外能力

`DeptTree.vue` 对外主要提供三种能力：

- 统一渲染组织树
- 选中节点后抛出 `select`
- 按钮点击后抛出 `add`

同时还通过 `defineExpose` 暴露：

- `loadTree`
- `selectedKeys`

这样页面可以在保存组织后主动刷新树。

## 数据加载逻辑

组件挂载后会执行 `loadTree()`。

内部逻辑很直接：

1. 从 `sessionStorage` 读取 `tenantId`
2. 调用 `getDepartmentTree({ tenantId })`
3. 将结果赋值给 `treeData`
4. 树组件根据 `fieldNames` 渲染节点

这说明它有一个明确前提：  
当前页面必须已经选定租户，否则树不会加载。

## 节点字段映射

组件里通过 `fieldNames` 做了树字段映射：

```ts
const fieldNames = {
  key: 'id',
  title: 'deptName',
  children: 'children'
}
```

也就是说，后端树结构至少需要保证：

- `id`
- `deptName`
- `children`

这三个字段稳定存在。

## 选中逻辑

用户点击树节点时，组件会执行 `onSelect(keys, info)`。

它做了两件事：

1. 将 `selectedKeys` 更新为当前节点
2. 把完整节点对象通过 `emit('select', keys, node)` 抛给页面

这里比较关键的是第二点。  
页面拿到的不是只有 ID，而是整棵节点的当前数据，这让页面可以直接读取：

- `id`
- `deptName`
- `orgType`
- `status`

避免再额外查一次轻量信息。

## 图标与节点状态渲染

组件没有只用一种文件夹图标，而是根据 `orgType` 显示不同图标：

- `group`
- `company`
- `subsidiary`
- `department`
- `team`

同时，节点 `status === false` 时会加禁用样式。

这套设计的作用是：

- 提高组织层级的可辨识度
- 让停用节点在 UI 上有弱化表现

## 页面联动链路

典型链路如下：

```text
DeptTree 选中节点
  -> emit('select', keys, node)
  -> 页面更新 currentDeptId
  -> 页面刷新右侧列表或详情
```

例如在 `system/department/index.vue` 中：

- 左侧 `DeptTree` 只负责树
- 右侧页面负责详情展示和编辑
- 保存成功后页面调用 `deptTreeRef.loadTree()`

这是一种很清晰的职责拆分。

## 在部门管理页中的实现方式

`src/views/system/department/index.vue` 是最标准的参考实现。

页面逻辑大致如下：

1. 左侧挂载 `DeptTree`
2. 节点选中后，调用 `getDepartment` 加载详情
3. 编辑或新增时，页面内部切换表单状态
4. 保存成功后刷新树
5. 根据当前节点重新回显右侧详情

也就是说：

- 树只负责“导航”
- 页面负责“详情与编辑”

## 为什么这样设计

### 1. 复用性更高

如果把“树加载、树渲染、图标、选中样式”每个页面都写一遍，维护成本很高。

### 2. 避免组件侵入业务

如果树组件把新增、编辑、删除、右侧查询全部包进去，它就会变成一个难复用的超级组件。

### 3. 适合左树右表范式

后台很多页面本质上都是：

- 左边组织或分类树
- 右边列表或详情

`DeptTree` 正好服务这个范式。

## 常见排查

### 树没有加载出来

优先检查：

1. `sessionStorage` 中是否已有 `tenantId`
2. `getDepartmentTree` 接口是否返回了树结构
3. 接口字段是否包含 `id/deptName/children`

### 点击节点后页面没反应

优先检查：

1. 页面是否监听了 `@select`
2. 页面是否使用了 `node.id` 或 `keys[0]`

### 新增组织后左树没刷新

优先检查：

1. 页面保存成功后是否调用了 `deptTreeRef.loadTree()`
2. `ref` 是否正确绑定到 `DeptTree`

### 节点图标不对

优先检查：

1. 后端返回的 `orgType` 是否符合组件内部约定值
2. 是否新增了组织类型但组件未补图标映射
