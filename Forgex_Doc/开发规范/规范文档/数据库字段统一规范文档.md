# 数据库字段统一规范文档

> 分类：开发规范 / 规范文档

本页作为数据库字段规范的提交版入口。

## 页面开发默认流程

除非需求明确要求特殊实现，新增或改造一个标准管理页面时，默认遵循以下流程：

1. 后端按标准四层架构落地：`controller`、`service`、`domain`、`mapper`。
2. `domain` 下按现有模块习惯拆分 `entity`、`dto`、`param`、`vo` 等对象。
3. 数据访问优先使用 MyBatis-Plus 和 MPJ，简单 CRUD、分页、条件查询、常规联表不得手写 SQL。
4. 仅当 SQL 特别复杂、MP/MPJ 难以表达或可维护性明显下降时，才允许在 XML 中手写 SQL；禁止注解 SQL。
5. 后端接口返回统一走项目公共返回结构、统一数据库配置消息和国际化提示枚举，不在业务代码中散落硬编码提示。
6. 前端列表页面默认走公共表格 / `FxDynamicTable`，列配置、查询条件、字典映射保持与后端字段一致。
7. 基础语言默认配置 5 种，涉及菜单、按钮、提示语、字段展示时同步维护多语言资源。
8. Java 代码、Vue 代码、SQL 字段备注都必须遵循开发规范补充中文注释或备注。
9. Windows 环境生成中文文件时必须显式注意 UTF-8 编码，避免中文注释、文档、枚举消息出现乱码。
10. 本地数据库默认环境：`127.0.0.1:3306`，用户名 `root`，密码 `123456`。

## 建表字段取名流程

新增表或新增字段前，必须先查本文档“常用字段标准表”：

1. 表中已有对应字段含义时，必须直接复用字段名、字段类型、长度和备注。
2. 不允许同一业务含义在不同表中出现多套命名，例如物料名称不得一处叫 `material_name`，另一处叫 `mat_name` 或拼错成 `matreil_name`。
3. 表中没有对应字段时，先按本文档命名规则新增到“常用字段标准表”，再用于建表。
4. 冗余字段也必须使用标准字段名，保证跨表查询、动态表格、导入导出、报表字段能统一识别。

## 核心约束

- 通用审计字段：`create_time`、`update_time`、`create_by`、`update_by`、`deleted`
- 多租户字段：`tenant_id`
- 名称、标题、标签等国际化字段优先采用 JSON / 长文本能力承载
- 新增 SQL、实体、前端类型需保持同步

## 常用字段标准表

### 公共字段

| 字段名 | 字段类型 | 字段长度 / 精度 | 字段备注 | 使用说明 |
|---|---|---:|---|---|
| `id` | `bigint` | 20 | 主键 ID | 所有业务表主键，默认雪花 ID |
| `tenant_id` | `bigint` | 20 | 租户 ID | 需要租户隔离的业务表必填，公共数据可为 `0` |
| `create_time` | `datetime` | - | 创建时间 | 所有业务表必填 |
| `update_time` | `datetime` | - | 更新时间 | 所有业务表必填 |
| `create_by` | `varchar` | 50 | 创建人 | 记录创建人账号或 ID |
| `update_by` | `varchar` | 50 | 更新人 | 记录最后更新人账号或 ID |
| `deleted` | `tinyint` | 1 | 逻辑删除：0=未删除，1=已删除 | 所有业务表必填 |
| `remark` | `varchar` | 500 | 备注 | 常规短备注优先使用该字段 |
| `sort_order` | `int` | 11 | 排序号 | 列表排序、配置排序 |
| `status` | `tinyint` | 1 | 状态：0=禁用，1=启用 | 通用启停状态 |
| `enabled` | `tinyint` | 1 | 是否启用：0=否，1=是 | 当语义是开关时使用 |
