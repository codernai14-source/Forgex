# Forgex AI 自动化测试手册

> 分类：开发规范 / 测试规范  
> 适用范围：Forgex Web 管理端、后端接口联调、数据库结果核验、部署环境冒烟  
> 维护原则：本文档只描述测试方法、检查点与提示词模板，不替代具体模块的功能说明。

## 一、测试目标

AI 自动化测试的目标不是简单“点一遍页面”，而是让 AI 以测试人员视角完成以下工作：

1. 验证功能主流程可用，包括登录、进入菜单、查询、新增、编辑、删除、导入、导出、启停、授权、审批等。
2. 验证页面状态正确，包括按钮可见性、表格列、分页、弹窗、抽屉、loading、成功/失败消息、登录失效提示。
3. 验证前后端链路一致，包括接口路径、请求参数、响应结构、表格字段、字典值、权限编码、租户上下文。
4. 验证异常场景可控，包括必填校验、重复数据、权限不足、接口失败、会话过期、网络超时、空数据。
5. 输出可复现报告，包括环境、账号、测试数据、操作步骤、实际结果、接口状态、截图或 trace、问题定位建议。

## 二、测试前准备

### 1. 环境确认

测试开始前，AI 需要先确认以下信息：

| 项目 | 检查内容 |
|---|---|
| 前端地址 | 本地通常为 `http://localhost:5173`，实际以启动日志或部署地址为准 |
| 后端地址 | 前端通过 `/api` 代理访问后端，需要确认后端服务可用 |
| 登录账号 | 需要包含账号、密码、租户、角色权限说明 |
| 测试租户 | 多租户场景必须记录当前租户，避免误测其他租户数据 |
| 测试数据 | 使用带明显前缀的数据，例如 `AI_TEST_模块_时间戳` |
| 数据清理 | 新增、导入、审批、授权类测试需要说明清理方式 |
| 浏览器 | 优先 Chromium；必要时补 Firefox、WebKit 兼容验证 |
| 权限边界 | 至少准备管理员账号和普通角色账号，用于权限差异测试 |

### 2. 前端启动检查

进入 `Forgex_MOM/Forgex_Fronted` 后执行：

```bash
npm install
npm run dev
```

如果只需要验证构建：

```bash
npm run build
```

AI 不应只看命令是否退出，还需要检查：

1. Vite 是否成功输出访问地址。
2. 控制台是否有 TypeScript、Vue、Less、依赖解析错误。
3. 浏览器打开页面后是否出现白屏。
4. Network 中静态资源是否有 404。
5. Console 中是否有阻断主流程的错误。

### 3. 后端与基础服务检查

联调测试前需要确认：

| 项目 | 检查方式 |
|---|---|
| 后端服务 | 网关、认证、系统、业务模块是否启动成功 |
| 数据库 | 当前环境连接、初始化脚本、测试租户数据是否存在 |
| Redis | 登录态、验证码、缓存相关功能是否可用 |
| Nacos | 前后端使用的接口地址、数据源、租户配置是否一致 |
| RocketMQ | 涉及消息、审批、异步任务时再检查 |
| 文件服务 | 导入、导出、上传、预览场景需要检查 |

## 三、推荐使用的 MCP

具体 MCP 名称以当前 AI 客户端实际暴露为准。测试时优先使用以下能力组合：

| MCP / 工具能力 | 用途 | 使用建议 |
|---|---|---|
| Playwright / Browser MCP | 打开页面、点击按钮、填写表单、读取可访问性快照、截图、录制 trace | 前端自动化测试的主工具；每次操作前后都要重新 snapshot |
| Filesystem / Workspace MCP | 读取页面代码、API 封装、路由、文档、测试产物 | 用于确认按钮文案、字段名、权限码、接口路径 |
| Shell / Terminal MCP | 启动前端、执行构建、查看日志、运行脚本 | 用于 `npm run dev`、`npm run build`、后端日志排查 |
| Database / MySQL MCP | 核验数据落库、清理测试数据、检查字典与表格配置 | 优先只读查询；清理或写入必须确认环境和数据范围 |
| Network / DevTools MCP | 查看接口请求、响应、状态码、失败原因 | Playwright 能力不足时使用，重点看 `/api` 请求 |
| Git MCP | 查看改动范围、确认测试覆盖与变更文件 | 回归测试时用于识别受影响模块 |
| Docs / Context MCP | 查询框架或组件文档 | 只在组件行为不明确时使用，项目内文档优先 |

最低可用组合：

```text
Playwright / Browser MCP + Filesystem / Workspace MCP + Shell / Terminal MCP
```

涉及数据一致性时增加：

```text
Database / MySQL MCP
```

## 四、AI 前端自动化测试通用流程

### 1. 建立上下文

AI 开始点击页面前，先阅读以下内容：

1. 当前需求或缺陷说明。
2. 对应模块文档，优先从 `Forgex_Doc/开发规范/模块文档映射/README.md` 找入口。
3. 前端页面文件，通常在 `Forgex_MOM/Forgex_Fronted/src/views`。
4. API 封装文件，通常在 `Forgex_MOM/Forgex_Fronted/src/api`。
5. 公共组件文档，重点是 `FxDynamicTable`、`BaseFormDialog`、HTTP 请求与消息提示。

### 2. 打开页面并登录

AI 操作顺序：

1. 打开前端地址。
2. 如果跳转到 `/login`，填写账号、密码、验证码或滑块。
3. 如果出现租户选择，选择指定测试租户。
4. 点击登录页主按钮，中文环境通常为“身份校验”。
5. 登录成功后等待进入 `/workspace/home` 或目标页面。

需要关注：

| 检查点 | 期望 |
|---|---|
| 登录按钮 | loading 后恢复或跳转，不应卡死 |
| 登录失败 | 展示后端错误消息，不应静默失败 |
| 租户选择 | 必须选择指定租户，sessionStorage 中应有 `tenantId` |
| 会话缓存 | sessionStorage 中应有 `account`、`tenantId`、`permissions` |
| 路由恢复 | 动态菜单加载成功，不应反复跳回登录页 |

### 3. 进入目标菜单

AI 优先通过左侧菜单进入页面；如果菜单较深，可以使用路由直接访问，但报告里要注明。

操作方式：

1. 在主布局中定位目标模块。
2. 展开一级、二级菜单。
3. 点击目标菜单项。
4. 等待页面主区域、查询区、表格区加载完成。
5. 如果直接访问路由，需要确认菜单高亮和页面标题是否正常。

需要关注：

| 检查点 | 期望 |
|---|---|
| 菜单可见 | 有权限账号能看到目标菜单 |
| 菜单隐藏 | 无权限账号不应看到目标菜单 |
| 路由 | URL 与菜单配置一致 |
| 页面标题 | 标题和菜单语义一致 |
| 白屏 | 页面不应出现 `Page not available yet` 或组件加载失败 |

### 4. 测试 FxDynamicTable 页面

Forgex 列表页优先使用 `FxDynamicTable`，AI 需要重点测试公共表格行为。

#### 查询区

AI 需要点击和检查：

| 操作 | 按钮 / 控件 | 检查点 |
|---|---|---|
| 输入查询条件 | 文本框、下拉框、日期范围 | 输入后模型不丢失，日期格式正确 |
| 执行查询 | “搜索” | 请求参数包含查询条件，表格刷新 |
| 清空查询 | “重置” | 查询条件清空，分页回到第一页 |
| 展开更多条件 | “展开” | 第 4 个及之后查询条件显示 |
| 收起更多条件 | “收起” | 额外条件收起，不影响已提交查询 |

AI 需要关注的消息和异常：

1. 查询请求不应弹成功消息。
2. 查询失败应弹出可读错误消息。
3. 空数据时应显示空状态，不应报错。
4. 查询后分页总数、当前页、表格行数要一致。

#### 工具栏

AI 需要按页面能力点击：

| 操作 | 常见按钮 | 检查点 |
|---|---|---|
| 新增 | “新增” | 打开弹窗或抽屉，标题为新增语义 |
| 批量删除 | “批量删除” | 未选中时禁用或提示；选中后弹确认 |
| 导入 | “导入” | 打开上传弹窗，模板下载可用 |
| 导出 | “导出” | 触发下载，不应阻塞页面 |
| 拉取公共配置 | “拉取公共配置” | 有确认或成功提示，数据刷新 |
| 刷新 | “刷新” | 表格重新请求 |

#### 列设置

`FxDynamicTable` 默认带“列设置”按钮。AI 需要测试：

1. 点击“列设置”。
2. 勾选或取消非操作列。
3. 点击上移、下移按钮调整顺序。
4. 点击“保存”。
5. 确认表格列立即变化。
6. 刷新页面后确认用户列配置仍生效。
7. 点击“重置”，确认恢复默认列。

注意：

1. `action` 操作列通常不可取消。
2. 保存列设置应出现成功提示或至少无错误。
3. 列设置浮层不应一打开就自动关闭。
4. 深色主题下列名、按钮、边框必须可见。

#### 表格内容和分页

AI 需要检查：

| 操作 | 检查点 |
|---|---|
| 首次加载 | 列名、数据、loading、空状态正常 |
| 分页切换 | 页码变化后请求参数正确 |
| 每页条数切换 | pageSize 变化后表格刷新 |
| 排序 | 后端收到排序字段和排序方向 |
| 横向滚动 | 固定列、操作列不遮挡 |
| 自定义列 | 标签、开关、字典翻译、日期格式正确 |
| 行选择 | 选中状态与批量按钮联动 |

### 5. 测试弹窗和抽屉表单

Forgex 表单弹层通常使用 `BaseFormDialog`。AI 需要按下面流程测试：

1. 点击“新增”或“编辑”。
2. 等待弹窗或抽屉打开。
3. 检查标题、字段、默认值、必填标记。
4. 不填必填项，点击“确认”，检查校验消息。
5. 填写合法数据，点击“确认”。
6. 检查按钮 loading、接口请求、成功提示、弹窗关闭、列表刷新。
7. 再次打开新增，确认没有残留上次数据。
8. 编辑已有数据，确认回显正确。
9. 点击“取消”或关闭按钮，确认弹窗关闭且未保存。

需要关注：

| 检查点 | 期望 |
|---|---|
| 确认按钮 | 触发表单校验和保存 |
| 取消按钮 | 关闭弹层，不发保存请求 |
| 遮罩点击 | 默认不关闭复杂表单，除非页面显式允许 |
| 保存成功 | 使用后端成功消息，不重复弹多个成功提示 |
| 保存失败 | 弹出后端错误消息，弹窗保留用户输入 |
| loading | 保存中按钮不可重复提交 |
| 列表刷新 | 新增、编辑、删除后表格刷新 |

### 6. 测试行操作

行操作通常由 `FxActionGroup` 渲染，最多平铺 3 个操作，其余进入“更多”。

AI 需要点击：

1. “编辑”。
2. “详情”或“查看”。
3. “删除”。
4. “启用 / 停用”或开关。
5. “授权”“配置”“复制”等模块特有操作。
6. 如果存在“更多”，点击“更多”后再点击下拉菜单内操作。

需要关注：

| 检查点 | 期望 |
|---|---|
| 权限过滤 | 无权限操作不可见 |
| 禁用态 | 禁用操作不可点击 |
| 危险操作 | 删除类操作有确认提示 |
| 更多菜单 | 展开后菜单项可点击，点击后浮层关闭 |
| 数据刷新 | 操作成功后行状态或列表刷新 |
| 错误提示 | 操作失败显示后端 message |

### 7. 测试删除和批量删除

AI 删除数据时必须使用测试数据，不得删除真实业务数据。

操作顺序：

1. 查询 `AI_TEST_` 前缀数据。
2. 点击行内“删除”。
3. 检查是否出现确认框。
4. 点击确认。
5. 检查成功消息和列表刷新。
6. 再次查询确认数据不存在。
7. 批量删除时先选中多条测试数据，再点击“批量删除”。

注意：

1. 删除前截图或记录被删数据主键。
2. 如果没有确认框，要在报告中标记为风险。
3. 删除失败时检查是否有外键引用、状态限制或权限不足。

### 8. 测试导入导出和上传

#### 导入

AI 需要测试：

1. 点击“导入”。
2. 点击“下载模板”。
3. 上传合法模板。
4. 上传字段缺失、格式错误、重复数据文件。
5. 检查导入成功和失败明细。
6. 导入后查询新增数据。

注意：

1. 文件大小、文件类型错误要有明确提示。
2. 导入失败不能产生半截脏数据，除非业务明确允许部分成功。
3. 导入接口不应导致页面长时间全局 loading 卡死。

#### 导出

AI 需要测试：

1. 无查询条件导出。
2. 带查询条件导出。
3. 空结果导出。
4. 大数据量导出。
5. 检查下载文件名、文件格式、列名、数据范围。

### 9. 测试国际化和主题

Forgex 支持多语言和主题配置，AI 至少需要覆盖中文主流程；涉及公共组件或提示文案变更时，需要补多语言测试。

检查项：

| 类型 | 检查内容 |
|---|---|
| 语言切换 | 菜单、按钮、表单标签、表格列、消息提示是否切换 |
| 后端提示 | 成功/失败 message 是否按当前语言展示 |
| 旧文案兼容 | 旧中文硬编码是否能被兼容翻译 |
| 深浅主题 | 表格、弹窗、列设置、下拉框、消息提示是否可读 |
| 布局密度 | 表格高度、分页、固定列、工具栏不应错位 |

### 10. 测试权限和多租户

AI 需要至少用两类账号测试：

1. 管理员账号：应看到完整菜单和操作。
2. 普通角色账号：只看到已授权菜单和按钮。

检查项：

| 场景 | 检查内容 |
|---|---|
| 菜单权限 | 未授权菜单不显示，直接访问路由应被拦截或无操作能力 |
| 按钮权限 | 新增、编辑、删除、导入、导出、授权按钮按权限显示 |
| 数据权限 | 普通账号只能看到授权范围内数据 |
| 租户隔离 | A 租户新增的数据不应在 B 租户出现 |
| 登录失效 | 后端返回 602 时弹“会话已过期，请重新登录”并跳转登录 |

### 11. 测试后端接口和消息

AI 在每个关键操作后都需要检查 Network：

| 操作 | 需要检查 |
|---|---|
| 查询 | 请求路径、pageNum、pageSize、query 条件、响应 records/total |
| 新增 | 请求体字段、租户头、响应 code、message |
| 编辑 | 主键、修改字段、更新时间 |
| 删除 | 主键或批量主键、确认提示、响应 message |
| 启停 | 状态字段、列表刷新后状态 |
| 导入导出 | 文件请求、blob 响应、失败提示 |

HTTP 消息规则：

1. 查询、详情、字典、树加载不应自动弹成功消息。
2. 新增、修改、删除、启停、授权、保存配置应显示成功消息。
3. 页面不应重复手写 `message.success` 导致双提示。
4. 失败消息优先展示后端 `message` 或 `msg`。
5. 静默请求不应打断用户操作。

## 五、模块测试范围清单

### 1. 登录、注册、初始化

必测：

1. 账号密码登录。
2. 验证码或滑块校验。
3. 租户选择和默认租户。
4. 登录失败提示。
5. 登录成功后菜单和个人首页。
6. 会话过期重新登录。
7. 初始化向导只在对应环境出现。

### 2. 系统管理

必测：

1. 用户、角色、菜单、部门、职位、租户、模块、字典。
2. 角色授权菜单和按钮。
3. 用户分配角色、重置密码、启停。
4. 菜单新增后动态路由是否生效。
5. 表格配置和用户列设置。
6. 国际化资源维护。
7. 系统配置、登录页配置、主题配置。
8. 操作日志、登录日志、在线用户。

重点风险：

1. 权限码和按钮显示不一致。
2. 菜单 component key 无法加载页面。
3. 多租户数据串租。
4. 动态表格字段配置与接口返回字段不一致。

### 3. 基础资料

必测：

1. 供应商、客户、工厂、物料、编码规则等基础 CRUD。
2. 字典字段展示和筛选。
3. 导入模板、导入校验、导出结果。
4. 状态启停。
5. 唯一性校验和引用限制。

重点风险：

1. 编码、名称、状态字段不一致。
2. 导入重复数据导致脏数据。
3. 删除被引用数据未提示。

### 4. 工作流审批

必测：

1. 审批任务配置列表。
2. 节点配置页面。
3. 发起审批。
4. 待办、已办、我发起的。
5. 同意、驳回、撤回、转办、加签等实际支持的动作。
6. 补偿中心和异常流程。
7. 审批消息通知。

重点风险：

1. 流程节点配置保存后未生效。
2. 表单参数和业务单据关联失败。
3. 审批后状态不同步。
4. 回调失败没有可追踪记录。

### 5. 集成平台

必测：

1. 第三方系统配置。
2. API 配置、参数配置、参数映射。
3. 认证方式、签名、请求头。
4. 连接测试或接口调用测试。
5. 启停、复制、删除。
6. 错误响应和超时提示。

重点风险：

1. 敏感字段明文展示。
2. 参数映射保存后丢失。
3. 接口测试失败但页面提示成功。

### 6. 报表和看板

必测：

1. 报表分类。
2. 报表模板。
3. 数据源配置和连接测试。
4. 查询参数和预览。
5. 图表渲染、导出。

重点风险：

1. SQL 或数据源配置错误导致页面白屏。
2. 大数据量渲染卡顿。
3. 导出结果与页面筛选不一致。

### 7. 标签打印

必测：

1. 标签模板。
2. 标签绑定。
3. 打印任务。
4. 打印记录。
5. 模板预览和实际打印数据。

重点风险：

1. 模板字段和业务数据字段不匹配。
2. 打印记录未落库。
3. 重复打印没有明显状态。

### 8. 个人首页与消息

必测：

1. 个人首页布局。
2. 拖拽、保存、恢复默认。
3. 收藏菜单。
4. 未读消息、消息模板、消息预览。
5. 个人资料和修改密码。

重点风险：

1. 布局保存后刷新丢失。
2. 未读数量和列表不一致。
3. 修改密码后登录态处理异常。

## 七、AI 必须观察的消息和状态

### 1. 成功消息

重点观察：

1. 新增成功。
2. 保存成功。
3. 更新成功。
4. 删除成功。
5. 启用或停用成功。
6. 导入成功。
7. 授权成功。
8. 重置成功。

成功消息不应重复出现两次。

### 2. 失败消息

重点观察：

1. 必填校验。
2. 唯一性校验。
3. 权限不足。
4. 租户未选择。
5. 会话已过期。
6. 网络错误。
7. 服务器错误。
8. 文件格式错误。
9. 数据被引用不可删除。

失败消息必须让用户知道下一步怎么处理，不能只显示 `undefined`、`null`、`[object Object]`。

### 3. Loading 和禁用态

重点观察：

1. 查询时表格 loading。
2. 保存时确认按钮 loading。
3. 全局 loading 是否按请求结束关闭。
4. 批量删除在未选中时是否禁用。
5. 重复点击确认按钮是否会重复提交。

### 4. 弹窗和确认框

重点观察：

1. 删除、批量删除、重置、提交审批等高风险操作必须有确认。
2. 弹窗关闭后再次打开不应残留旧数据。
3. 保存失败时弹窗不应自动关闭。
4. 取消操作不应发保存请求。

### 5. 浏览器控制台

测试期间发现以下内容需要记录：

1. Vue runtime error。
2. Unhandled Promise rejection。
3. 组件加载失败。
4. 资源 404。
5. 跨域或代理错误。
6. ResizeObserver 频繁报错导致功能异常。

## 八、AI 自动化测试注意事项

1. 每次使用 Playwright 元素引用前必须先 snapshot，页面变化后重新 snapshot。
2. 不要凭旧的元素编号继续点击，弹窗、分页、菜单展开后元素引用容易失效。
3. 优先通过按钮文本、label、placeholder、role 定位元素；不要依赖脆弱的 CSS 层级。
4. 不要删除、修改生产或共享真实数据；新增测试数据必须带 `AI_TEST_` 前缀。
5. 测试删除前先确认目标数据是本轮测试生成的数据。
6. 文件导入测试要准备合法文件和非法文件。
7. 涉及审批、消息、异步任务时，需要等待后端处理完成后再断言。
8. 查询类接口不要期望成功 toast；写操作才需要检查成功提示。
9. 如果页面文案受多语言影响，定位时优先使用当前语言下的真实文案。
10. 如果测试失败，先区分是环境问题、权限问题、测试数据问题、前端问题还是后端问题。
11. 记录每个失败步骤的 URL、账号、租户、请求路径、请求体、响应体、截图。
12. 对于直接访问路由的测试，仍需补一次菜单进入验证。
13. 对于列设置、布局配置、个人首页这类用户个性化功能，必须刷新页面验证持久化。
14. 对于多租户功能，必须切换租户或账号验证数据隔离。
15. 对于权限功能，必须同时验证“看得见”和“看不见”。

## 十二、回归测试建议

代码变更后，AI 应按影响范围选择回归层级：

| 变更类型 | 建议回归 |
|---|---|
| 只改文案 | 当前页面展示、多语言、消息提示 |
| 只改样式 | 当前页面、弹窗、表格、深浅主题、不同窗口宽度 |
| 改公共组件 | 至少 3 个使用该组件的代表页面 |
| 改 HTTP 层 | 登录、查询、新增、编辑、删除、会话过期、失败提示 |
| 改动态表格 | 查询、列设置、分页、操作列、表格高度 |
| 改权限 | 管理员和普通账号双角色验证 |
| 改租户 | 两个租户数据隔离验证 |
| 改导入导出 | 模板、合法文件、非法文件、导出内容 |
| 改审批 | 配置、发起、审批、回调、消息通知全链路 |

公共组件改动不要只测一个页面。优先选取：

1. 一个系统管理页面。
2. 一个基础资料页面。
3. 一个业务复杂页面，例如工作流或集成平台页面。
