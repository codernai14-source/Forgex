# 项目架构设计文档

> 版本：**V0.6.5**
> 更新时间：**2026-04-13**

## 一、文档目标

本文档用于说明 Forgex 当前的项目结构、核心技术边界、前后端协作方式，以及后续文档应该如何围绕这些边界进行组织。

## 二、项目定位

Forgex 是一套面向企业级场景的全栈开发脚手架，核心目标是把企业项目里高频、重复、跨模块复用的能力沉淀为统一底座，减少业务项目从零搭建成本。

当前仓库覆盖三端：

- Web 管理端：`Vue 3 + TypeScript + Vite`
- Backend 微服务：`Spring Boot 3 + Spring Cloud + Spring Cloud Alibaba`
- Android 移动端：`Kotlin + Compose + Hilt + Retrofit + DataStore`

## 四、后端架构分层

### 4.1 服务划分

| 模块 | 主要职责 |
|---|---|
| `Forgex_Common` | 统一返回、国际化、租户上下文、通用配置、公共 DTO/VO、表格配置能力 |
| `Forgex_Auth` | 登录、验证码、注册、租户选择、第三方登录、权限内部接口 |
| `Forgex_Sys` | 系统管理、字典、消息中心、国际化、仪表盘、动态表格、编码规则 |
| `Forgex_Workflow` | 流程配置、流程发起、待办/已办、审批执行、回调 |
| `Forgex_Report` | 报表分类、模板、数据源、设计与预览 |
| `Forgex_Gateway` | 网关统一入口、上下文透传、跨服务访问入口 |
| `Forgex_Job` | 分布式任务调度接入 |
| `Forgex_Basic` | 基础业务扩展位 |

### 4.2 设计原则

1. **Common 只提供公共能力，不依赖业务模块。**
2. **Auth 负责认证与权限来源，不承载具体业务实现。**
3. **Sys 负责平台基础能力，Workflow/Report 负责业务中台能力。**
4. **所有服务通过统一上下文读取租户、用户、语言信息。**

## 五、前端架构分层

| 层级 | 说明 |
|---|---|
| `src/api` | 所有接口请求封装，统一走 `http.ts` |
| `src/components/common` | 动态表格、列设置、字典标签、弹窗、多语言输入等公共组件 |
| `src/views` | 模块页面实现，按系统管理、工作流、报表等业务领域拆分 |
| `src/store` | 登录态、路由、标签页、主题等全局状态 |
| `src/locales` | 国际化文案与语言切换入口 |

前端设计原则：

1. **接口调用不在页面里直接处理重复消息提示。**
2. **公共交互优先抽象到 `components/common`。**
3. **表格类页面优先采用配置驱动接入。**
4. **权限、动态路由、租户态统一通过全局链路收口。**

## 六、核心上下文链路

### 6.1 登录与租户链路

```text
前端登录 -> Auth 返回用户态/租户列表 -> 选择租户 -> Gateway 透传上下文
-> 下游服务写入 TenantContext / UserContext / LangContext -> 业务执行
```

### 6.2 国际化链路

```text
前端请求头 X-Lang / Accept-Language
-> LangWebInterceptor / LangContext
-> R + I18nPrompt / I18nBusinessException
-> RMessageI18nAdvice 翻译
-> 前端 http.ts 消费 message 并提示
```

### 6.3 动态表格链路

```text
页面传入 tableCode + request
-> FxDynamicTable 拉取表格配置
-> CommonTableController / FxTableConfigService 组装配置
-> 页面 request 拉业务数据
-> ColumnSettingButton 保存用户个性化列设置
```
