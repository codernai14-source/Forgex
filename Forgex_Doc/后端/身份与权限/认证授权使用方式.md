# 认证授权使用方式

> 分类：后端 / 身份与权限
> 版本：**V0.6.0**
> 更新时间：**2026-04-17**

本文档说明认证授权能力在项目中的实际使用方式，重点覆盖：

1. 后端代码如何接入登录与权限校验。
2. 前端页面如何对接登录、选租户、按钮权限。
3. 新增接口、新增登录方式时应如何扩展。
4. 常见联调与排查步骤。

## 1. 常用接口

认证模块常用接口如下：

| 接口 | 用途 |
|---|---|
| `POST /auth/login` | 登录并返回租户列表 |
| `POST /auth/chooseTenant` | 选择租户并建立登录态 |
| `POST /auth/logout` | 退出登录 |
| `POST /auth/register` | 邀请码注册 |
| `POST /auth/captcha/image` | 获取图片验证码 |
| `POST /auth/captcha/slider` | 获取滑块验证码 |
| `POST /auth/captcha/slider/validate` | 校验滑块 |
| `GET /auth/publicKey` | 获取前端传输加密公钥 |

## 2. 后端代码如何使用

### 2.1 新增受保护接口

后端新增业务接口时，默认应该判断它是否需要登录与权限保护。如果需要，至少做两件事：

1. 接口放在已受登录态保护的业务模块中。
2. 在 Controller 方法上补 `@RequirePerm(...)`。

示例：

```java
@RequirePerm("sys:user:list")
@PostMapping("/page")
public R<IPage<SysUserVO>> page(@RequestBody SysUserQueryDTO dto) {
    return R.ok(userService.page(dto));
}
```

接入要求：

- 权限码命名与菜单按钮保持一致。
- `Controller` 注解写的是最终约束，不要只靠前端按钮隐藏。
- 查询、新增、编辑、删除建议拆成独立权限码。

### 2.2 如何判断接口是否漏接权限

开发自检时按下面顺序检查：

1. 这个接口是否要求登录后才能访问。
2. 如果要求登录，是否已经具备后端登录态拦截。
3. 如果要做角色权限控制，是否加了 `@RequirePerm`。
4. 菜单按钮权限码是否和接口注解一致。

如果只有前端 `v-permission`，没有后端 `@RequirePerm`，那么只是“页面上看不见”，不是“真正安全”。

### 2.3 新增登录方式

如果要接入新的登录方式，例如短信登录、企业微信登录、钉钉登录，不要直接修改 `AuthController`，也不要把逻辑堆到 `AuthServiceImpl.login()` 里。正确做法是：

1. 新建登录策略类，实现对应策略接口。
2. 在策略中处理自己的参数校验、身份认证逻辑。
3. 注册到 `LoginStrategyFactory` 可识别的终端和登录类型组合中。
4. 必要时复用 `AuthServiceImpl` 中已有公共方法。

推荐扩展顺序：

```text
前端传 loginTerminal + loginType
  -> LoginStrategyFactory 识别
  -> 新策略.login(param)
  -> 统一返回租户列表
```

这样做的好处是：

- 不会破坏已有账号密码登录链路。
- PC、C 端等不同终端可以走不同策略。
- 后续问题排查时，调用链非常清晰。

### 2.4 新增租户选择方式

如果不同终端在“选租户后如何建立登录态”上存在差异，应扩展 `ChooseTenantStrategyFactory`，不要把分支写死在 Controller 中。

一般只有以下场景需要扩展：

- 不同终端 token 体系不同
- 移动端和管理端返回结构不同
- 第三方登录场景下选租户需要附带额外上下文

### 2.5 注册接口如何使用

后端使用注册能力时，建议把它视为“用户 + 租户关系初始化”能力，而不是单纯新增用户。

邀请码由系统管理的邀请码页面创建。创建时必须选择绑定角色，注册成功后用户会统一加入该邀请码绑定的角色；未绑定角色、角色停用或角色不属于邀请码租户时，邀请码校验和注册都会失败，避免生成无权限账号。

注册接口是公开链路，不依赖当前登录态的租户上下文；后端会临时忽略请求侧租户过滤，始终以邀请码自身绑定的 `tenant_id` 来校验角色并写入用户租户关系。

联调时重点确认：

1. 邀请码是否存在、是否未失效。
2. 邀请码是否绑定了默认租户和启用角色。
3. 密码是否按当前策略加密。
4. 用户表、用户租户关系表、用户角色关系表是否都成功落库。

## 3. 前端页面如何使用

### 3.1 登录页面联调顺序

前端接入登录能力时，推荐按以下顺序联调：

1. 调 `/auth/publicKey`
   获取密码加密公钥。
2. 调验证码接口
   根据当前配置决定走图片验证码还是滑块验证码。
3. 调 `/auth/login`
   提交账号、密码、验证码信息，获取租户列表。
4. 调 `/auth/chooseTenant`
   用户选定租户后换取正式 token。
5. 拉取用户信息、菜单、权限
   页面根据这些数据渲染路由与按钮。

### 3.2 页面按钮权限如何使用

前端按钮权限必须与后端 `@RequirePerm` 使用同一套权限码。典型写法如下：

```vue
<a-button v-permission="'integration:api-config:add'" type="primary">
  新增
</a-button>
```

接入要求：

- 权限码不要写显示文案。
- 页面按钮、操作列链接、顶部批量操作按钮都要加权限控制。
- 不能因为页面上隐藏了按钮，就省略后端权限注解。

### 3.3 页面为什么会“看不到新增按钮”

如果页面上看不到新增按钮，一般按以下顺序排查：

1. 前端组件是否写了 `v-permission`
2. 权限码是否与后端一致
3. 当前登录用户在当前租户下是否真的有该权限
4. 菜单、按钮权限是否已经分配给角色

以接口配置页为例，前端实际就是：

- `integration:api-config:add`

如果这个权限未下发，新增按钮就不会显示。

## 4. 开发接入示例

### 4.1 后端新增一个受权限保护的接口

```java
@RestController
@RequestMapping("/demo/order")
public class DemoOrderController {

    @RequirePerm("demo:order:add")
    @PostMapping("/create")
    public R<Long> create(@RequestBody DemoOrderCreateParam param) {
        return R.ok(orderService.create(param));
    }
}
```

你还需要同步完成：

1. 菜单/按钮配置中新增 `demo:order:add`
2. 给目标角色分配该权限
3. 前端按钮也使用 `demo:order:add`

### 4.2 前端按钮接权限

```vue
<a-button v-permission="'demo:order:add'" type="primary" @click="openCreateDialog">
  新增订单
</a-button>
```

### 4.3 登录页调用顺序示例

```ts
const publicKey = await getPublicKey()
const encryptedPassword = encrypt(password, publicKey)
const tenants = await login({
  account,
  password: encryptedPassword,
  captchaId,
  captchaCode,
  loginTerminal: 'B',
  loginType: 'ACCOUNT_PASSWORD'
})
await chooseTenant({
  tenantId: selectedTenantId,
  loginTerminal: 'B'
})
```

## 5. 联调建议

### 5.1 登录联调

登录联调不要一上来就查权限，先分两段验证：

1. `/auth/login` 是否已经正确返回租户列表
2. `/auth/chooseTenant` 是否真正返回 token 并让后续接口带上 token

第一段有问题，优先查账号、密码、验证码、密码解密。
第二段有问题，优先查租户归属、StpUtil 登录、session 写入、网关透传。

### 5.2 权限联调

权限联调建议按以下顺序：

1. 页面按钮是否显示
2. 接口是否能正常调用
3. 切换角色后权限是否随之变化
4. 切换租户后权限是否符合预期

因为 Forgex 权限是租户维度的，所以“同一用户在 A 租户有权限、在 B 租户没权限”是正常现象。

## 6. 常见问题排查

### 6.1 登录失败

优先检查：

1. 前端是否先获取了公钥
2. 密码是否重复加密或未加密
3. 验证码模式是否匹配当前系统配置
4. Redis 中失败次数是否导致账号锁定

### 6.2 选租户后仍然无登录态

优先检查：

1. `/auth/chooseTenant` 是否成功执行
2. 响应里是否真的拿到了 token
3. 前端是否保存并透传 token
4. 网关转发时是否丢失认证头

### 6.3 接口报无权限

优先检查：

1. `@RequirePerm` 是否已经加上
2. 权限码是否拼写一致
3. 当前角色是否拥有权限
4. `StpInterfaceImpl` 是否能装载到该权限

### 6.4 页面按钮不显示

优先检查：

1. 页面按钮权限码
2. 后端接口权限码
3. 菜单按钮分配
4. 当前租户是否切换正确

### 6.5 新增接口忘记保护

后端 review 时可以把下面这张检查单作为最低要求：

- 是否必须登录
- 是否需要权限控制
- 是否已加 `@RequirePerm`
- 是否创建了对应菜单或按钮权限码
- 是否给角色完成授权
- 是否已在前端按钮加 `v-permission`

## 7. 推荐实践

- Controller 负责暴露接口，认证规则放在注解和服务层，不要把权限判断写成页面耦合逻辑。
- 登录方式扩展走策略工厂，权限扩展走菜单/按钮权限码，不要两边各自发明一套命名。
- 联调时先通登录，再通选租户，再看菜单按钮，最后看业务接口。
