# 认证授权实现逻辑

> 分类：后端 / 身份与权限
> 版本：**V0.6.0**
> 更新时间：**2026-04-17**

本文档说明 Forgex 后端认证授权链路在代码中的真实落点、核心调用顺序、权限装载方式与常见排查入口。它面向后端开发、联调开发、问题定位人员，重点回答 4 个问题：

1. 登录请求到底从哪里进入、往哪里走。
2. 选择租户后 token、权限、角色是如何落地的。
3. `@RequirePerm` 和 Sa-Token 的关系是什么。
4. 登录、无权限、按钮不显示时应该查哪里。

## 1. 模块定位

认证授权能力主要分布在以下模块中：

- `Forgex_Auth`
  负责登录、登出、注册、验证码、租户选择、Sa-Token 会话建立、权限与角色装载。
- `Forgex_Common`
  负责请求上下文透传，包括用户、租户、语言上下文。
- 业务模块 Controller
  通过 `@RequirePerm` 声明接口权限点。

认证模块并不是只做“账号密码校验”，它实际上包含两段链路：

1. `login`
   负责校验登录参数、调用对应登录策略、返回用户可选租户列表。
2. `chooseTenant`
   负责在用户已经确认身份后，真正建立当前租户下的登录态，并让 Sa-Token 可以拿到角色、权限、菜单。

## 3. 接口入口与职责分工

`AuthController` 是统一入口，当前已经明确暴露出以下关键接口：

- `POST /auth/login`
  用户登录，返回当前账号绑定的租户列表。
- `POST /auth/chooseTenant`
  用户选择租户，建立最终登录态。
- `POST /auth/captcha/image`
  获取图片验证码。
- `POST /auth/captcha/slider`
  获取滑块验证码。
- `POST /auth/captcha/slider/validate`
  校验滑块轨迹并返回校验令牌。
- `GET /auth/publicKey`
  获取前端传输加密公钥。
- `POST /auth/logout`
  用户登出。
- `POST /auth/register`
  邀请码注册。

职责边界如下：

- `AuthController`
  只负责接收入参、调用服务层、统一返回 `R<T>`。
- `AuthServiceImpl`
  负责编排认证流程，但登录方式和选租户方式都通过工厂分发，不在 Controller 中硬编码。
- `LoginStrategyFactory`
  按“终端 + 登录类型”选择对应登录策略。
- `ChooseTenantStrategyFactory`
  按终端选择对应的租户切换策略。
- `StpInterfaceImpl`
  负责给 Sa-Token 提供角色列表、权限列表。

## 4. 登录主链路

### 4.1 主流程

账号密码登录的真实调用顺序可以概括为：

```text
POST /auth/login
  -> AuthController.login(param)
  -> AuthServiceImpl.login(param)
  -> 标准化 terminal / type
  -> LoginStrategyFactory.getStrategy(loginTerminal, loginType)
  -> 对应策略.login(param)
  -> AuthServiceImpl.doAccountPasswordLogin(param)
  -> 校验验证码 / 解密密码 / 校验账号状态 / 校验密码策略
  -> 查询用户绑定租户
  -> 返回租户列表
```

### 4.2 `AuthServiceImpl.login()` 做了什么

`AuthServiceImpl.login()` 本身不直接写死登录实现，它主要负责：

- 从参数中解析登录终端 `loginTerminal`
- 解析登录方式 `loginType`
- 做归一化和默认值处理
- 调用 `loginStrategyFactory.getStrategy(loginTerminal, loginType).login(param)`

这意味着以后新增登录方式时，不应该把大量 `if/else` 塞回 `AuthServiceImpl.login()`，而应该扩展策略。

### 4.3 账号密码登录核心逻辑

账号密码登录实际落在 `AuthServiceImpl.doAccountPasswordLogin()`。这部分通常包含以下职责：

1. 验证验证码
   如果当前安全配置要求校验验证码，会先校验图片验证码或滑块验证码令牌。
2. 处理前端传输加密
   如果前端密码经过公钥加密，这里会先解密。
3. 查询用户
   根据账号查询用户记录，并校验是否存在。
4. 校验用户状态
   包括是否禁用、是否锁定、是否删除等。
5. 校验失败次数和锁定策略
   失败次数通常在 Redis 中累计，达到阈值后限制继续登录。
6. 校验密码
   按系统配置的密码算法进行比对。
7. 查询用户绑定租户
   当前系统不是登录后直接进入业务，而是先返回当前用户可访问的租户列表。
8. 返回租户列表
   前端拿到列表后调用 `chooseTenant` 完成正式登录。

### 4.4 验证码相关链路

验证码入口在 `AuthController`，实际能力在 `CaptchaService`：

- 图片验证码：`/auth/captcha/image`
- 滑块验证码生成：`/auth/captcha/slider`
- 滑块验证码校验：`/auth/captcha/slider/validate`

排查验证码问题时，一般从 3 个点入手：

1. 前端提交的 `captchaId` 是否和服务端生成的 ID 对应。
2. 滑块模式下，是否先调用过校验接口拿到二次校验 token。
3. Redis 或验证码缓存是否因为过期时间过短导致校验失败。

### 4.5 公钥获取与密码传输加密

`GET /auth/publicKey` 用于前端获取密码传输加密公钥。典型使用方式是：

1. 登录页加载时请求公钥。
2. 前端用公钥加密密码。
3. 后端在登录链路中解密后再做密码比对。

如果出现“密码明明正确但总是失败”，除了查数据库中的密码算法外，还要查：

- 前端是否真的使用了最新公钥。
- 解密逻辑与前端加密算法是否匹配。
- 登录参数里的加密标识有没有传错。

## 5. 选租户主链路

### 5.1 为什么要分成两步

Forgex 的登录态是“用户 + 当前租户”组合而成的，因此先登录得到租户列表，再选择租户建立最终会话。

这可以避免以下问题：

- 一个账号对应多个租户时，后端不知道应该进入哪个租户。
- 菜单、角色、权限都与当前租户强相关，必须在租户确定后才能装载。

### 5.2 调用顺序

```text
POST /auth/chooseTenant
  -> AuthController.chooseTenant(param)
  -> AuthServiceImpl.chooseTenant(param)
  -> ChooseTenantStrategyFactory.getStrategy(loginTerminal)
  -> 对应策略.chooseTenant(param)
  -> AuthServiceImpl.doChooseTenant(param)
  -> 校验用户是否属于目标租户
  -> StpUtil.login(...)
  -> 写 SaSession
  -> 记录当前 token / loginId / tenantId
  -> 返回当前租户下的用户信息
```

### 5.3 `doChooseTenant()` 的关键动作

`AuthServiceImpl.doChooseTenant()` 是真正建立登录态的核心方法，一般会做以下事情：

1. 校验租户归属
   防止用户选择不属于自己的租户。
2. 组装登录标识
   登录标识中通常会带上用户与租户信息，便于后续按租户区分会话。
3. 调用 `StpUtil.login(idKey)`
   正式建立 Sa-Token 登录态。
4. 获取并写入 `SaSession`
   将当前用户、当前租户、语言等会话信息写入 session。
5. 获取 token
   通过 `StpUtil.getTokenValue()` 返回给前端。
6. 返回当前用户 DTO
   前端再依据用户信息、菜单接口和按钮权限渲染页面。

### 5.4 会话建立后谁来提供权限

会话建立后，Sa-Token 在做权限校验时会回调 `StpInterfaceImpl`：

- `getPermissionList(Object loginId, String loginType)`
  返回当前登录态对应的权限码集合。
- `getRoleList(Object loginId, String loginType)`
  返回当前登录态对应的角色编码集合。

这两个方法是 `@RequirePerm` 能够工作的前提。

## 6. 权限模型与 `@RequirePerm`

### 6.1 核心关系

接口权限并不是直接硬编码在 Controller 中的布尔值，而是通过以下链路完成：

```text
Controller 方法标注 @RequirePerm("module:resource:action")
  -> 权限拦截器读取注解
  -> Sa-Token 检查当前登录态权限列表
  -> StpInterfaceImpl.getPermissionList(...)
  -> 从角色、菜单、按钮权限关系中装载权限码
  -> 命中则放行，不命中则拦截
```

### 6.2 业务接口如何受保护

集成平台模块的 Controller 已经是典型用法，例如：

- `integration:third-system:view`
- `integration:third-system:add`
- `integration:api-config:add`
- `integration:api-call-log:view`

只要接口上声明了 `@RequirePerm`，并且当前用户所属角色拥有对应按钮或菜单权限，接口就能通过。

### 6.3 为什么会出现“页面按钮不显示”

前端按钮是否显示，通常依赖与后端一致的权限码。例如接口配置页面：

- 后端新增接口使用的是 `@RequirePerm("integration:api-config:add")`
- 前端新增按钮使用的是 `v-permission="'integration:api-config:add'"`

这两边任何一边写错，都会出现：

- 页面按钮不显示
- 或者按钮显示但点击报无权限

## 7. 注册链路

`POST /auth/register` 的核心目标不是只创建用户，而是把用户和租户关系初始化好。典型流程如下：

```text
POST /auth/register
  -> 校验参数、验证码、邀请码
  -> 校验账号是否重复
  -> 按密码策略加密密码
  -> 写入用户表
  -> 写入用户租户关系表
  -> 初始化角色/部门/岗位等默认关系
  -> 返回注册结果
```

邀请码在这个链路中通常不只是“允许注册”的凭证，它还承载：

- 默认租户
- 默认角色
- 默认部门
- 默认岗位

因此注册失败时，不要只看 `sys_user`，还要查邀请码绑定关系。

## 8. 与租户上下文的关系

认证模块与租户隔离并不是完全独立的：

- 选择租户后，请求期会通过拦截器把 `tenantId` 写入 `TenantContext`
- 业务 SQL 依赖这个上下文做自动租户隔离
- 权限查询、菜单查询、用户信息查询都依赖当前租户

## 9. 常见排查

### 9.1 登录接口返回成功但选租户后仍然未登录

优先检查：

1. `AuthServiceImpl.doChooseTenant()` 是否真正执行到 `StpUtil.login(...)`
2. token 是否成功返回给前端
3. 网关是否透传了 token
4. `UserTenantWebInterceptor` 是否正确把用户、租户上下文写入当前请求

### 9.2 登录成功但接口全都报无权限

优先检查：

1. `StpInterfaceImpl.getPermissionList()` 是否查到了权限码
2. 当前租户下角色和菜单、按钮关系是否完整
3. Controller 上 `@RequirePerm` 写的权限码是否与菜单按钮权限码一致
4. 是否误用了错误租户登录

### 9.3 页面按钮不显示

优先检查：

1. 前端 `v-permission` 的权限码是否和后端一致
2. 当前角色是否拥有该按钮权限
3. 当前菜单树是否正确下发到前端
4. 语言包是否只影响显示文本，而不是把权限码当菜单名显示

### 9.4 账号被锁定

优先检查：

1. Redis 中失败次数是否未清理
2. 锁定窗口配置是否过长
3. 前端是否因为密码传输加密异常导致服务端持续认为密码错误

### 9.5 新接口忘记接权限

这是后端开发最容易漏掉的问题之一。判断方法很简单：

1. 查 Controller 方法是否加了 `@RequirePerm`
2. 查菜单/按钮是否创建了对应权限码
3. 查前端按钮或页面是否使用相同权限码

如果只做了第 2、3 步，没有做第 1 步，那么接口仍然可能在后端“裸奔”。

## 10. 开发建议

- 新增受保护接口时，优先先定义权限码，再写 Controller 注解，再写前端按钮权限。
- 新增登录方式时，优先扩展策略，不要改成巨型分支逻辑。
- 所有认证相关接口统一返回 `R<T>`，保持前后端联调稳定。
- 登录问题先看 `login`，权限问题先看 `chooseTenant + StpInterfaceImpl`，不要混在一起排查。
