# 认证授权

> 分类：后端 / 身份与权限  
> 版本：**V0.5.0**  
> 更新时间：**2026-04-13**

## 一、概述

Forgex 认证授权模块（`Forgex_Auth`）负责用户登录、注册、登出、权限校验、验证码等核心安全功能。采用 **策略模式** 实现多终端、多方式登录。

## 二、登录流程

### 2.1 登录接口

```
POST /auth/login
```

**请求参数 `LoginParam`：**

| 字段 | 类型 | 说明 |
|---|---|---|
| `loginTerminal` | String | 登录终端：`B`(后台) / `C`(客户端/安卓) |
| `loginType` | String | 登录方式：`ACCOUNT_PASSWORD` / `WECHAT` / `DINGTALK` |
| `account` | String | 账号 |
| `password` | String | 密码（可能经过 SM2 加密） |
| `captcha` | String | 验证码内容 |
| `captchaId` | String | 验证码 ID（图片验证码时使用） |

### 2.2 登录策略模式

系统通过 `LoginStrategyFactory` 根据终端和方式分发到不同策略：

| 策略类 | 终端 | 方式 |
|---|---|---|
| `BAccountPasswordLoginStrategy` | B 端 | 账号密码 |
| `CAccountPasswordLoginStrategy` | C 端 | 账号密码 |
| 微信/钉钉策略 | B/C | 第三方登录 |

### 2.3 账号密码登录详细流程

```
1. 获取账号与密码
2. 校验账号是否为空
3. 查询用户信息（sys_user 表跨租户查询）
4. 校验账号是否存在、是否禁用
5. 检查账号是否被锁定（Redis 记录失败次数）
6. 密码传输解密（如果配置了 SM2 加密传输）
7. 密码校验（通过 CryptoProviders 支持 bcrypt/SM2/SM4/AES/RSA）
8. 验证码校验（支持无/图片/滑块三种模式）
9. 清除登录失败状态
10. 查询用户绑定的租户列表
11. 返回租户列表（前端选择租户后调用 chooseTenant）
```

### 2.4 租户选择

```
POST /auth/chooseTenant
```

用户选择租户后：
1. 生成 Token（Sa-Token）
2. 将用户信息写入 Redis Session
3. 记录登录成功日志（异步）
4. 返回用户信息与权限

### 2.5 验证码配置

验证码通过 `sys_config` 表的 `login.captcha` 键配置：

```json
{
  "mode": "image",           // none / image / slider
  "keyPrefix": "captcha:",
  "expireSeconds": 120,
  "width": 130,
  "height": 48,
  "length": 4
}
```

| 验证码模式 | 说明 |
|---|---|
| `none` | 不需要验证码 |
| `image` | 图片验证码，需传 captchaId + captcha |
| `slider` | 滑块验证码，需传滑块 token |

**验证码相关接口：**

```
GET  /auth/captcha/image       → 获取图片验证码（返回 base64 + captchaId）
GET  /auth/captcha/slider      → 获取滑块验证码（返回背景图 + 滑块图）
POST /auth/captcha/slider/validate → 校验滑块（返回 token）
GET  /auth/captcha/config      → 获取当前验证码模式配置
```

### 2.6 登录安全策略

通过 `security.login` 配置键管理：

| 配置项 | 说明 |
|---|---|
| `maxFailCount` | 允许的最大连续失败次数 |
| `lockMinutes` | 锁定时长（分钟） |
| `failWindowMinutes` | 失败计数窗口（分钟） |

失败状态存储在 Redis 中：`login:fail:count:{account}`。

### 2.7 密码传输加密

如果系统配置了 `security.crypto.transport`（SM2 算法），前端会用公钥加密密码后传输：

```
前端: password = SM2.encrypt(rawPassword, publicKey)
后端: rawPassword = SM2.decrypt(password, privateKey)
```

公钥获取接口：
```
GET /auth/publicKey
```

## 三、注册流程

### 3.1 注册接口

```
POST /auth/register
```

**请求参数 `RegisterParam`：**

| 字段 | 类型 | 必填 | 说明 |
|---|---|---|---|
| `account` | String | 是 | 注册账号 |
| `password` | String | 是 | 密码（可 SM2 加密） |
| `username` | String | 否 | 显示名 |
| `inviteCode` | String | 是 | 邀请码 |
| `phone` | String | 否 | 手机号 |
| `email` | String | 否 | 邮箱 |
| `captcha` | String | 按配置 | 验证码 |
| `captchaId` | String | 按配置 | 验证码 ID |

### 3.2 注册详细流程

```
1. 参数校验（账号、密码、邀请码非空）
2. 验证码校验（根据 login.captcha 配置）
3. 校验邀请码：
   - 邀请码是否存在（sys_invite_code 表）
   - 邀请码是否启用
   - 邀请码是否过期
   - 邀请码是否超过最大注册数
4. 校验账号是否已存在
5. 密码传输解密（如配置 SM2）
6. 密码加密存储（按 PasswordPolicy 策略）
7. 创建用户（sys_user 表）
8. 关联部门和职位（如邀请码配置了 departmentId）
9. 创建用户与租户绑定（sys_user_tenant 表）
10. 更新邀请码使用次数
11. 分配默认角色（如邀请码配置了 roleIds）
12. 返回成功
```

### 3.3 邀请码管理

邀请码存储在 `sys_invite_code` 表：

| 字段 | 说明 |
|---|---|
| `invite_code` | 邀请码字符串 |
| `tenant_id` | 绑定的租户 |
| `department_id` | 注册用户自动关联的部门 |
| `position_id` | 注册用户自动关联的职位 |
| `role_ids` | 注册用户自动分配的角色 |
| `max_register_count` | 最大可注册数量 |
| `used_count` | 已使用次数 |
| `expire_time` | 过期时间 |
| `status` | 是否启用 |

### 3.4 注册入口控制

在系统基础配置中可控制是否显示注册入口：

```json
// sys_config: system.basic
{
  "showRegisterEntry": true,
  "registerUrl": "/register"
}
```

## 四、登出

```
POST /auth/logout
```

登出流程：
1. 获取当前 Token
2. 回写登录日志的登出时间
3. 清除 Redis 中的 Session
4. Sa-Token 注销当前登录

## 五、授权配置

### 5.1 权限模型

Forgex 采用 **模块 → 菜单 → 按钮** 三级权限模型：

```
模块 (sys_module)
  └── 菜单 (sys_menu, type=menu)
        ├── 页面按钮 (sys_menu, type=button)
        └── 子菜单
```

### 5.2 权限注解 `@RequirePerm`

```java
@Target({ElementType.METHOD, ElementType.TYPE})
@Retention(RetentionPolicy.RUNTIME)
public @interface RequirePerm {
    String[] value();  // 需要的权限键列表
}
```

**使用示例：**

```java
@RequirePerm("sys:user:list")
@PostMapping("/page")
public R<IPage<SysUserVO>> page(@RequestBody SysUserQueryDTO dto) {
    // ...
}

@RequirePerm("sys:user:edit")
@PostMapping("/save")
public R<Boolean> save(@RequestBody SysUserDTO dto) {
    // ...
}
```

### 5.3 权限校验流程

```
请求 → PermissionInterceptor
  → 检查方法/类上是否有 @RequirePerm
  → 从上下文获取当前用户角色权限列表
  → 校验用户是否拥有所需权限
  → 无权限则返回 R.fail(StatusCode.UNAUTHORIZED, ...)
```

### 5.4 权限键命名规范

```
格式：{模块}:{资源}:{操作}
示例：
  sys:user:list        → 用户列表
  sys:user:edit        → 编辑用户
  sys:user:delete      → 删除用户
  sys:role:assign      → 角色分配
  sys:excel:exportConfig:list → Excel 导出配置列表
```

### 5.5 角色与权限分配

- **角色绑定用户**：通过 `sys_user_role` 关联表
- **角色绑定菜单权限**：通过 `sys_role_menu` 关联表
- **数据权限范围**：角色可配置数据范围（全部/本部门/本人等）

### 5.6 动态路由

登录选择租户后，后端返回用户可访问的菜单树，前端据此生成动态路由：

```
GET /sys/menu/routes?account=xxx
→ 返回用户在当前租户下的模块、菜单、按钮权限树
→ 前端据此渲染侧边栏与按钮可见性
```

## 六、密码策略

通过 `sys_config` 中 `security.password.policy` 配置：

| 配置项 | 默认值 | 说明 |
|---|---|---|
| `store` | `bcrypt` | 密码存储算法：bcrypt / sm2 / sm4 / aes / rsa |
| `defaultPassword` | `Aa123456` | 管理员重置密码时使用的默认密码 |
