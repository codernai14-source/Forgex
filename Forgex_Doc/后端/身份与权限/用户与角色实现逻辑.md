# 用户与角色实现逻辑

> 分类：后端 / 身份与权限  
> 版本：**V0.6.0**  
> 更新时间：**2026-04-17**

本文档说明用户、角色、部门、岗位以及权限绑定在 Forgex 后端中的实现关系。

## 领域对象

| 对象 | 说明 |
|---|---|
| 用户 `sys_user` | 账号主体、登录基础信息 |
| 角色 `sys_role` | 权限集合载体 |
| 用户角色关系 `sys_user_role` | 用户与角色多对多绑定 |
| 角色菜单关系 `sys_role_menu` | 角色与菜单、按钮权限绑定 |
| 用户租户关系 `sys_user_tenant` | 用户可访问的租户集合 |

## 用户管理核心链路

```text
用户新增/更新
  -> 参数校验
  -> 校验账号唯一性
  -> 按密码策略加密默认密码或新密码
  -> 保存 sys_user
  -> 保存部门、岗位、角色关系
  -> 保存用户租户关系
```

## 角色授权链路

```text
角色保存
  -> 保存角色基本信息
  -> 绑定 sys_role_menu
  -> 角色授权范围生效
  -> 用户下次登录或刷新权限后获得新权限
```

角色本身不直接决定登录，真正生效是在用户绑定角色后，由权限装载逻辑将角色菜单、按钮、数据范围汇总成当前用户权限集合。

## 角色新增与编辑

角色管理页新增按钮固定显示为“新增角色”，打开表单后由前端按新增/编辑状态分别调用 `/sys/role/create` 与 `/sys/role/update`。

前端表单字段使用 `roleCode` 表示角色编码；后端保存 `sys_role` 时需要将 `roleCode` 映射到实体字段 `roleKey`，最终落库到 `sys_role.role_key`。查询返回时继续将 `roleKey` 映射回 `roleCode`，保持前后端字段口径一致。

## 用户状态与密码

用户一般包含以下关键状态：

- 启用 / 禁用
- 默认租户
- 默认角色集合
- 密码更新时间
- 删除标记

密码重置、修改密码与管理员初始化密码最终都受 `security.password.policy` 控制。

## 与注册流程的关系

邀请码注册完成后，往往会同步初始化：

1. `sys_user`
2. `sys_user_tenant`
3. 默认部门 / 岗位
4. 默认角色

因此“注册”只是入口，“用户与角色”才是完整的身份归属落点。

## 与权限体系的关系

用户和角色的关系主要体现在三层：

1. 用户是否能登录，由用户状态与密码决定。
2. 用户登录后能进入哪些菜单，由角色菜单决定。
3. 用户能点哪些按钮、调用哪些接口，由按钮权限键与 `@RequirePerm` 共同决定。
