# 多租户实现逻辑

> 分类：后端 / 租户与上下文
> 版本：**V0.6.5**
> 更新时间：**2026-04-17**

本文档说明 Forgex 多租户能力在后端的真实实现方式，重点说明：

1. 当前租户和当前用户上下文存放在哪里。
2. 请求期间上下文是怎么写入和清理的。
3. MyBatis-Plus 如何自动补租户字段和审计字段。
4. 哪些表、服务、Mapper 会被显式忽略租户隔离。
5. 异步线程为什么容易丢租户上下文。

## 1. 设计目标

Forgex 采用共享库多租户模式，核心目标不是“每张表都单独分库”，而是让绝大多数业务 SQL 自动带上当前租户条件。

系统希望达成以下效果：

1. 默认所有租户业务数据都按 `tenant_id` 隔离。
2. 插入数据时尽量自动补全 `tenant_id`、`create_by`、`update_by`、时间等字段。
3. 公共配置允许使用 `tenant_id = 0` 作为全局默认。
4. 少数必须跨租户访问的表和方法必须显式登记为忽略规则，不能偷偷绕过。

## 3. 上下文模型

### 3.1 `TenantContext`

`TenantContext` 使用 `ThreadLocal<Long>` 保存当前线程的租户 ID，对外提供：

- `set(Long tenantId)`
- `get()`
- `clear()`

它的作用是让当前请求在进入 Service、Mapper、MyBatis 插件时都能拿到当前租户。

### 3.2 `UserContext`

`UserContext` 同样使用 `ThreadLocal<Long>` 保存当前线程用户 ID，对外提供：

- `set(Long userId)`
- `get()`
- `clear()`

它主要用于自动审计字段填充，例如：

- `createBy`
- `updateBy`

### 3.3 `TenantContextIgnore`

`TenantContextIgnore` 是一个临时开关，用于告诉当前线程“这段逻辑暂时不要走租户隔离”。它通常只应该被切面或极少量受控代码使用，不应该在业务代码里随意滥用。

## 4. 请求期上下文链路

### 4.1 写入链路

请求期间，用户和租户信息通常由网关或认证信息透传到微服务，请求到达服务后由拦截器写入上下文：

```text
网关透传用户、租户、语言信息
  -> UserTenantWebInterceptor.preHandle(...)
  -> UserContext.set(uid)
  -> TenantContext.set(tid)
  -> 后续 Service / Mapper / MyBatis 插件读取上下文
```

`UserTenantWebInterceptor` 是业务真正依赖的上下文写入点，它会把请求头中的用户与租户信息写入当前线程。

### 4.2 清理链路

请求结束后必须清理上下文，否则线程复用时容易造成串租户、串用户。

当前清理主要由以下位置完成：

- `TenantContextInterceptor.afterCompletion(...)`
  调用 `TenantContext.clear()`
- `UserTenantWebInterceptor` 中的清理逻辑
  清理 `UserContext` 和 `TenantContext`

正确认知是：

- 写入依赖请求拦截器
- 清理也依赖请求拦截器
- `ThreadLocal` 不会自动清空

## 5. SQL 隔离与自动填充

### 5.1 默认隔离思路

当业务表接入多租户后，查询、更新、删除通常都应自动按当前 `tenant_id` 过滤。插入时则由自动填充逻辑补 `tenantId`。

这意味着业务开发时通常不需要手动每次都写：

```sql
where tenant_id = 当前租户
```

但前提是：

- 当前线程已经有正确的 `TenantContext`
- 该表没有被配置为忽略租户
- 实体字段命名与自动填充约定一致

### 5.2 `TenantMetaObjectHandler`

`TenantMetaObjectHandler` 是多租户和审计字段自动填充的关键类。

插入时会尝试填充：

- `tenantId`
- `createBy`
- `updateBy`
- `createTime`
- `updateTime`
- `deleted`

更新时会尝试填充：

- `updateTime`
- `updateBy`

它依赖两个上下文：

- `TenantContext.get()`
- `UserContext.get()`

因此，如果新增数据时发现：

- `tenant_id` 为空
- `create_by` 为空
- `update_by` 为空

首先不要怀疑实体类，先看当前线程上下文是否已经写入。

## 6. 忽略租户机制

### 6.1 为什么需要忽略

不是所有表都应该按租户隔离。例如：

- 用户基础表
- 租户基础表
- 用户租户关系表
- 系统级配置表
- 部分全局国际化和表格配置表

这类表如果强行加上当前租户过滤，反而会导致登录、选租户、全局配置读取异常。

### 6.2 忽略规则由谁维护

忽略规则核心由 `TenantIgnoreRegistry` 维护，并由 `TenantIgnoreAspect` 在运行时决定是否打开 `TenantContextIgnore`。

支持的粒度包括：

- 表级忽略
- Service 级忽略
- Mapper 方法级忽略

### 6.3 当前默认忽略的典型对象

根据现有代码，默认忽略对象已经包含以下典型表：

- `sys_user`
- `sys_tenant`
- `sys_user_tenant`
- `sys_config`
- `sys_tenant_ignore`
- `fx_i18n_message`
- `fx_table_config`
- `fx_table_column_config`

这些表的共同特点是：

- 它们是系统基础能力表，不能简单按当前租户硬切
- 它们常常需要支持“当前租户优先，公共配置兜底”的模式

### 6.4 切面如何生效

`TenantIgnoreAspect` 会在命中忽略规则时：

1. 打开 `TenantContextIgnore`
2. 执行目标逻辑
3. 最终 `clear()`

因此忽略租户是一个受控的、带作用域的开关，不应该当成全局状态长期保留。

## 7. 公共配置 `tenant_id = 0` 回退模型

Forgex 中不少配置能力使用“租户配置优先、公共配置兜底”的模式，公共配置用 `tenant_id = 0` 表示。

典型读取顺序如下：

```text
先查当前 tenant_id
  -> 查不到时再查 tenant_id = 0
  -> 仍然查不到时才使用代码默认值
```

这个模式常见于：

- 系统配置
- 国际化消息
- 动态表格配置
- 消息模板类配置

这也是为什么这些表经常出现在忽略租户名单中。它们并不是“完全不区分租户”，而是由业务自行实现双层查询逻辑。

## 8. 异步线程中的上下文问题

### 8.1 为什么容易丢

`TenantContext` 和 `UserContext` 都基于 `ThreadLocal`。一旦进入：

- `@Async`
- 自定义线程池
- `CompletableFuture`

新线程默认不会继承当前请求线程中的上下文。

### 8.2 典型后果

如果异步任务里直接操作数据库，常见结果包括：

- 插入时 `tenant_id` 没被自动填充
- 审计字段 `create_by/update_by` 为空
- 查询时查不到当前租户数据
- 甚至误查到公共表数据

### 8.3 正确处理方式

进入异步线程前先取出上下文：

```java
Long tenantId = TenantContext.get();
Long userId = UserContext.get();
```

进入异步线程后重新设置：

```java
TenantContext.set(tenantId);
UserContext.set(userId);
try {
    // 执行业务
} finally {
    UserContext.clear();
    TenantContext.clear();
}
```

只复制 `tenantId` 不复制 `userId` 也不够，因为审计字段会依赖 `UserContext`。

## 9. 典型业务案例

### 9.1 集成平台

集成平台中的第三方系统与接口配置是租户级数据，因此：

- `ThirdSystemServiceImpl` 会读取 `TenantContext.get()`
- `ApiConfigServiceImpl` 依赖租户上下文或自动填充处理租户字段

而调用日志又会结合异步写入，因此更加需要注意上下文复制。

### 9.2 认证授权

认证阶段的用户、租户、配置表查询中，会频繁命中忽略租户规则。否则登录前根本拿不到基础用户和租户关系。

## 10. 常见排查

### 10.1 查询不到数据

优先检查：

1. 当前请求是否真正写入了 `TenantContext`
2. 当前租户是否正确
3. 表是否不小心被误加入忽略名单
4. SQL 是否又手动拼了一层错误租户条件

### 10.2 查到了不该看的数据

优先检查：

1. 是否误开了 `TenantContextIgnore`
2. `TenantIgnoreRegistry` 是否配置错误
3. Mapper 或 Service 是否被错误登记为忽略

### 10.3 新增数据 `tenant_id` 为空

优先检查：

1. `TenantContext` 是否为空
2. 实体字段是否命名为 `tenantId`
3. `TenantMetaObjectHandler` 是否生效
4. 是否手动 set 了空值把自动填充覆盖掉

### 10.4 审计字段为空

优先检查：

1. `UserContext` 是否已写入
2. 是否进入了异步线程但没有复制上下文
3. 实体字段名称是否与自动填充约定一致

### 10.5 异步任务串租户或丢租户

优先检查：

1. 异步前是否取出了租户和用户上下文
2. 异步线程里是否重新 `set`
3. 任务结束后是否 `clear`

## 11. 开发建议

- 默认所有业务表都按租户接入，不要因为开发方便就先跳过。
- 公共配置统一使用 `tenant_id = 0` 模式，不要用 `null` 表示公共数据。
- 忽略租户必须可登记、可审计、可复查，不能靠散落在业务代码里的开关。
- 任何异步数据库操作都要先想清楚上下文复制问题。
