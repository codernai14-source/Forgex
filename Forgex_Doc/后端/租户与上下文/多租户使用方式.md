# 多租户使用方式

> 分类：后端 / 租户与上下文
> 版本：**V0.6.0**
> 更新时间：**2026-04-17**

本文档面向后端开发与联调人员，说明多租户能力在实际开发中的接入方式、代码写法、页面联调注意点与排查方法。

## 1. 后端代码如何使用

### 1.1 新业务表如何接入多租户

新建业务表时，如果它属于租户业务数据，建议至少具备以下字段：

- `tenant_id`
- `create_by`
- `create_time`
- `update_by`
- `update_time`
- `deleted`

原因如下：

- `tenant_id`
  用于数据隔离。
- `create_by/update_by`
  便于追踪是谁创建和修改。
- `create_time/update_time`
  便于排序、审计、问题定位。
- `deleted`
  适配逻辑删除模型。

代码侧配合要求：

1. 实体字段命名与自动填充约定保持一致。
2. 不要在插入前主动把这些字段 set 成错误值。
3. 保证请求进入 Service 前已经建立好 `TenantContext` 和 `UserContext`。

### 1.2 查询代码如何写

大多数场景下，业务代码不需要手动拼租户条件，应该依赖系统上下文与多租户插件自动处理。

推荐写法：

```java
LambdaQueryWrapper<DemoOrder> wrapper = Wrappers.lambdaQuery();
wrapper.eq(DemoOrder::getStatus, "ENABLED");
return this.list(wrapper);
```

不推荐一上来就手工写：

```java
wrapper.eq(DemoOrder::getTenantId, TenantContext.get());
```

除非当前场景明确需要自己控制双层查询或跨租户逻辑。否则手写容易出现：

- 重复拼接
- 当前线程上下文为空时误判
- 后续忽略租户场景下逻辑冲突

### 1.3 新增或更新代码如何写

插入和更新时，优先依赖 `TenantMetaObjectHandler` 自动补字段。

推荐做法：

```java
DemoOrder entity = new DemoOrder();
entity.setOrderCode(param.getOrderCode());
entity.setOrderName(param.getOrderName());
this.save(entity);
```

一般不需要手动写：

```java
entity.setTenantId(TenantContext.get());
entity.setCreateBy(String.valueOf(UserContext.get()));
```

只有在特殊场景下，例如某些对象不走通用自动填充、或者必须显式覆盖时，才建议手工赋值。

### 1.4 公共配置如何写

如果你新增的是系统级公共配置表，建议采用：

- 当前租户配置优先
- `tenant_id = 0` 兜底

典型写法思路：

1. 先按当前租户查询
2. 查不到再查 `tenant_id = 0`
3. 还查不到再用代码默认值

这类表通常不要简单依赖自动租户隔离直查，否则会把公共配置挡掉。

### 1.5 什么时候可以忽略租户

满足以下条件之一，才考虑忽略租户：

1. 这是系统级基础表，不属于某个租户独占数据。
2. 登录前就必须访问，否则认证链路无法启动。
3. 这是公共配置表，需要自己实现“当前租户优先 + 全局兜底”。
4. 这是跨租户维护能力，并且已经明确授权和审计。

不建议因为“查不到数据”就直接把表加到忽略名单里。先确认当前请求上下文是否正确，再决定是否忽略。

### 1.6 如何配置忽略规则

当前忽略规则由 `TenantIgnoreRegistry` 与 `TenantIgnoreAspect` 共同控制。接入时的建议是：

1. 先确认是表级、Service 级还是 Mapper 方法级问题。
2. 优先使用最小影响范围的忽略粒度。
3. 留下注释或文档，说明为什么必须忽略。
4. 上线后复查是否还有忽略必要。

开发原则：

- 能不忽略就不忽略
- 能缩小范围就不要放大到整张表或整个 Service

### 1.7 异步任务如何复制上下文

异步任务只要访问数据库，就要手动复制上下文。推荐模板如下：

```java
Long tenantId = TenantContext.get();
Long userId = UserContext.get();

executor.execute(() -> {
    TenantContext.set(tenantId);
    UserContext.set(userId);
    try {
        demoService.doSomething();
    } finally {
        UserContext.clear();
        TenantContext.clear();
    }
});
```

这是开发里最容易漏的点之一。只要一漏，就会出现：

- 数据新增没有 `tenant_id`
- 更新人字段为空
- 查询结果不对

## 2. 页面如何联调用多租户

### 2.1 页面为什么几乎不直接操作租户字段

大多数前端页面不会直接传 `tenantId` 作为业务查询条件，而是通过登录和选租户后由 token、网关、拦截器自动建立上下文。

所以页面联调重点不是“有没有传 `tenantId` 字段”，而是：

1. 当前用户是否已经正确选中租户登录
2. token 是否透传成功
3. 后端上下文是否已经写入

### 2.2 页面切换租户后的预期

切换租户后，同一个页面重新请求接口时，理论上应表现为：

- 列表数据切换为目标租户数据
- 菜单和按钮权限随租户变化
- 当前租户没有的数据不会再显示

如果页面切换租户后数据没变化，不要先查页面缓存，先查后端 `TenantContext` 是否真的变化了。

## 3. 开发接入清单

新功能开发时，可以用下面这张清单自检：

- 这张表是不是租户级业务表
- 是否已包含 `tenant_id`
- 实体字段是否支持自动填充
- 是否存在公共配置兜底需求
- 是否真的需要忽略租户
- 是否存在异步落库
- 异步代码是否复制并清理上下文

## 4. 常见问题排查

### 4.1 查不到数据

按以下顺序查：

1. 当前用户是否选中了正确租户
2. 请求头中的租户信息是否透传
3. `UserTenantWebInterceptor` 是否写入了 `TenantContext`
4. 表是否本应忽略租户却没有忽略
5. 是否手工写了错误的 `tenant_id` 条件

### 4.2 查到了其他租户的数据

按以下顺序查：

1. 是否误命中了忽略规则
2. 是否手工打开了 `TenantContextIgnore`
3. Mapper 方法是否被错误登记为忽略
4. 是否查询了本就属于公共配置的表

### 4.3 新增数据 `tenant_id` 没写进去

按以下顺序查：

1. 当前线程里 `TenantContext.get()` 是否为空
2. `TenantMetaObjectHandler` 是否生效
3. 实体字段是否叫 `tenantId`
4. 是否在 save 前手工覆盖了空值
5. 是否在异步线程中执行且没复制上下文

### 4.4 更新人、创建人为空

按以下顺序查：

1. `UserContext.get()` 是否为空
2. 当前请求是否经过统一上下文拦截器
3. 是否进入了异步线程
4. 字段命名是否和自动填充约定一致

### 4.5 公共配置读不出来

按以下顺序查：

1. 查询逻辑是否实现了 `tenant_id = 0` 兜底
2. 该表是否被多租户插件自动拦截
3. 是否需要加入忽略租户名单
4. 当前租户数据是否覆盖了公共数据

## 5. 推荐实践

- 业务表默认接租户，不要先写成单租户再补救。
- 公共配置统一用 `tenant_id = 0`，这样后续排查路径最统一。
- 忽略租户必须最小化使用，并且要能说明原因。
- 异步任务凡是会查库、写库的，都默认先处理上下文复制。
