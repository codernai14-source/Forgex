# 后端公共能力与核心功能手册

> 版本：**预览版 V0.5.0**  
> 更新时间：**2026-04-13**

本文档聚焦 V0.5.0 版本中最值得优先复用的后端公共能力，说明它们的定位、实现逻辑与推荐使用方式。

## 一、统一返回模型：`R<T>`

### 1.1 定位

`R<T>` 是后端统一响应包装类，位于：

- `Forgex_Common/src/main/java/com/forgex/common/web/R.java`

它用于统一返回：

- `code`
- `message`
- `data`
- 内部国际化元信息

### 1.2 推荐用法

| 场景 | 推荐写法 |
|---|---|
| 查询成功并返回数据 | `R.ok(data)` |
| 写操作成功并返回提示 | `R.ok(CommonPrompt.UPDATE_SUCCESS)` |
| 写操作成功且提示带参数 | `R.okWithArgs(CommonPrompt.SAVE_SUCCESS, "用户信息")` |
| 业务失败 | `R.fail(CommonPrompt.OPERATION_FAILED)` |
| 特定状态码失败 | `R.fail(StatusCode.NOT_FOUND, CommonPrompt.NOT_FOUND)` |

### 1.3 实现逻辑

1. Controller 返回 `R`。
2. `messageCode` / `i18n` 保留内部翻译元信息。
3. `RMessageI18nAdvice` 在响应写回前翻译 `message`。
4. 前端 `http.ts` 根据 `code/message` 自动提示。

## 二、异常体系：`BusinessException` 与 `I18nBusinessException`

### 2.1 定位

统一异常体系的目标是：

- 让 Service 层专注抛业务异常；
- 让 Controller 层只负责返回结果；
- 让国际化提示与错误码可统一管理。

### 2.2 推荐使用建议

| 场景 | 建议 |
|---|---|
| 普通业务校验失败 | 抛 `BusinessException` |
| 需要国际化提示 | 抛 `I18nBusinessException` |
| 需要状态码 + 多语言模板 | 使用 `I18nBusinessException(code, prompt, args...)` |

### 2.3 实现逻辑

- Service 抛出异常；
- 全局异常处理器捕获并转换为 `R.fail(...)`；
- 若使用国际化异常，则最终仍回到 `R` 的翻译链路。

## 三、国际化链路：`LangContext` + `RMessageI18nAdvice`

### 3.1 核心角色

| 组件 | 作用 |
|---|---|
| `LangContext` | 保存当前请求语言上下文 |
| `LangWebInterceptor` | 从请求头解析语言并写入上下文 |
| `I18nMessageService` | 根据 `module + promptCode` 解析多语言文案 |
| `RMessageI18nAdvice` | 响应出参翻译 |

### 3.2 推荐接入方式

1. 枚举实现 `I18nPrompt`；
2. Controller / Service 中使用 `R.ok(prompt)`、`R.fail(prompt)` 或抛 `I18nBusinessException`；
3. 数据库维护 `fx_i18n_message` 与语言类型配置。

### 3.3 适用范围

- 公共提示消息
- 模块提示消息
- 参数化模板消息
- 站内消息、字典标签、前端成功提示

## 四、多租户与上下文能力

### 4.1 核心类

| 类 / 组件 | 作用 |
|---|---|
| `TenantContext` | 当前线程租户上下文 |
| `UserContext` | 当前线程用户上下文 |
| `TenantContextIgnore` | 当前调用临时忽略租户隔离 |
| `TenantIgnoreRegistry` | 动态管理忽略租户规则 |
| `TenantMetaObjectHandler` | 自动填充租户与审计字段 |

### 4.2 设计结论

- 默认所有租户业务都应经过租户隔离。
- 公共配置统一使用 `tenant_id = 0`。
- 忽略租户隔离必须是显式、可审计、可收敛的，不应靠“空租户兜底跳过”。

### 4.3 常见使用场景

- 查询当前租户数据：直接从上下文获取 `TenantContext.get()`。
- 写入审计字段：依赖元对象处理器自动填充。
- 公共配置：通过 `tenant_id=0` 与租户配置形成回退机制。

## 五、配置驱动表格能力

### 5.1 核心入口

- Controller：`CommonTableController`
- 服务：`FxTableConfigService`、`FxUserTableConfigService`
- 配置表：`fx_table_config`、`fx_table_column_config`、`fx_user_table_config`

### 5.2 能力说明

| 能力 | 说明 |
|---|---|
| 公共配置 | 通过 `tenant_id=0` 提供默认表格结构 |
| 租户配置 | 在公共配置基础上覆盖租户差异 |
| 用户列设置 | 用户可个性化控制列显隐和排序 |
| 查询字段驱动 | 后端配置直接决定查询表单渲染方式 |

### 5.3 推荐接入思路

1. 为业务页面定义稳定的 `tableCode`；
2. 在配置表维护表格与列元数据；
3. 前端页面传入 `tableCode + request`；
4. 若需要用户列设置，则直接复用现有接口与前端组件。

## 七、V0.5.0 后端开发建议

1. **Controller 只返回 `R`，不要直接返回裸对象。**
2. **写操作尽量返回有意义的 `message`，前端才能自动提示。**
3. **新增模块提示时优先定义 Prompt 枚举，而不是写死中文。**
4. **涉及租户与语言的异步任务，要显式复制上下文。**
5. **新增列表页优先走动态表格配置驱动，而不是重复手写表格结构。**
