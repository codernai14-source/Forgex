# 网关与路由使用方式

> 分类：后端 / 模块专题
> 版本：**V0.6.5**
> 更新时间：**2026-04-17**

本文档说明网关能力在项目中的实际使用方式，重点回答：

1. 新增服务、新增接口前缀时网关该怎么配。
2. 哪些接口应该放白名单，哪些绝不能放。
3. 前端和后端联调时怎么判断问题在网关还是在下游。
4. `401`、`404`、路由不通、fallback、上下文丢失时如何排查。

## 1. 后端如何使用网关

### 1.1 新增服务路由

如果新增了一个后端微服务，网关至少要补一条 route，配置位置在：

- `D:\mine_product\forgex\Forgex_MOM\Forgex_Backend\Forgex_Gateway\src\main\resources\application.yml`

典型配置要素包括：

1. `id`
2. `uri`
3. `predicates`
4. `filters`

参考现有写法：

```yaml
- id: api-demo
  uri: lb://forgex-demo
  predicates:
    - Path=/api/demo/**
  filters:
    - StripPrefix=1
    - name: RequestRateLimiter
      args:
        key-resolver: "#{@ipKeyResolver}"
        redis-rate-limiter.replenishRate: ${FORGEX_GATEWAY_DEMO_RATE:1000}
        redis-rate-limiter.burstCapacity: ${FORGEX_GATEWAY_DEMO_BURST:3000}
    - name: CircuitBreaker
      args:
        name: api-demo
        fallbackUri: forward:/fallback/sys
```

接入时要一起考虑：

- 是否需要去掉 `/api` 前缀
- 走哪个服务名
- 是否需要限流
- fallback 要走哪个兜底服务

默认限流按企业内网/低并发业务场景放宽：登录与授权路由为 300/s、突发 600，其余系统与业务路由为 1000/s、突发 3000。生产环境如需收紧，可通过 `FORGEX_GATEWAY_*_RATE` 和 `FORGEX_GATEWAY_*_BURST` 环境变量覆盖。

### 1.2 新增系统级公共接口

如果新增的是系统初始化、登录前配置、license 修复、第三方公开调用等接口，需要评估是否加入白名单。白名单判断写在：

- `GatewayAuthGlobalFilter.needAuth(...)`

正确做法是：

1. 先确认这个接口是否必须在未登录状态下访问
2. 再确认是否有其他安全约束，例如签名、token、IP 白名单
3. 最后才把它加入白名单

不要因为联调方便就把业务接口随手加到白名单里。

### 1.3 新增需要登录保护的接口

如果是普通业务接口，一般不需要改网关代码，只要：

1. 路径位于受保护前缀下，例如 `/api/sys/**`、`/api/basic/**`、`/api/wf/**`
2. 下游 Controller 正确做权限保护

这类接口会自动先经过网关登录态检查，再进入下游权限控制。

### 1.4 新增第三方公开调用入口

这类接口是最容易配错的一类，因为它既不能走普通登录态，又不能完全裸奔。建议：

1. 只放行最小必要路径
2. 在业务服务内用 token、签名、白名单 IP 等方式继续校验
3. 配合 `ClientIpPropagationFilter` 使用 `X-Client-IP`

## 2. 前端如何联调

### 2.1 判断问题在网关还是在下游

联调时可以按下面顺序判断：

1. 请求有没有到达网关入口
2. 网关是否直接返回错误
3. 如果网关已放行，下游服务是否收到请求
4. 下游服务收到后是登录态问题、权限问题还是业务异常

简单理解：

- 网关拦截
  通常表现为 `401`、license 错误、路由找不到、fallback
- 下游拒绝
  通常表现为业务异常、权限码不匹配、参数校验失败

### 2.2 前端为什么有时“看起来是接口报错，其实是路由不通”

当前前端统一通过 `/api/...` 访问后端。如果网关 route 没配对，就会出现：

- `404`
- `NotFoundException`
- 进入网关统一异常处理

所以看到“页面报错”时，不要先怀疑页面逻辑，先确认该前缀有没有对应网关 route。

### 2.3 初始化与登录页联调

登录页、初始化向导页依赖很多白名单接口，例如：

- 系统基础配置
- 登录验证码配置
- 语言列表
- 初始化状态
- license 状态

如果这些页面一打开就报未登录，优先查白名单而不是前端 token。

## 3. 常见使用场景

### 3.1 给新模块开路由

使用顺序：

1. 在 `application.yml` 增加 route
2. 决定是否 `StripPrefix=1`
3. 增加限流配置
4. 增加 fallback
5. 启动后验证 `/api/模块前缀/**` 是否能进对应服务

### 3.2 给新公共接口开白名单

使用顺序：

1. 明确 path 和 HTTP method
2. 在 `needAuth(...)` 中加精确条件
3. 自测未登录状态能否访问
4. 再确认不会误放开同前缀其他业务接口

### 3.3 排查上下文透传问题

如果后端服务里拿不到租户、用户、语言，先查网关透传是否成功：

- `X-User-Id`
- `X-Tenant-Id`
- `X-Account`
- `X-Lang`
- `X-Client-IP`

这一步不通，后面查 `TenantContext` 基本都是白查。

## 4. 常见问题排查

### 4.1 接口 401 或提示未登录

优先检查：

1. 请求路径是不是误命中了受保护前缀
2. 本来应该白名单的接口有没有加入白名单
3. token 是否透传
4. 网关是否成功读取 Sa-Token session

### 4.2 接口 404

优先检查：

1. 网关 route 是否存在
2. Path 前缀是否匹配
3. 是否少了 `StripPrefix=1`
4. 下游服务名是否和注册中心一致

### 4.3 接口被 fallback 兜底

优先检查：

1. 下游服务是否存活
2. 熔断器是否已触发
3. fallbackUri 是否配置正确
4. 网关返回的是不是下游不可用的统一提示

当前网关通过 Resilience4j TimeLimiter 将下游响应等待时间配置为 10s。开发环境刚进入页面时，如果下游模块仍在预热或初始化连接池，10s 内完成响应不会进入 fallback；超过该时间仍会按熔断降级返回统一提示。

### 4.4 下游服务拿不到租户和用户

优先检查：

1. 网关是否解析到了登录 session
2. `TenantPropagationGlobalFilter` 是否执行
3. 请求头是否包含 `X-User-Id` 和 `X-Tenant-Id`
4. 下游拦截器是否有读取逻辑

### 4.5 第三方集成调用异常

优先检查：

1. 公开调用入口是否在白名单
2. 授权 token 校验接口是否在白名单
3. `X-Client-IP` 是否正确透传
4. 业务服务是否继续做了 IP 白名单或 token 校验

### 4.6 license 相关页面打不开

优先检查：

1. license 状态、申请信息、刷新接口是否仍在白名单
2. 当前 license 目录、公钥、环境变量是否正确
3. 网关是否把本来应该放开的修复入口也拦截了

## 5. 推荐实践

- 白名单只给“必须未登录访问”的接口，不要把业务接口放进去。
- 每个新模块上线前都做一次 route、自测限流、自测 fallback。
- 排查问题先分网关层和业务层，避免改错地方。
- 任何依赖租户和用户上下文的模块，都先确认网关透传头是否正常。
