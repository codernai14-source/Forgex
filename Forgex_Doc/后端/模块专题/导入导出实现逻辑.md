# 导入导出实现逻辑

> 分类：后端 / 模块专题
> 版本：**V0.6.5**
> 更新时间：**2026-04-17**

本文档说明 Forgex Excel 导入导出能力在后端中的真实实现方式，重点覆盖：

1. 导出配置、导入配置、模板下载、业务导出之间的关系。
2. `tableCode` 在整套链路中的作用。
3. Excel 模板是如何根据配置动态生成的。
4. 公共导入如何根据处理器 Bean 分发到业务逻辑。
5. `DICT / JSON / PROVIDER` 三种下拉数据源如何工作。

## 1. 模块定位

Forgex 的导入导出不是简单的“某个业务自己写一个 EasyExcel 导出”，而是统一抽象成配置驱动能力。

核心思路是：

- 每个业务表或页面定义一个稳定的 `tableCode`
- 后台维护该 `tableCode` 的导出配置和导入配置
- 导出时按配置生成文件
- 导入模板时按配置生成列头、校验和下拉项
- 导入执行时按配置找到业务处理器，并把导入模式、导入配置、导入数据传入业务处理器

这样带来的好处是：

- 列顺序、列标题、列样式、是否必填都可以配置化
- 不同页面可以复用统一的 Excel 引擎
- 导入入口、权限判断、结果结构可以统一收口
- 国际化、字典、下拉项、样式规则可以统一处理

## 3. 核心模型

### 3.1 导出配置

导出配置对应：

- `FxExcelExportConfig`
- `FxExcelExportConfigItem`
- `FxExcelExportConfigDTO`

它描述的是：

- 这个 `tableCode` 导出什么
- 每一列叫什么
- 每一列对应哪个字段
- 列顺序、宽度、样式如何

### 3.2 导入配置

导入配置对应：

- `FxExcelImportConfig`
- `FxExcelImportConfigItem`
- `FxExcelImportConfigDTO`

它描述的是：

- 这个 `tableCode` 的导入模板有哪些列
- 这个 `tableCode` 对应哪个处理器 Bean
- 这个 `tableCode` 执行导入时需要哪个权限码
- 每列属于哪个 Sheet
- 每列映射到哪个业务字段
- 字段类型是什么
- 是否必填
- 下拉数据源是什么

导入主表新增字段：

- `handlerBeanName`：业务导入处理器 Bean 名称，对应数据库字段 `handler_bean_name`。
- `importPermission`：业务导入权限码，对应数据库字段 `import_permission`。

导入明细新增字段：

- `sheetCode`：Sheet 稳定编码，对应数据库字段 `sheet_code`，单 Sheet 默认 `main`。
- `sheetName`：Sheet 展示名称，对应数据库字段 `sheet_name`。

### 3.3 `tableCode`

`tableCode` 是整套链路的主键。无论是：

- 配置分页查询
- 配置详情
- 保存配置
- 下载模板
- 发起导出

最终都依赖同一个稳定的 `tableCode`。

如果 `tableCode` 改来改去，整个配置链路就会断。

## 4. 配置管理链路

### 4.1 导出配置

`ExcelConfigController` 提供了：

- `/sys/excel/exportConfig/page`
- `/sys/excel/exportConfig/detail`
- `/sys/excel/exportConfig/save`
- `/sys/excel/exportConfig/delete`

对应权限点包括：

- `sys:excel:exportConfig:list`
- `sys:excel:exportConfig:edit`
- `sys:excel:exportConfig:delete`

服务侧由 `ExcelConfigServiceImpl` 负责：

- `pageExportConfig`
- `getExportConfig`
- `getExportConfigByCode`
- `saveExportConfig`
- `deleteExportConfig`

### 4.2 导入配置

Controller 提供了：

- `/sys/excel/importConfig/page`
- `/sys/excel/importConfig/detail`
- `/sys/excel/importConfig/detailByCode`
- `/sys/excel/importConfig/save`
- `/sys/excel/importConfig/delete`
- `/sys/excel/template/download`
- `/sys/excel/import/execute`

服务侧由 `ExcelConfigServiceImpl` 负责：

- `pageImportConfig`
- `getImportConfig`
- `getImportConfigByCode`
- `saveImportConfig`
- `deleteImportConfig`

这两套配置不是互相替代关系，而是分别服务于模板生成和文件导出。

## 5. 导出链路

### 5.1 业务导出入口

当前系统中已有典型导出能力，例如：

- 登录日志导出
- 用户导出

由 `ExcelExportServiceImpl` 负责，核心步骤如下：

```text
前端传 tableCode + 查询条件
  -> ExcelExportServiceImpl
  -> ExcelConfigService.getExportConfigByCode(tableCode)
  -> 查询业务数据并转成行数据
  -> ExcelFileService.buildExportFile(config, rows)
  -> 设置下载头并输出文件流
```

### 5.2 文件名与响应头

`ExcelExportServiceImpl` 负责：

- `getSafeFilename`
- `Content-Disposition` 下载头处理

这一步是为了避免中文文件名、特殊字符文件名导致浏览器下载异常。

### 5.3 配置与业务数据解耦

导出能力有两个维度：

1. 业务服务负责准备数据行
2. Excel 引擎负责按照配置把这些数据渲染成文件

所以如果出现“导出的列不对”，可能有两类原因：

1. 导出配置错了
2. 业务数据 DTO 字段没映射对

## 6. 导入模板生成链路

### 6.1 下载模板

模板下载入口是：

- `/sys/excel/template/download`

调用顺序如下：

```text
前端传 tableCode
  -> ExcelConfigController.downloadTemplate
  -> ExcelConfigService.getImportConfigByCode(tableCode)
  -> ExcelFileService.buildImportTemplateXlsx(config)
  -> 返回 xlsx 模板
```

### 6.2 模板里有哪些内容

模板不是只生成一行标题，它通常会按配置生成：

- 国际化列头
- 必填提示
- 列顺序
- 列样式
- 下拉框校验
- 类型提示

这些能力都集中在 `ExcelFileServiceImpl` 中。

## 7. 公共导入执行链路

### 7.1 前端解析与提交

公共导入升级后，Excel 文件解析主要由前端 `CommonImportDialog.vue` 完成。组件会根据 `tableCode` 拉取导入配置，然后使用 `xlsx` 解析 `.xlsx / .xls / .csv`。

前端本地处理内容包括：

- 文件类型与 10MB 大小校验。
- 空行过滤。
- 表头顺序和列名匹配。
- 必填字段校验。
- 单 Sheet 或多 Sheet 数据按 `sheetCode` 分组。
- 前 5 行预览、全部数据查看和校验结果展示。

提交到后端的数据结构为：

```text
tableCode
importMode
importConfig
importData
```

其中 `importMode` 取值为：

- `ADD`
- `UPDATE`
- `COVER`

### 7.2 后端统一执行入口

统一执行入口是：

- `/sys/excel/import/execute`

核心链路如下：

```text
前端传 tableCode + importMode + importData
  -> ExcelConfigController.executeImport
  -> ExcelConfigService.getImportConfigByCode(tableCode)
  -> 校验 importPermission
  -> FxExcelImportExecuteService.execute(param)
  -> 根据 handlerBeanName 从 Spring 容器获取 FxExcelImportHandler
  -> handler.handle(param)
  -> 返回 FxExcelImportResultDTO
```

### 7.3 处理器分发规则

`FxExcelImportExecuteServiceImpl` 只负责公共控制逻辑，不写任何具体业务导入规则。它主要处理：

- `handlerBeanName` 是否配置。
- Spring 容器中是否存在对应 Bean。
- Bean 是否实现 `FxExcelImportHandler`。
- `importData` 是否为空。
- `importMode` 是否合法。

具体新增、修改、覆盖语义由业务处理器落地：

- `ADD`：新增，已存在的数据跳过。
- `UPDATE`：修改，只更新已存在的数据。
- `COVER`：覆盖，由业务处理器按业务边界清空或覆盖目标数据，再导入全部数据。

业务处理器需要根据本业务的唯一键做判断。例如：

- 用户导入按 `account` 判断。
- 供应商导入按 `supplierCode` 判断。
- 语言类型导入按 `langCode` 判断。

### 7.4 多 Sheet 数据结构

多 Sheet 导入依赖 `FxExcelImportConfigItem.sheetCode` 和 `sheetName`。模板生成时会按 `sheetCode` 分组创建多个 Sheet；前端解析后也会按同样的 `sheetCode` 组织数据。

供应商导入当前使用：

- `main`：供应商主表。
- `detail`：供应商详情。
- `contact`：联系人。
- `qualification`：资质。

CSV 文件只能表达单 Sheet 数据，因此公共导入组件只允许 CSV 用于单 Sheet 导入。

## 8. 下拉数据源机制

### 8.1 三种数据源

当前系统支持三种模板下拉数据源：

- `DICT`
  从数据字典拿选项
- `JSON`
  从静态 JSON 配置拿选项
- `PROVIDER`
  从 Provider 动态生成选项

### 8.2 Provider 注册中心

Provider 统一由：

- `TemplateOptionProviderRegistry`

管理。

它负责：

- 注册 Provider
- 判断某个编码是否存在
- 获取某个 Provider 的选项列表
- 返回所有已注册 Provider 编码

如果配置了不存在的 Provider，会直接抛业务异常，而不是悄悄降级。

### 8.3 已有 Provider 示例

当前代码里已经有一些 Provider 示例：

- `FactoryListOptionProvider`
- `DeptByFactoryOptionProvider`
- `RoleListOptionProvider`
- `JsonOptionProvider`

这些 Provider 说明导入模板的下拉项并不只能写死，而是可以根据上下文动态取值。

## 9. 国际化与提示

导入导出模块的提示语和异常并不是硬编码字符串，当前已经接入：

- `ExcelPromptEnum`
- 统一返回和国际化链路

这意味着：

- Provider 找不到
- 下拉项获取失败
- 模板解析失败
- 处理器未配置
- 处理器不存在
- 处理器类型错误
- 导入数据为空
- 导入方式无效
- 导入权限未配置或无权限
- 导入完成统计

这类问题最终可以通过统一国际化提示返回给前端。

## 10. 与租户的关系

`ExcelConfigController` 已经引入了 `TenantContext`。这意味着导入导出配置和数据导出并不完全是全局单例逻辑，很多场景仍然要考虑当前租户。

常见影响包括：

- 同一 `tableCode` 在不同租户下是否允许有不同配置
- 导出的业务数据是否只取当前租户
- 模板下拉项是否按当前租户过滤

这部分能力在业务实现里必须和租户上下文保持一致。

## 11. 常见排查

### 11.1 导出列顺序或标题不对

优先检查：

1. 导出配置项顺序是否正确
2. `tableCode` 是否拿错
3. 配置列字段名和业务 DTO 字段是否对应

### 11.2 模板没有下拉项

优先检查：

1. 当前列的数据源类型是 `DICT / JSON / PROVIDER` 哪一种
2. 如果是 `PROVIDER`，对应编码是否已注册
3. `dataSourceValue` 是否写对

### 11.3 导入解析报类型错误

优先检查：

1. 前端导入配置字段类型是否正确
2. Excel 中输入的实际格式
3. 业务处理器是否按字段类型读取 `importData`

### 11.4 公共导入提示处理器不存在

优先检查：

1. `fx_excel_import_config.handler_bean_name` 是否为空。
2. Bean 名称是否和 `@Component("...")` 完全一致。
3. 对应 Bean 是否在当前服务的 Spring 扫描范围内。
4. 跨模块导入是否通过代理处理器或 Feign 正确转发。

### 11.5 公共导入提示无权限

优先检查：

1. `fx_excel_import_config.import_permission` 是否配置。
2. 当前登录用户是否拥有该权限码。
3. 菜单权限表中是否存在对应按钮权限。

### 11.6 导出下载文件名乱码

优先检查：

1. 是否走了 `ExcelExportServiceImpl` 统一下载逻辑
2. `Content-Disposition` 是否正确设置

### 11.7 改了页面 `tableCode` 后模板全失效

优先检查：

1. 前端 `tableCode` 是否和后台配置中的 `tableCode` 一致
2. 是否只改了页面代码，没有同步迁移配置记录

## 12. 开发建议

- `tableCode` 一旦投产尽量保持稳定，不要轻易改名。
- 导出列、模板列、校验规则优先走配置，不要在页面或业务里重复硬编码。
- Provider 只做选项生成，不要把复杂业务校验塞进模板引擎。
- 公共导入入口只负责配置、权限和处理器分发，业务唯一性和落库规则放在业务处理器中。
- 新业务导入要同步维护 `handlerBeanName`、`importPermission`、前端 `tableCode` 和权限菜单。
