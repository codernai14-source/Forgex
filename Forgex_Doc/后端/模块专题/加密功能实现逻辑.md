# 加密功能实现逻辑

> 分类：后端 / 模块专题
> 版本：**V0.6.5**
> 更新时间：**2026-04-17**

本文说明 Forgex 后端加密能力的真实落点、运行链路与排查重点。当前项目里的加密不是单点实现，而是由密码存储、传输加密、字段透明加解密、文件加密、KMS 密钥管理几条链路组合完成。

## 代码入口

| 能力 | 主要代码位置 | 说明 |
|---|---|---|
| 算法 Provider 选择 | `Forgex_MOM/Forgex_Backend/Forgex_Common/src/main/java/com/forgex/common/crypto/CryptoProviders.java` | 根据策略键解析具体加密实现 |
| 字段透明加解密注解 | `Forgex_MOM/Forgex_Backend/Forgex_Common/src/main/java/com/forgex/common/crypto/FieldEncrypt.java` | 标记实体字段是否启用透明加解密 |
| MyBatis 字段拦截器 | `Forgex_MOM/Forgex_Backend/Forgex_Common/src/main/java/com/forgex/common/crypto/FieldEncryptInterceptor.java` | `INSERT/UPDATE` 前加密，`SELECT` 后解密 |
| 文件加密服务 | `Forgex_MOM/Forgex_Backend/Forgex_Common/src/main/java/com/forgex/common/crypto/FileEncryptService.java` | 文件加密能力接口 |
| 文件加密实现 | `Forgex_MOM/Forgex_Backend/Forgex_Common/src/main/java/com/forgex/common/crypto/FileEncryptServiceImpl.java` | AES/SM4 文件加密、摘要计算 |
| 加密配置模型 | `Forgex_MOM/Forgex_Backend/Forgex_Sys/src/main/java/com/forgex/sys/domain/config/CryptoConfig.java` | 聚合加密相关配置结构 |
| KMS 服务接口 | `Forgex_MOM/Forgex_Backend/Forgex_Sys/src/main/java/com/forgex/sys/service/KmsService.java` | 密钥生成、轮换、禁用等能力 |
| KMS 服务实现 | `Forgex_MOM/Forgex_Backend/Forgex_Sys/src/main/java/com/forgex/sys/service/impl/KmsServiceImpl.java` | 主密钥包装业务密钥并记录审计 |

## 总体链路

```text
配置中心 / 数据库存储加密参数
  -> CryptoProviders 按策略解析具体算法
  -> 业务链路按场景调用
     -> 登录密码存储 / 校验
     -> 传输加密
     -> 实体字段透明加解密
     -> 文件上传前后的加解密
     -> KMS 生成与管理业务密钥
```

## 算法 Provider 解析机制

`CryptoProviders` 是整个加密模块的统一入口。它会根据策略字符串和配置中心参数创建对应的 Provider，当前已支持：

- `bcrypt`
- `argon2`
- `scrypt`
- `pbkdf2`
- `sm2`
- `sm4`
- `aes`
- `rsa`

几个关键点：

- `bcrypt` 是空策略时的默认兜底。
- `sm2` 会读取 `security.crypto.transport`。
- `sm4` 会读取 `security.crypto.sm4`。
- `aes` 会读取 `security.crypto.aes`。
- `rsa` 会读取 `security.crypto.rsa.publicKey`、`security.crypto.rsa.privateKey`。
- Provider 既能用于“密码校验类场景”，也能复用在“字段可逆加密类场景”。

## 字段透明加解密链路

字段加密的核心是 `@FieldEncrypt + FieldEncryptInterceptor`。

### 运行顺序

```text
MyBatis 执行 INSERT / UPDATE
  -> FieldEncryptInterceptor.intercept()
  -> 扫描参数对象中带 @FieldEncrypt 的 String 字段
  -> 按字段 algorithm 解析 Provider
  -> 加密后写回对象
  -> SQL 入库

MyBatis 执行 SELECT
  -> 查询完成
  -> 拦截器扫描结果对象
  -> 发现带 ENC: 前缀的字段则解密
  -> 返回给业务层明文
```

### 真实实现特点

- 只处理 `String` 类型字段。
- 会缓存字段元数据，避免每次反射全量扫描。
- 对 `Map` 参数做了兼容，适配 MyBatis-Plus 常见参数包装。
- 使用 `ENC:` 前缀标记已加密值，防止重复加密。
- 读取链路只会解密带 `ENC:` 前缀的数据，因此历史明文数据不会被误解密。

## 文件加密实现

文件加密由 `FileEncryptServiceImpl` 提供，能力包括：

- `encryptFile` / `decryptFile`
- `encryptStream` / `decryptStream`
- `computeDigest`
- `verifyDigest`

支持算法：

- `aes`，使用 AES-256-GCM
- `sm4`，使用 SM4-CBC

配置与密钥来源：

- AES 密钥读取 `security.crypto.aes.keyHex`
- SM4 密钥读取 `security.crypto.sm4.keyHex`

如果配置不存在，服务会自动生成随机密钥并回写配置库。

## KMS 密钥管理实现

`KmsServiceImpl` 是密钥管理中心，主要能力包括：

- `generateKey`
- `getActiveKey`
- `getKey`
- `rotateKey`
- `disableKey`

支持密钥类型：

- `AES`
- `SM4`
- `RSA`
- `SM2`

主密钥工作方式：

```text
首次调用 KMS
  -> 读取 security.kms.master
  -> 不存在则自动生成 256-bit 主密钥
  -> 保存到配置中心

生成业务密钥
  -> 先生成原始业务密钥
  -> 再用主密钥做 AES/GCM 加密包装
  -> 持久化到 sys_kms_key
  -> 写 sys_kms_key_log 审计日志
```

## 配置结构

`CryptoConfig` 当前聚合了以下配置块：

- `sm4`
- `aes`
- `rsa`
- `kms`
- `fileEncrypt`
- `fieldEncrypt`

## 与认证链路的关系

虽然认证授权有独立文档，但登录密码存储、传输加密都复用这里的统一加密底座，因此配置和排查要一起看。

## 常见问题与排查

### 字段被重复加密

先看数据库值是否带 `ENC:` 前缀。当前拦截器正是通过这个前缀防止重复加密，如果业务代码手工再加一层密文，会导致读取异常。

### 查询后仍是密文

通常检查三点：

1. 字段是否真加了 `@FieldEncrypt`
2. 字段类型是否为 `String`
3. 数据库存量数据是否带 `ENC:` 前缀且算法与当前配置一致

### 换环境后文件解不开

高概率是不同环境自动生成了不同的 `security.crypto.aes` 或 `security.crypto.sm4` 配置。自动生成适合开发环境，生产环境必须固定并做好配置迁移。
