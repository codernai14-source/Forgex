# 工作流使用方式

> 分类：后端 / 模块专题
> 版本：**V0.6.0**
> 更新时间：**2026-04-17**

本文档说明工作流模块在项目中的实际使用方式，重点覆盖：

1. 后端如何创建流程、发布流程、发起审批、处理审批。
2. 页面如何配置流程图、发起审批、处理待办。
3. 业务系统如何接解释器、通知和回调。
4. 常见联调与排查步骤。

## 1. 后端代码如何使用

### 1.1 新建一个审批流程

后端创建审批流程通常不是一次性把所有节点都塞进去，而是按“基础信息 -> 草稿流程图 -> 发布”三步走。

常用接口：

- `/wf/task/config/create`
- `/wf/task/config/draft/base/save`
- `/wf/task/config/draft/graph/save`
- `/wf/task/config/draft/publish`

实际推荐顺序：

1. 先创建或获取草稿编辑器
2. 保存草稿基础信息
3. 保存草稿流程图
4. 发布草稿

### 1.2 基础信息怎么配

流程基础信息通常包含：

- `taskName`
- `taskCode`
- `interpreterBean`
- `formType`
- `formPath`
- `status`
- `remark`

建议：

- `taskCode` 保持稳定，不要频繁改
- `interpreterBean` 只有在业务需要挂接解释器时才配置
- `formType` 和前端页面或低代码表单能力保持一致

### 1.3 审批图怎么配

审批图相关接口：

- `/wf/task/config/draft/graph/get`
- `/wf/task/config/draft/graph/save`

配置时至少要确认：

1. 是否有开始节点和结束节点
2. 每个审批节点是否配置了审批人规则
3. 分支节点是否配置了条件
4. 是否允许加签、转交

只把节点拖出来但不配审批人，是最常见的配置错误之一。

### 1.4 发布流程

发布接口：

- `/wf/task/config/draft/publish`

发布后才会生成正式可运行版本。未发布草稿只对配置页面可见，不会在发起审批页中作为正式可用流程出现。

### 1.5 发起审批

发起接口：

- `/wf/execution/start`

后端发起审批时一般需要传：

- `taskCode`
- `formContent`
- 必要时传 `selectedApprovers`

示例思路：

```java
WfExecutionStartParam param = new WfExecutionStartParam();
param.setTaskCode("LEAVE_APPROVAL");
param.setFormContent("{\"days\":3,\"reason\":\"事假\"}");
Long executionId = executionService.startExecution(param);
```

### 1.6 审批通过、驳回、转交、加签

常用接口：

- `/wf/execution/approve`
- `/wf/execution/reject`
- `/wf/execution/transfer`
- `/wf/execution/addSign`

这些操作不要自己直接改表状态，必须走统一服务：

- `WfExecutionServiceImpl`

因为它内部还会处理：

- 审批实例更新
- 动作日志
- 通知
- 引擎推进
- 待办生成

### 1.7 批量处理与补偿

工作流还提供了：

- `/wf/execution/batch/approve`
- `/wf/execution/batch/transfer`
- `/wf/execution/batch/remind`
- `/wf/execution/compensate`
- `/wf/execution/timeout/retry`

适用场景：

- 批量审批
- 批量转交
- 批量催办
- 异常恢复
- 超时任务补偿

## 2. 业务系统如何接入工作流

### 2.1 使用解释器

如果业务系统想在审批关键节点执行自己的逻辑，推荐接入解释器，而不是直接改工作流引擎代码。

接入步骤：

1. 实现 `IApprovalInterpreter`
2. 注册为 Spring Bean
3. 在流程配置中填写 `interpreterBean`

适合放在解释器里的逻辑：

- 审批开始时初始化业务状态
- 审批通过后更新业务单据状态
- 审批驳回后回写业务状态
- 审批结束后做收尾动作

不适合放在解释器里的逻辑：

- 替代引擎推进流程
- 修改审批实例底层状态

### 2.2 使用回调

如果业务系统不想直接依赖解释器，也可以使用回调能力。

相关接口：

- 注册：`/wf/callback/register`
- 注销：`/wf/callback/unregister`

适用场景：

- 审批完成后通知外部系统
- 异构服务之间解耦

### 2.3 使用通知

通知由工作流统一负责，业务系统一般不需要自行在每个审批动作里再发一遍。

常见通知场景：

- 待办通知
- 审批通过通知
- 审批驳回通知
- 审批完成通知

如果通知内容要业务化，优先通过模板和解释器补数据，不要在页面里拼接通知文本。

## 3. 页面如何使用

### 3.1 流程配置页面

页面位置：

- `D:\mine_product\forgex\Forgex_MOM\Forgex_Fronted\src\views\workflow\taskConfig\index.vue`

页面用途：

1. 查看流程列表
2. 新建流程草稿
3. 编辑基础信息
4. 进入节点配置页面
5. 发布草稿

页面中已接权限：

- `wf:taskConfig:add`
- `wf:taskConfig:edit`
- `wf:taskConfig:config`
- `wf:taskConfig:delete`

### 3.2 节点配置页面

页面位置：

- `D:\mine_product\forgex\Forgex_MOM\Forgex_Fronted\src\views\workflow\taskConfig\nodes.vue`

页面用途：

1. 配置流程图节点
2. 配置节点规则
3. 配置审批人
4. 保存草稿
5. 发布草稿

联调时重点看：

- 节点是否完整连通
- 分支条件是否正确
- 审批人是否能解析

### 3.3 发起审批页面

页面位置：

- `D:\mine_product\forgex\Forgex_MOM\Forgex_Fronted\src\views\workflow\execution\start.vue`

页面用途：

1. 列出已启用可发起的流程
2. 打开对应表单
3. 提交 `taskCode + formContent`

这页只会显示可用流程，所以如果流程已经配置但发起页看不到，优先查是否发布和启用。

### 3.4 我的待办、已办、我发起的

页面位置：

- `myTask/pending.vue`
- `myTask/processed.vue`
- `myTask/initiated.vue`

用途分别是：

- 待办处理
- 已处理查询
- 我发起的审批跟踪与撤销

其中待办页已经接了这些按钮权限：

- `wf:execution:approve`
- `wf:execution:reject`
- `wf:execution:transfer`
- `wf:execution:addSign`
- `wf:execution:remind`

发起页中的撤销按钮则依赖：

- `wf:execution:cancel`

## 4. 联调建议

### 4.1 新流程联调顺序

推荐顺序：

1. 创建流程草稿
2. 保存基础信息
3. 配节点和审批人
4. 发布草稿
5. 在发起页确认流程可见
6. 发起一笔审批
7. 在待办页处理
8. 在已办和我发起页查看流转结果

### 4.2 业务集成联调顺序

如果还接了解释器、通知、回调，推荐再补两步：

9. 验证解释器是否按预期触发
10. 验证通知或回调是否发出

## 5. 常见问题排查

### 5.1 流程配置了但发起页看不到

优先检查：

1. 是否已经发布草稿
2. 流程状态是否启用
3. 前端是否调用的是 `listTaskConfig({ status: 1 })`

### 5.2 发起审批成功但没有待办

优先检查：

1. 第一个审批节点是否配置了审批人
2. 审批人规则是否能解析出目标用户
3. 是否误把流程配置成了只草稿未发布版本

### 5.3 待办页看不到审批按钮

优先检查：

1. 当前用户是否有对应权限码
2. 当前用户是否真的是该条待办审批人
3. 页面 `v-permission` 是否与后端权限码一致

### 5.4 审批后流程不继续

优先检查：

1. 当前节点后的下一节点是否存在
2. 分支条件是否命中
3. 下一节点审批人是否能解析

### 5.5 加签或转交失败

优先检查：

1. 当前节点规则是否允许加签 / 转交
2. 目标审批人 ID 是否合法
3. 当前审批实例是否仍处于可操作状态

### 5.6 通知没发出来

优先检查：

1. `WorkflowNotificationService` 是否被调用
2. 模板编码是否存在
3. 模板消息失败后是否已降级直发

### 5.7 回调没收到

优先检查：

1. 是否已注册回调
2. `taskCode` 是否一致
3. 回调地址是否可达
4. 审批最终状态是否真的触发了回调时机

### 5.8 补偿中心数据不对

优先检查：

1. 是否确实存在异常执行
2. `compensate` 或 `timeout/retry` 是否已执行
3. 动作处理器是否记录了补偿日志

## 6. 推荐实践

- 工作流定义一定走草稿 -> 发布，不要跳过。
- 每个审批节点都给审批人解析留兜底方案。
- 业务接入优先用解释器或回调，不要直接改引擎。
- 按钮权限、接口权限、待办归属三件事要一起检查，少查一项都容易误判。
