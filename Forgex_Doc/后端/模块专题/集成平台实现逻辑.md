# 集成平台实现逻辑

> 分类：后端 / 模块专题
> 版本：**V0.6.0**
> 更新时间：**2026-04-17**

本文档说明接口平台 / 集成平台模块在后端和前端中的真实实现结构。重点覆盖：

1. 第三方系统、接口配置、参数配置、映射、授权、调用日志之间的关系。
2. 核心 Controller、Service 在哪里。
3. 权限点是如何保护这些接口的。
4. 调用记录为什么采用按月分表。
5. 遇到“新增报错、看不到新增按钮、调用记录报错、菜单名显示代码”时从哪里查。

## 1. 模块定位

集成平台模块用于统一管理外部系统接入与 API 配置，核心目标包括：

- 管理第三方系统基本信息
- 管理接口配置
- 管理入参、出参参数结构
- 管理参数映射关系
- 管理第三方授权信息
- 记录并查询接口调用日志

它不是一个单点页面，而是一条完整链路：

```text
第三方系统
  -> API 配置
  -> 参数配置
  -> 参数映射
  -> 授权配置
  -> 实际调用
  -> 调用日志
```

## 3. 权限保护模型

集成平台模块的核心接口基本都已加 `@RequirePerm(...)`。这说明它不是无权限裸接口。

### 3.1 第三方系统权限点

`ThirdSystemController` 中已明确使用：

- `integration:third-system:view`
- `integration:third-system:add`
- `integration:third-system:edit`
- `integration:third-system:delete`
- `integration:third-system:batch-delete`

### 3.2 接口配置权限点

`ApiConfigController` 中已明确使用：

- `integration:api-config:view`
- `integration:api-config:add`
- `integration:api-config:edit`
- `integration:api-config:delete`

参数配置与参数映射也分别使用：

- `integration:api-config:config-param`
- `integration:api-config:config-mapping`

### 3.3 调用记录权限点

`ApiCallLogController` 中主要使用：

- `integration:api-call-log:view`

### 3.4 授权配置权限点

`ThirdAuthorizationController` 中使用：

- `integration:third-system:auth`

所以如果出现“按钮不显示”或“点击报无权限”，第一原则不是怀疑系统没接权限，而是先核查：

1. 后端接口权限码
2. 前端按钮权限码
3. 菜单按钮授权
4. 当前租户下的角色权限

## 4. 第三方系统链路

### 4.1 控制层职责

`ThirdSystemController` 负责：

- 分页查询
- 列表查询
- 详情查询
- 新增
- 编辑
- 删除
- 批量删除

这是整个集成平台链路的第一层基础数据。

### 4.2 服务层职责

`ThirdSystemServiceImpl` 是第三方系统业务核心。根据现有代码，它具备这些特征：

- 分页列表查询走 MyBatis-Plus `page`
- 新增时会执行 `save(system)`
- 逻辑中显式读取 `TenantContext.get()`

这说明第三方系统配置是租户级数据，而不是全局共享数据。

### 4.3 写入时的租户与审计

`ThirdSystemServiceImpl` 中对当前租户依赖较明显：

- 当前租户来自 `TenantContext.get()`
- 当前操作人通常也会参与创建和更新字段写入

因此“第三方系统新增报错”时，要重点看下面几个方向：

1. 当前请求是否已经带上租户上下文
2. 必填字段是否完整
3. 目标表结构是否包含租户与审计字段
4. 自动填充是否生效

## 5. API 配置链路

### 5.1 控制层职责

`ApiConfigController` 负责：

- 分页查询
- 列表查询
- 详情
- 新增
- 编辑
- 删除
- 批量删除
- 启用
- 停用

这说明接口配置不仅是静态定义，还带有运行状态控制。

### 5.2 服务层职责

`ApiConfigServiceImpl` 主要负责：

- 分页条件构建
- DTO 与实体转换
- `save(config)` 持久化
- 启用、停用、更新、删除等状态管理

代码中已经出现 `TenantContext.get()`，说明它同样是租户级配置。

### 5.3 与参数配置、参数映射的关系

接口配置只是主体，真正决定“怎么入参、怎么出参、怎么映射”的是：

- `ApiParamConfigController / ServiceImpl`
- `ApiParamMappingController / ServiceImpl`

因此接口配置新增成功后，仍然需要继续完成：

1. 参数树配置
2. 参数映射配置
3. 授权配置

否则这个 API 仍然只是“定义存在”，不一定能被正确调用。

## 6. 调用日志链路

### 6.1 入口

调用日志查询入口是 `ApiCallLogController`，主要负责：

- 分页查询日志
- 查询详情
- 查询统计类信息

### 6.2 服务实现特征

`ApiCallLogServiceImpl` 的实现非常关键，现有代码已经明确表现出以下结构：

- 月份格式：`yyyyMM`
- 表前缀：`fx_api_call_log_`
- 异步写入方法：`asyncSaveLog(ApiCallLog logEntity)`

这说明调用日志不是单表，而是按月分表。

### 6.3 为什么要按月分表

调用日志天然增长快，如果全压到一张表，会带来：

- 单表数据量过大
- 查询慢
- 索引膨胀
- 历史清理困难

按月分表后：

- 写入落在当前月表
- 查询会根据时间范围推算需要扫描哪些月表
- 统计和分页在服务层做跨表聚合

### 6.4 查询链路

典型查询思路如下：

```text
前端传时间范围
  -> ApiCallLogController
  -> ApiCallLogServiceImpl
  -> 根据开始结束时间计算月份列表
  -> 拼出 fx_api_call_log_yyyyMM 表名集合
  -> 跨表分页 / 详情 / 统计
```

因此“调用记录报错”非常可能不是页面本身的问题，而是以下几类原因：

1. 查询时间范围为空或格式不对
2. 对应月份分表不存在
3. 跨表 SQL 或 mapper 组装异常
4. 某个月表结构和其他月份不一致

### 6.5 异步写日志与上下文

日志写入通过 `asyncSaveLog` 异步落库，这意味着它天然受上下文复制影响。如果日志记录里涉及：

- `tenantId`
- `createBy`
- 其他依赖上下文的字段

就必须确认异步线程中是否正确继承或复制了上下文。

相关说明见：

- [多租户实现逻辑](./../租户与上下文/多租户实现逻辑.md)

## 8. 用户问题对应的排查逻辑

### 8.1 第三方系统配置新增报错

优先检查：

1. 前端提交字段是否完整
2. 后端 DTO 到实体转换是否遗漏必填字段
3. `TenantContext.get()` 是否为空
4. 数据表字段约束、唯一索引是否冲突
5. 当前用户是否有 `integration:third-system:add`

### 8.2 接口配置页面看不到新增按钮

优先检查：

1. 当前用户是否拥有 `integration:api-config:add`
2. 菜单按钮权限是否已授权给当前角色
3. 当前租户下角色权限是否正确
4. 前端指令 `v-permission` 是否正常读取到权限集合

### 8.3 调用记录报错

优先检查：

1. 前端时间范围参数格式
2. 目标月份日志表是否存在
3. 月表结构是否齐全
4. 跨表查询 SQL 是否能兼容所有月份
5. 查询条件是否传了非法 `apiConfigId`

### 8.4 首页菜单名显示的是代码

这类问题通常不是权限本身，而是国际化或菜单名称渲染问题。常见原因包括：

1. 菜单配置里保存的是国际化 key，但前端没有翻译到文本
2. 菜单接口返回的是代码字段，页面直接显示了代码
3. 国际化资源缺失，导致 key 原样显示

它和接口权限是不同层的问题，但经常在同一页面里同时出现。

## 9. 开发建议

- 第三方系统、接口配置、授权、参数配置都按租户级业务理解，不要误当成全局配置。
- 接口和页面都已经接了权限模型，出现问题优先查权限码和角色授权，不要直接判断“后端没接权限”。
- 调用日志问题要重点关注分表，不要只盯 Controller。
- 页面显示代码值时，优先排查国际化资源和菜单配置字段。
