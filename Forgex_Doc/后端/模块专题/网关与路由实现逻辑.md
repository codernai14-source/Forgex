# 网关与路由实现逻辑

> 分类：后端 / 模块专题
> 版本：**V0.6.0**
> 更新时间：**2026-04-17**

本文档说明 Forgex 网关模块在后端中的真实实现结构，重点覆盖：

1. 请求进入网关后先经过哪些过滤器。
2. 哪些接口会被白名单直接放行，哪些接口必须鉴权。
3. 网关如何做 license 校验、限流、熔断和异常兜底。
4. 网关如何把用户、租户、语言、客户端 IP 透传给下游服务。
5. 接口出现 `401`、`404`、熔断兜底、上下文丢失时应该查哪里。

## 1. 模块定位

Forgex 网关不是简单的“请求转发层”，它同时承担了以下职责：

- 统一入口
- 路由分发
- 白名单放行
- 登录态校验
- license 有效性校验
- 限流
- 熔断与降级
- 用户、租户、语言、IP 等上下文透传
- 网关级异常统一返回

网关模块名为 `Forgex_Gateway`，启动类是：

- `D:\mine_product\forgex\Forgex_MOM\Forgex_Backend\Forgex_Gateway\src\main\java\com\forgex\gateway\ForgexGatewayApplication.java`

## 3. 请求主链路

请求经过网关的大致链路如下：

```text
客户端请求进入 9000 网关
  -> ClientIpPropagationFilter 提取真实客户端 IP
  -> GatewayAuthGlobalFilter 判断是否需要鉴权
  -> 如果需要鉴权，先校验 license，再校验登录态
  -> GatewayLoginSessionSupport 解析 token / session
  -> TenantPropagationGlobalFilter 透传 X-User-Id / X-Tenant-Id / X-Lang / X-Account
  -> 根据 Route 配置转发到对应微服务
  -> 若触发熔断则走 FallbackController
  -> 若网关层报错则走 GatewayGlobalExceptionHandler
```

真正要理解的是两层：

1. `GatewayAuthGlobalFilter` 决定“要不要拦截”
2. `TenantPropagationGlobalFilter` 决定“把什么上下文带给下游”

## 4. 路由配置

路由统一配置在：

- `src/main/resources/application.yml`

当前已经明确配置的路由包括：

- `/api/auth/**` -> `forgex-auth`
- `/api/sys/**` -> `forgex-sys`
- `/api/basic/**` -> `forgex-basic`
- `/api/files/**` -> `forgex-sys`
- `/api/wf/**` -> `forgex-workflow`
- `/api/app/**` -> `forgex-sys`
- `/api/integration/**` -> `forgex-integration`
- `/api/report/**` -> `forgex-report`

其中大多数路由都带了：

- `RequestRateLimiter`
- `CircuitBreaker`

多数业务路由还会通过 `StripPrefix=1` 把 `/api` 前缀去掉再转发给下游。

## 5. 鉴权与白名单逻辑

### 5.1 总入口

网关鉴权总入口是：

- `GatewayAuthGlobalFilter.filter(...)`

它的核心逻辑可以概括为：

```text
请求进入
  -> 判断 path + method 是否需要鉴权
  -> 如果不需要，直接放行
  -> 如果需要，先检查 license 是否有效
  -> license 有效后再继续下游处理
```

### 5.2 白名单判定

白名单判断集中在：

- `GatewayAuthGlobalFilter.needAuth(String path, HttpMethod method)`

当前源码里已经明确放行了不少初始化和公共接口，例如：

- `/api/auth/**`
- `GET /api/sys/config/system-basic`
- `GET /api/sys/config/login-captcha`
- `POST /api/sys/i18n/languageType/listEnabled`
- `GET /api/sys/init/status`
- `POST /api/sys/init/apply`
- license 相关状态与刷新接口
- `GET /api/basic/module/ping`
- 第三方集成平台公开调用入口和授权校验入口
- `GET/HEAD /api/files/**`

这里要特别注意两类接口：

1. 系统初始化前必须可访问的接口
2. 第三方系统调用平台时必须可访问的公开入口

这也是为什么“白名单”不是只给登录页用的。

### 5.3 哪些路径默认需要鉴权

从现有逻辑看，下面这些前缀下的大多数业务请求都被视为需要鉴权：

- `/api/sys/`
- `/api/basic/`
- `/api/files/`
- `/api/app/`
- `/api/wf/`
- `/api/report/`
- 集成平台受保护接口前缀

因此某个接口即使后端 Controller 没有额外写权限注解，只要没有命中白名单，仍然会先经过网关登录态拦截。

## 6. License 校验

### 6.1 校验位置

license 校验在：

- `GatewayAuthGlobalFilter`

它依赖：

- `LicenseManager`
- `LicenseRuntimeInfo`
- `LicenseStatus`

### 6.2 校验顺序

当前逻辑是：

1. 先判断当前请求是不是白名单
2. 如果不是白名单，再判断 license 是否有效
3. 如果无效，直接返回 license 相关错误，不再继续下游路由

这意味着：

- 业务接口异常可能不是登录问题，而是 license 失效
- 初始化、license 状态查看等接口需要被白名单放行，否则系统会陷入“无法修复 license”的死循环

### 6.3 错误返回

当前实现会根据 license 状态映射成不同的提示，例如：

- 未授权
- 已过期
- 宽限期
- 非法 license

并由网关直接返回统一结构，而不是把请求继续打到业务服务。

## 7. 登录态与上下文透传

### 7.1 `GatewayLoginSessionSupport`

`GatewayLoginSessionSupport` 负责从 token / Sa-Token session 中解析登录信息，重点字段包括：

- token
- loginId
- `LOGIN_USER_ID`
- `LOGIN_TENANT_ID`
- account

它还会维护在线用户 TTL：

- `fx:online:user:{tenantId}:{userId}`

### 7.2 `TenantPropagationGlobalFilter`

这个过滤器的作用不是“做鉴权”，而是把登录态解析出的上下文透传给下游服务。当前明确透传的请求头包括：

- `X-User-Id`
- `X-Tenant-Id`
- `X-Account`
- `X-Lang`

其中 `X-Lang` 还会优先尝试从 Redis 读取：

- `fx:lang:{tenantId}:{userId}`

这就是下游服务里 `UserTenantWebInterceptor` 能写入 `TenantContext`、`UserContext` 的前提。

### 7.3 与多租户文档的关系

网关只负责透传，不负责把上下文写进业务线程。真正写入 `TenantContext` 的逻辑在下游公共拦截器里，详见：

- [多租户实现逻辑](./../租户与上下文/多租户实现逻辑.md)

## 8. 客户端 IP 透传

### 8.1 过滤器位置

- `ClientIpPropagationFilter`

### 8.2 提取优先级

源码里已经明确真实 IP 的提取顺序：

1. `X-Forwarded-For` 第一个 IP
2. `X-Real-IP`
3. `RemoteAddress`

然后再透传给下游：

- `X-Client-IP`

这对日志审计、第三方授权 IP 白名单、调用记录都很关键。

## 9. 限流与熔断

### 9.1 限流

限流配置来自：

- `application.yml`
- `RateLimitConfig.ipKeyResolver()`

当前网关使用按 IP 限流的思路，不同路由配置了不同的：

- `replenishRate`
- `burstCapacity`

例如登录接口限流更严格，系统接口和管理接口限流会相对放宽。

### 9.2 熔断

大多数业务路由都配置了 `CircuitBreaker`，并指定了 `fallbackUri`，例如：

- `forward:/fallback/sys`
- `forward:/fallback/basic`
- `forward:/fallback/files`

一旦下游服务不可用，网关会直接进入 `FallbackController`。

## 10. Fallback 与网关异常处理

### 10.1 `FallbackController`

`FallbackController` 提供：

- `/fallback/{service}`

它的作用是：

- 返回统一兜底响应
- 优先通过系统国际化接口解析提示文案
- 失败时再走本地降级翻译

### 10.2 `GatewayGlobalExceptionHandler`

这个类负责处理网关层异常，例如：

- `NotFoundException`
- `ResponseStatusException`
- 其他未捕获异常

它和 `FallbackController` 的区别是：

- `FallbackController` 面向“下游服务降级”
- `GatewayGlobalExceptionHandler` 面向“网关本身抛异常”

## 11. 常见排查

### 11.1 接口返回 401 或未登录

优先检查：

1. 请求路径是否命中了白名单
2. token 是否有效
3. 网关是否正确读取到了 session
4. 下游请求头里是否已经带上 `X-User-Id` / `X-Tenant-Id`

### 11.2 系统提示 license 无效

优先检查：

1. `LicenseManager.current()` 返回状态
2. 当前请求是不是本该放行的 license 管理接口
3. 网关启动参数中的 license 路径和公钥配置

### 11.3 路由转发不到服务

优先检查：

1. `application.yml` 中的 route `id / uri / Path` 是否正确
2. 服务是否已注册到注册中心
3. 是否需要 `StripPrefix=1`
4. 当前请求是不是应该被转发到别的服务模块

### 11.4 下游服务拿不到租户和用户上下文

优先检查：

1. `TenantPropagationGlobalFilter` 是否命中
2. `GatewayLoginSessionSupport` 是否成功解析 userId / tenantId
3. 下游公共拦截器是否正确读取 `X-User-Id`、`X-Tenant-Id`

### 11.5 第三方调用或文件预览被误拦截

优先检查：

1. 对应公开接口是否已经加入白名单
2. HTTP 方法是否和白名单里的 method 完全一致
3. 路径前缀是否与网关判定逻辑一致

### 11.6 请求进入 fallback

优先检查：

1. 下游服务是否存活
2. 注册中心里该服务是否健康
3. 熔断阈值是否被打爆
4. 当前 fallback 被转发到哪个 `/fallback/{service}`

## 12. 开发建议

- 白名单一定要最小化，而且要写清楚“为什么必须放行”。
- 任何新增微服务接口前缀，都要同时检查 route、限流、熔断、鉴权策略。
- 网关只做统一入口和上下文透传，不要把复杂业务逻辑塞进过滤器。
- 排查问题时先分清是“网关拦截”还是“下游业务拒绝”，不要混着查。
