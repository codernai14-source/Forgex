# 文件上传实现逻辑

> 分类：后端 / 模块专题  
> 版本：**V0.6.0**  
> 更新时间：**2026-04-17**

本文说明 Forgex 文件上传模块的真实实现链路，包括上传入口、公开访问、存储策略选择、不同存储后端以及与加密能力的协同点。

## 代码入口

| 能力 | 代码位置 | 说明 |
|---|---|---|
| 上传接口 | `Forgex_MOM/Forgex_Backend/Forgex_Sys/src/main/java/com/forgex/sys/controller/FileController.java` | `POST /sys/file/upload` |
| 公共文件访问 | `Forgex_MOM/Forgex_Backend/Forgex_Sys/src/main/java/com/forgex/sys/controller/PublicFileController.java` | `GET /files/**` |
| 文件服务实现 | `Forgex_MOM/Forgex_Backend/Forgex_Sys/src/main/java/com/forgex/sys/service/impl/FileServiceImpl.java` | 上传、读取、媒体类型识别 |
| 存储工厂 | `Forgex_MOM/Forgex_Backend/Forgex_Sys/src/main/java/com/forgex/sys/service/storage/FileStorageFactory.java` | 决定用本地、OSS 还是 MinIO |
| 上传配置模型 | `Forgex_MOM/Forgex_Backend/Forgex_Sys/src/main/java/com/forgex/sys/domain/config/FileUploadConfig.java` | `file.upload.settings` 结构 |
| 本地存储实现 | `Forgex_MOM/Forgex_Backend/Forgex_Sys/src/main/java/com/forgex/sys/service/storage/LocalStorageService.java` | 本地磁盘上传 |
| MinIO 存储实现 | `Forgex_MOM/Forgex_Backend/Forgex_Sys/src/main/java/com/forgex/sys/service/storage/MinioStorageService.java` | MinIO 上传/下载/删除/URL |
| OSS 存储实现 | `Forgex_MOM/Forgex_Backend/Forgex_Sys/src/main/java/com/forgex/sys/service/storage/OssStorageService.java` | OSS 上传/下载/删除/URL |
| 上传配置辅助接口 | `Forgex_MOM/Forgex_Backend/Forgex_Sys/src/main/java/com/forgex/sys/controller/SysConfigController.java` | 默认访问地址、服务器目录选择 |

## 总体链路

```text
前端选择文件
  -> POST /sys/file/upload
  -> FileController.upload()
  -> FileServiceImpl.upload()
  -> FileStorageFactory.getDefault()
  -> 选择 LOCAL / OSS / MINIO
  -> storage.upload(file)
  -> storage.getUrl(relativePath)
  -> 返回访问 URL
```

## 上传入口实现

`FileController` 主要负责：

1. 接收 `MultipartFile`
2. 判空
3. 调用 `FileService.upload()`

真正的上传策略不在 Controller，而在 `FileServiceImpl` 与 `FileStorageFactory`。

## 文件服务实现

`FileServiceImpl` 负责把“接口层”和“存储层”串起来。

核心方法：

- `upload(MultipartFile file)`
- `getFile(String filename)`
- `getMediaType(String filename)`
- `getBaseDir()`

关键配置键：

- `file.upload.settings`
- `file.upload.path`

## 存储策略选择逻辑

`FileStorageFactory` 决定“这次上传实际走哪种存储”。

### 优先级

1. 全局配置 `file.upload.settings`
2. 租户表配置 `sys_file_storage`
3. 本地存储兜底

### 支持类型

- `LOCAL`
- `OSS`
- `MINIO`

## 本地存储逻辑

本地模式主要依赖：

- `file.upload.settings.localUploadPath`
- `file.upload.path`
- `file.upload.settings.accessPrefix`
- `file.upload.settings.publicBaseUrl`

本地路径解析顺序：

```text
优先读取 file.upload.settings.localUploadPath
  -> 如果未配置
  -> 使用 Spring 配置 file.upload.path
  -> 默认 C:/forgex/data/uploads
```

其中：

- `file.upload.settings.localUploadPath` 是系统管理中的数据库配置，优先级最高。
- `file.upload.path` 是应用启动时可用的兜底路径，适合通过环境变量 `FORGEX_UPLOAD_DIR` 控制部署目录。
- LOCAL 模式下如果配置了 `file.upload.settings.publicBaseUrl`，返回 URL 会按 `publicBaseUrl + accessPrefix + "/" + relativePath` 生成。
- `publicBaseUrl` 会统一规范化：只填 `192.168.x.x` 或填 `http://192.168.x.x` 且未指定端口时，按网关补为 `http://192.168.x.x:9000/api`；`accessPrefix` 统一补齐为 `/files` 形式。
- 推荐公网/内网浏览器访问链路是网关 `http://<服务器IP>:9000/api/files/<relativePath>`；直连系统服务时才是 `http://<服务器IP>:9002/files/<relativePath>`。

## 配置辅助接口

文件上传配置页通过 `SysConfigController` 提供辅助接口：

| 接口 | 权限 | 说明 |
|---|---|---|
| `GET /sys/config/file-upload/runtime-defaults` | `sys:config:view` | 枚举后端服务器 IPv4 候选，返回推荐 `publicBaseUrl=http://<ip>:9000/api`、`accessPrefix=/files` 和预览示例 |
| `GET /sys/config/file-upload/folders/roots` | `sys:config:view` | 返回后端服务器磁盘根目录，只返回目录节点 |
| `POST /sys/config/file-upload/folders/children` | `sys:config:view` | 按目录路径懒加载子目录，最多返回 300 个目录 |
| `POST /sys/config/file-upload/folders/create` | `sys:config:edit` | 在选中的后端服务器目录下新建文件夹 |

目录选择器面向后端服务器文件系统，不会读取或返回文件内容，也不是选择浏览器客户端电脑目录。

## 文件记录落库

上传成功后，`FileServiceImpl.upload()` 会调用 `ISysFileRecordService.saveUploadRecord()`，将以下信息写入 `sys_file_record`：

- `module_code`
- `module_name`
- `original_name`
- `stored_name`
- `file_type`
- `content_type`
- `file_size`
- `relative_path`
- `access_url`
- `storage_type`
- `storage_config_id`

因此前端展示登录背景、系统 Logo、用户头像、租户 Logo 时，都可以直接使用接口返回的 URL，或读取数据库里的 `access_url` 字段。

系统基础配置 `system.basic` 保存和读取时会对 `systemLogo`、`loginBackgroundImage`、`loginBackgroundVideo` 做媒体 URL 规范化。历史数据如果保存成 `/files/**` 或 `/api/files/**`，返回前会按当前 `file.upload.settings.publicBaseUrl` 补成完整网关地址；新保存的数据也会按同一规则落库，避免前端页面把相对路径请求到自身服务。

## 公共访问与预览逻辑

`PublicFileController` 通过 `GET /files/**` 提供公开读取能力。

处理流程：

```text
请求 /files/xxx/yyy.png
  -> 从 HandlerMapping 提取真实相对路径
  -> fileService.getFile(filePath)
  -> 获取 Resource
  -> fileService.getMediaType(filePath)
  -> 设置 Content-Disposition: inline
  -> 浏览器直接预览
```

## MinIO 存储逻辑

`MinioStorageService` 会从 `providerConfigJson` 解析以下字段：

- `endpoint`
- `accessKey`
- `secretKey`
- `bucketName`
- `domain`

实现特点：

- 上传前会自动检查 bucket 是否存在，不存在则创建。
- 文件路径按日期分层，如 `yyyy/MM/dd/uuid.ext`。
- 如果配置了 `domain`，返回自定义域名 URL。

## OSS 存储逻辑

`OssStorageService` 会从 `providerConfigJson` 解析以下字段：

- `endpoint`
- `accessKeyId`
- `accessKeySecret`
- `bucketName`
- `domain`

实现特点：

- 文件路径同样按日期分层。
- 如果配置了 `domain`，URL 直接使用自定义域名。
- 未配置 `domain` 时，返回 `https://bucket.endpoint/filePath`。

## 与文件加密的协同点

当前上传主链路默认是“上传原文件 -> 存储 -> 返回 URL”，并没有在 `FileServiceImpl.upload()` 内部强制调用 `FileEncryptService`。

如果业务需要做敏感附件加密，通常有两种接法：

1. 在上传逻辑中先加密再交给存储服务
2. 在对象存储前把内容转为加密流

## 常见排查

### 上传成功但链接打不开

重点检查：

1. `storageType`
2. `providerConfigJson`
3. 返回 URL 的域名前缀
4. MinIO bucket / OSS bucket 是否可访问
5. LOCAL 返回 URL 是否包含网关端口和 `/api` 前缀，例如 `http://<服务器IP>:9000/api/files/xxx.png`

### 页面能上传但图片不预览

重点检查：

1. URL 是否走网关 `/api/files/**`，或在直连系统服务时走 `/files/**`
2. 文件是否实际落在本地目录
3. `Content-Type` 是否识别正确
4. 如果 URL 是 `http://<ip>/files/xxx.png`，说明缺少端口，浏览器会走 80 端口，应保存为 `http://<ip>:9000/api`

### 多租户下文件串用

重点看 `sys_file_storage` 默认配置是否按租户正确维护，以及上传路径、桶、域名是否区分租户。
