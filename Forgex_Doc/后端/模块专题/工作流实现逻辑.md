# 工作流实现逻辑

> 分类：后端 / 模块专题
> 版本：**V0.6.0**
> 更新时间：**2026-04-17**

本文档基于工作流模块真实源码说明 Forgex 工作流能力的实现结构，重点覆盖：

1. 流程配置如何区分草稿版和发布版。
2. 发起审批、审批通过、驳回、转交、加签、撤销、补偿分别怎么走。
3. 节点、审批人、分支规则、解释器、通知、回调之间如何协同。
4. 前端页面入口在哪，如何和后端接口对齐。

## 1. 模块定位

Forgex 工作流模块不是外接第三方引擎的薄封装，而是系统内部自研的一套审批引擎，模块名为：

- `Forgex_Workflow`

它同时覆盖了：

- 审批任务配置
- 草稿编辑与发布
- 流程图节点编排
- 审批执行
- 审批实例与动作日志
- 审批人解析
- 回调通知
- 站内消息 / 模板消息
- 超时与补偿治理

启动类位于：

- `D:\mine_product\forgex\Forgex_MOM\Forgex_Backend\Forgex_Workflow\src\main\java\com\forgex\workflow\ForgexWorkflowApplication.java`

## 3. 数据模型分层

工作流核心实体大致可分为三层：

### 3.1 配置层

- `WfTaskConfig`
- `WfTaskNodeConfig`
- `WfTaskNodeApprover`
- `WfTaskNodeRule`

这一层描述“流程应该怎么跑”。

### 3.2 执行层

- `WfTaskExecution`
- `WfTaskExecutionDetail`
- `WfTaskExecutionApprover`
- `WfTaskApprovalInstance`

这一层描述“某一次审批实际跑到了哪里”。

### 3.3 日志与待办层

- `WfTaskApprovalActionLog`
- `WfMyTask`

这一层描述“谁处理过、当前谁待办、历史动作是什么”。

## 4. 任务配置链路

### 4.1 配置入口

任务配置入口在 `WfTaskConfigController`，当前已经提供：

- `/wf/task/config/page`
- `/wf/task/config/list`
- `/wf/task/config/get`
- `/wf/task/config/getByCode`
- `/wf/task/config/create`
- `/wf/task/config/update`
- `/wf/task/config/delete`
- `/wf/task/config/updateStatus`

并且已经接入权限：

- `wf:taskConfig:view`
- `wf:taskConfig:add`
- `wf:taskConfig:edit`
- `wf:taskConfig:delete`
- `wf:taskConfig:config`

### 4.2 草稿与发布模型

`WfTaskConfigServiceImpl` 里最重要的特征之一，是配置不是直接“在线编辑正式版”，而是走：

- 草稿版
- 发布版

相关方法包括：

- `getOrCreateDraftEditor(...)`
- `saveDraftBaseInfo(...)`
- `getDraftGraph(...)`
- `saveDraftGraph(...)`
- `publishDraft(...)`

这说明配置流程大致如下：

```text
创建或获取草稿
  -> 保存草稿基础信息
  -> 保存草稿流程图
  -> 发布草稿
  -> 生成新的发布版本
```

这样做的好处是：

- 正在跑的流程不会被半成品配置破坏
- 配置编辑和正式运行隔离
- 页面可以同时显示 draftVersion / publishedVersion

### 4.3 配置页面与节点页面

前端任务配置列表页：

- `src/views/workflow/taskConfig/index.vue`

节点图配置页：

- `src/views/workflow/taskConfig/nodes.vue`

列表页已经明确使用了权限控制：

- `wf:taskConfig:add`
- `wf:taskConfig:edit`
- `wf:taskConfig:config`
- `wf:taskConfig:delete`

节点页会和以下接口配合：

- `/wf/task/config/draft/editor`

- `/wf/task/config/draft/graph/get`

## 5. 流程图与节点配置

### 5.1 节点类型

从现有前端页面和服务结构看，流程图至少包含这些关键节点：

- 开始节点
- 审批节点
- 分支节点
- 结束节点

节点配置会落到：

- 节点基本信息
- 审批规则
- 审批人规则
- 分支条件
- 是否允许加签
- 是否允许转交

### 5.2 审批人配置

审批人并不是简单写死用户 ID。`WfEngineServiceImpl` 中已经有审批人解析逻辑，支持根据规则解析 approver ids。

结合常量和前端 DTO，审批人来源通常包括：

- 指定用户
- 发起人自选
- 角色
- 部门
- 其他规则型来源

### 5.3 分支规则

分支规则由节点规则驱动，运行时由引擎决定走哪条边。前端节点配置页和后端 `WfTaskNodeRule` / `WfBranchRuleDTO` 一起完成这部分定义。

## 6. 发起审批链路

### 6.1 入口

发起审批入口位于：

- `WfExecutionController.startExecution(...)`
- 对应接口：`POST /wf/execution/start`
- 权限：`wf:execution:start`

前端对应页面：

- `src/views/workflow/execution/start.vue`
- `src/views/workflow/execution/startForm.vue`

### 6.2 执行顺序

发起审批的大致链路如下：

```text
前端选择任务并填写表单
  -> /wf/execution/start
  -> WfExecutionServiceImpl.startExecution(...)
  -> 读取已发布任务配置
  -> 创建 execution / detail / approval instance
  -> 解析第一个审批节点和审批人
  -> 生成我的待办
  -> 触发解释器 onStart
  -> 通知当前待办审批人
```

### 6.3 表单类型

工作流任务配置当前至少支持两类表单：

- 自定义表单路径
- 低代码表单

前端相关组件包括：

- `LowCodeFormDesigner.vue`
- `LowCodeFormRenderer.vue`
- `formRegistry.ts`

这意味着工作流引擎本身不关心表单怎么渲染，它只保存任务配置和表单内容。

## 7. 审批处理链路

### 7.1 审批通过

入口：

- `/wf/execution/approve`
- `WfExecutionServiceImpl.approve(...)`
- 权限：`wf:execution:approve`

当前实现明确会做的事情包括：

1. 校验当前人是否可处理
2. 更新审批实例状态
3. 记录动作日志
4. 调用解释器 `onApprove`
5. 触发 `WorkflowNotificationService.notifyApproved(...)`
6. 由引擎推进到下一个节点或结束

### 7.2 驳回

入口：

- `/wf/execution/reject`
- `WfExecutionServiceImpl.reject(...)`
- 权限：`wf:execution:reject`

当前实现明确会：

1. 更新当前审批状态
2. 调用解释器 `onReject`
3. 调用 `notifyRejected(...)`
4. 根据驳回类型决定是退回上一步还是驳回结束

### 7.3 转交

入口：

- `/wf/execution/transfer`
- `WfExecutionServiceImpl.transfer(...)`
- 权限：`wf:execution:transfer`

转交本质上是把当前待办责任人变更为目标审批人，并记录动作日志。

### 7.4 加签

入口：

- `/wf/execution/addSign`
- `WfExecutionServiceImpl.addSign(...)`
- 权限：`wf:execution:addSign`

当前实现已经通过：

- `ApprovalActionHandlerFactory`
- `ApprovalActionHandlerType.ADD_SIGN`

接入动作处理器。

`handleAddSignAction(...)` 中会：

1. 从当前审批实例构造新的待办审批实例
2. 插入新的 `WfTaskApprovalInstance`
3. 创建待办任务
4. 记录动作日志

## 8. 批量处理与治理

### 8.1 批量处理

`WfExecutionServiceImpl` 已经提供：

- `batchApprove(...)`
- `batchTransfer(...)`
- `batchRemind(...)`

说明待办页不只是单条处理，也支持批量动作。

### 8.2 补偿与超时治理

当前实现还提供：

- `compensateExecution(...)`
- `retryTimeoutJobs(...)`

这部分属于治理层能力，面向：

- 超时任务重试
- 异常执行补偿
- 治理中心介入处理

前端对应页面入口：

- `src/views/workflow/governance/compensation/index.vue`

## 9. 引擎层职责

### 9.1 `WfEngineServiceImpl`

这个类是流程推进核心，不只是 CRUD。它的职责包括：

- 解析当前执行节点
- 解析审批人
- 计算下一节点
- 处理分支规则
- 创建审批实例
- 创建待办
- 推进流程状态

它是真正意义上的“审批引擎”。

### 9.2 审批人解析

源码里已经出现：

- `resolveApproverIdsByRule(...)`
- `parseJsonApproverIds(...)`

说明审批人既可能来自静态配置，也可能来自运行时规则。

## 10. 解释器机制

### 10.1 注册中心

解释器由：

- `ApprovalInterpreterRegistry`

统一管理。

它会按 beanName 缓存和获取：

- `IApprovalInterpreter`

### 10.2 生命周期

解释器主要参与这些节点：

- `onStart`
- `onApprove`
- `onReject`
- `onEnd`

这意味着业务系统可以在审批关键时刻挂接自己的业务逻辑，而不需要改工作流主干代码。

### 10.3 示例

当前模块内已有示例：

- `LeaveApprovalInterpreter`
- `ApprovalMessageNotifier`
- `EmptyApprovalInterpreter`

## 11. 通知机制

### 11.1 服务位置

- `WorkflowNotificationService`

### 11.2 通知类型

当前已经明确支持：

- `WF_PENDING`
- `WF_APPROVED`
- `WF_REJECTED`
- 完成通知

服务中已经存在：

- 待办通知
- 审批通过通知
- 驳回通知
- 完成通知

### 11.3 降级思路

通知优先尝试模板消息发送，失败后降级为直发模式，而不是整个审批动作失败。

这是比较重要的设计取向：

- 通知失败不应阻断审批主链路

## 12. 回调机制

### 12.1 入口

- `WfCallbackController`
- `WfCallbackServiceImpl`

### 12.2 能力

当前已支持：

- `registerCallback(...)`
- `unregisterCallback(...)`
- `triggerCallback(...)`

说明审批完成后，工作流模块可以主动通知外部业务系统。

`WfCallbackServiceImpl` 当前维护了一个内存注册表：

- `callbackRegistry`

它会按 `taskCode` 找到对应回调 URL 或 callback bean，然后触发回调。

## 14. 常见排查

### 14.1 创建流程后在列表里看不到

优先检查：

1. 是否只保存了草稿，未发布
2. 当前任务状态是否启用
3. 当前用户是否有 `wf:taskConfig:view`

### 14.2 发起审批后没有生成待办

优先检查：

1. 是否存在已发布版本
2. 首节点审批人规则是否能解析出审批人
3. `WfEngineServiceImpl` 是否成功创建审批实例和我的待办

### 14.3 审批按钮不显示

优先检查：

1. 前端 `v-permission` 权限码
2. 当前角色是否有 `wf:execution:approve/reject/transfer/addSign`
3. 当前用户是否真的属于该待办审批人

### 14.4 审批通过了但流程没有推进

优先检查：

1. 当前节点规则是否配置完整
2. 下一个节点是否有可解析审批人
3. `WfEngineServiceImpl` 推进逻辑是否命中分支条件

### 14.5 通知没有发出

优先检查：

1. `WorkflowNotificationService` 是否被调用
2. 模板消息是否可用
3. 是否已经走了降级直发

### 14.6 业务回调没收到

优先检查：

1. 回调是否已注册
2. `taskCode` 是否匹配
3. `WfCallbackServiceImpl.triggerCallback(...)` 是否执行
4. 回调 URL 是否可达

### 14.7 超时或异常任务未恢复

优先检查：

1. 补偿中心是否已经发起补偿
2. `retryTimeoutJobs(...)` 是否执行
3. 超时动作处理器是否生效

## 15. 开发建议

- 配置编辑始终走草稿 -> 发布，不要直接改线上流程定义。
- 业务回调、通知、解释器都应视为“附加能力”，不要反过来绑死主流程。
- 节点审批人规则一定要有兜底，否则最容易出现“流程发起成功但没人审批”。
- 页面按钮权限和后端接口权限要严格一一对应。
