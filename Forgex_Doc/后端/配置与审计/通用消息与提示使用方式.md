# 通用消息与提示使用方式

> 分类：后端 / 配置与审计
> 版本：**V0.6.5**
> 更新时间：**2026-04-17**

本文重点讲“代码如何正确返回消息”和“前端页面如何正确消费消息”。

## 代码如何使用

### 1. 成功提示

推荐写法：

```java
return R.ok(CommonPrompt.SAVE_SUCCESS);
return R.okWithArgs(CommonPrompt.UPDATE_SUCCESS, "用户信息");
return R.okWithArgs(CommonPrompt.DELETE_SUCCESS, "角色");
```

如果需要返回数据：

```java
return R.okWithArgsAndData(CommonPrompt.SAVE_SUCCESS, data, "租户配置");
```

### 2. 失败提示

推荐写法：

```java
return R.fail(CommonPrompt.OPERATION_FAILED);
return R.fail(StatusCode.NOT_FOUND, CommonPrompt.NOT_FOUND);
throw new I18nBusinessException(CommonPrompt.NOT_FOUND);
throw new I18nBusinessException(CommonPrompt.ACCOUNT_LOCKED, lockMinutes);
```

### 3. 参数化消息

`CommonPrompt` 中大量模板带占位符，例如：

- `保存成功：{0}`
- `账号已被锁定，请{0}分钟后再试`

这类提示不要拼字符串，直接传参数：

```java
return R.okWithArgs(CommonPrompt.SAVE_SUCCESS, "菜单配置");
throw new I18nBusinessException(CommonPrompt.ACCOUNT_LOCKED, 30);
```

### 4. 新业务模块如何扩展

如果消息明显属于单独业务域，建议新建自己的 Prompt 枚举并实现 `I18nPrompt`，不要把所有消息都塞到 `CommonPrompt`。

适合单独建枚举的场景：

- 工作流节点提示
- 报表发布提示
- 集成平台回调提示
- 特定业务单据流转提示

## 页面如何使用

### 前端统一入口

前端请求层真实入口在：

- `Forgex_MOM/Forgex_Fronted/src/api/http.ts`

当前前端已经支持：

- 自动附带 `X-Lang`
- 自动附带 `Accept-Language`
- 成功请求自动显示后端 `message`
- 失败请求自动显示后端 `message`
- 静默请求 `silentHttp`
- `showSuccessMessage` 控制

### 页面推荐写法

页面里尽量只发请求，不再手写重复提示：

```ts
await http.post('/sys/user/save', form)
```

而不是：

```ts
await http.post('/sys/user/save', form)
message.success('保存成功')
```

因为后端已经可以统一返回标准提示，前端重复弹一次会导致体验不一致。

### 什么情况下用 `silentHttp`

适合：

- 轮询
- 未读数刷新
- 草稿恢复
- 页面初始化静默查询

不适合：

- 保存
- 删除
- 提交
- 授权

这类有明确用户操作反馈的请求，应该保留后端消息并让前端自动消费。

## 推荐接入步骤

### 新增一个写操作接口

1. 先判断是否可复用 `CommonPrompt`
2. 成功时返回 `R.ok(prompt)` 或 `R.okWithArgs(prompt, ...)`
3. 失败时返回 `R.fail(...)` 或抛 `I18nBusinessException`
4. 不要直接写中文字符串
5. 前端页面只发请求，不重复弹消息

### 新增一个业务模块提示体系

1. 新建模块 Prompt 枚举
2. 定义 `module + promptCode + defaultTemplate`
3. 在 `fx_i18n_message` 配多语言文案
4. 页面联调时验证不同语言是否生效

## 常见误区

### 误区 1：后端和前端各弹一次

这样会造成双提示。推荐以后端为准，前端统一请求层自动消费。

### 误区 2：所有失败都写成一个“操作失败”

虽然可以兜底，但不利于排查。参数错误、权限问题、资源不存在、数据库异常最好用不同提示码。

### 误区 3：提示只写中文

这样后续接国际化或租户差异化时会非常痛苦。应该始终优先走 Prompt 体系。

## 联调排查

| 现象 | 排查方式 |
|---|---|
| 页面没有成功提示 | 后端是否返回了 message，前端是否用了 `silentHttp` |
| 后端返回了提示码但页面是空白 | `RMessageI18nAdvice` 是否执行、语言是否正常 |
| 页面提示重复 | 是否同时用了后端自动提示和页面手工 `message.success` |
| 多语言不生效 | `X-Lang`、`Accept-Language`、`fx_i18n_message` 是否匹配 |
