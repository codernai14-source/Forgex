# 通用消息与提示实现逻辑

> 分类：后端 / 配置与审计  
> 版本：**V0.6.0**  
> 更新时间：**2026-04-17**

本文说明 Forgex 后端“通用消息与提示”在代码里的真实实现方式。这里的“消息”不是简单字符串，而是一条从 Prompt 枚举、统一返回、异常体系、国际化解析到前端自动提示的完整链路。

## 代码入口

| 能力 | 代码位置 | 说明 |
|---|---|---|
| 通用提示枚举 | `Forgex_MOM/Forgex_Backend/Forgex_Common/src/main/java/com/forgex/common/i18n/CommonPrompt.java` | 平台级提示码与默认模板 |
| 提示抽象接口 | `Forgex_MOM/Forgex_Backend/Forgex_Common/src/main/java/com/forgex/common/i18n/I18nPrompt.java` | 统一定义 `module/code/template` |
| 统一返回模型 | `Forgex_MOM/Forgex_Backend/Forgex_Common/src/main/java/com/forgex/common/web/R.java` | 承载 `code/message/data/i18n` |
| 国际化业务异常 | `Forgex_MOM/Forgex_Backend/Forgex_Common/src/main/java/com/forgex/common/exception/I18nBusinessException.java` | 抛出带提示码与参数的异常 |
| 普通业务异常 | `Forgex_MOM/Forgex_Backend/Forgex_Common/src/main/java/com/forgex/common/exception/BusinessException.java` | 抛出普通业务异常 |
| 全局异常处理 | `Forgex_MOM/Forgex_Backend/Forgex_Common/src/main/java/com/forgex/common/web/GlobalExceptionHandler.java` | 异常转 `R` |
| 出参翻译拦截器 | `Forgex_MOM/Forgex_Backend/Forgex_Common/src/main/java/com/forgex/common/web/RMessageI18nAdvice.java` | 返回前解析最终文案 |
| 语言上下文 | `Forgex_MOM/Forgex_Backend/Forgex_Common/src/main/java/com/forgex/common/i18n/LangContext.java` | 维护 `X-Lang` 与默认语言 |
| 模板配置服务 | `Forgex_MOM/Forgex_Backend/Forgex_Common/src/main/java/com/forgex/common/service/template/TemplateConfigServiceImpl.java` | 查询租户/公共模板覆盖 |
| 前端提示托管 | `Forgex_MOM/Forgex_Fronted/src/api/http.ts` | 自动显示成功/失败消息 |

## 总体链路

```text
Controller / Service
  -> 返回 R.ok(prompt) / R.fail(prompt)
  -> 或抛出 I18nBusinessException / BusinessException
  -> GlobalExceptionHandler 转为 R
  -> RMessageI18nAdvice 解析 messageCode / i18n
  -> I18nMessageService / 模板配置服务解析最终文案
  -> 返回前端
  -> 前端 http.ts 自动提示 success / error
```

## Prompt 枚举体系

`CommonPrompt` 是平台级消息中心，当前已内置大量通用提示，例如：

- 保存、修改、删除、上传、导入、导出成功
- 未登录、无权限、参数错误
- 文件上传失败、数据不存在
- 账号锁定、验证码错误、授权失效

### 结构特点

每个枚举项都包含：

- `module`
- `promptCode`
- `defaultTemplate`

例如：

- `SAVE_SUCCESS("common", "SAVE_SUCCESS", "保存成功：{0}")`
- `NOT_LOGIN("common", "NOT_LOGIN", "未登录或登录过期")`

这意味着消息的核心不是一段中文，而是“提示码 + 默认模板”。

## `R<T>` 如何承载消息

`R<T>` 除了常规的：

- `code`
- `message`
- `data`

还包含两个关键内部字段：

- `messageCode`
- `i18n`

### 实现目的

- `messageCode`
  保存本次返回绑定的 `I18nPrompt`
- `i18n`
  保存跨服务、网关或前端可能需要的二次翻译元信息

### 参数传递方式

`R.okWithArgs()`、`R.failWithArgs()` 会把参数：

1. 序列化到 `message`
2. 同步写入 `i18n.args`

因此返回对象在翻译前就已经携带了足够的上下文。

## 异常到消息的转换

`GlobalExceptionHandler` 负责把异常统一收口成 `R`。

### 当前已处理的典型异常

- `NotLoginException`
- `NotRoleException`
- `I18nBusinessException`
- `BusinessException`
- `MethodArgumentNotValidException`
- `BindException`
- `HttpMessageNotReadableException`
- `CannotGetJdbcConnectionException`
- `DataAccessException`
- 其他 `Exception`

### 关键行为

- 未登录返回 `StatusCode.NOT_LOGIN + CommonPrompt.NOT_LOGIN`
- 无权限返回 `StatusCode.UNAUTHORIZED + CommonPrompt.NO_PERMISSION`
- 参数错误统一归并到 `CommonPrompt.BAD_REQUEST`
- 数据库连接与数据访问异常也映射到统一提示码

所以用户最终看到的提示并不是异常类名，而是被映射过的标准消息体系。

## 出参翻译实现

`RMessageI18nAdvice` 会拦截所有 `R<?>` 返回体。

### 处理逻辑

1. 如果 `messageCode` 不为空
   说明本次返回明确绑定了 Prompt
2. 解析 `message` 中的占位符参数
3. 调用 `I18nMessageService.resolve(prompt, args)`
4. 把翻译结果写回 `message`
5. 清空 `messageCode` 和 `i18n`

### 另一条分支

如果没有 `messageCode`，但有 `i18n` 元信息且 `message` 为空，则会用：

- `module`
- `code`
- `args`

重新解析文案。

这就是为什么系统既能支持本地服务内翻译，也能支持跨服务/网关补翻译。

## 语言上下文实现

`LangContext` 用 `ThreadLocal` 保存当前请求语言。

### 语言解析顺序

1. `X-Lang`
2. `Accept-Language`
3. 默认值 `zh-CN`

### 标准化规则

例如：

- `zh`、`zh-cn` -> `zh-CN`
- `zh-tw`、`zh-hk` -> `zh-TW`
- `en`、`en-us` -> `en-US`

这保证了前端传入不同格式时，后端仍能映射到统一语言编码。

## 租户级模板覆盖

除了 `CommonPrompt + fx_i18n_message` 这条通用翻译链，系统还支持：

- `sys_response_message_template`

由 `TemplateConfigServiceImpl` 负责查询与同步。

### 覆盖优先级

```text
当前租户模板
  -> 公共模板 tenant_id = 0
  -> 默认模板 / 提示码
```

这意味着同一提示在不同租户里可以显示不同文案，而不需要发版改代码。

## 与前端提示的联动

前端 `src/api/http.ts` 已确认有这些行为：

- 请求自动带 `Accept-Language`
- 请求自动带 `X-Lang`
- 成功写操作可自动弹后端 `message`
- 错误请求统一弹后端 `message`
- 支持 `silentHttp`
- 支持 `showSuccessMessage`
- 支持消息防重复抖动

所以后端这套“通用消息与提示”不是孤立设计，而是已经被前端统一请求层消费。

## 为什么不要硬编码字符串

如果 Controller / Service 到处直接写：

```java
return R.ok("保存成功");
```

会直接失去：

- 多语言能力
- 租户模板覆盖能力
- 前后端统一提示规范
- 提示码级别排查能力

这也是为什么该模块的实现重点是“提示码驱动”，不是“字符串驱动”。

## 常见问题排查

### 后端返回了 `R` 但前端没弹提示

先查：

1. 是否返回了有效 `message`
2. 前端是否走 `http` 还是 `silentHttp`
3. 是否被重复提示抑制机制挡住了

### 返回消息没翻译

优先查：

1. `messageCode` 是否有值
2. `LangContext` 当前语言是什么
3. `fx_i18n_message` 是否有对应记录
4. 是否被租户模板覆盖

### 异常提示混乱

重点看异常是否绕过了：

- `I18nBusinessException`
- `BusinessException`
- `GlobalExceptionHandler`
