# 统一返回与国际化实现逻辑

> 分类：后端 / 配置与审计  
> 版本：**V0.6.0**  
> 更新时间：**2026-04-17**

本文说明 `R<T>`、异常体系、语言上下文、多语言消息解析与响应翻译在 Forgex 后端中的真实实现方式。

## 统一响应链路

```text
Controller / Service
  -> return R.ok(...) / R.fail(...)
  -> 或抛出 I18nBusinessException / BusinessException
  -> GlobalExceptionHandler 兜底
  -> RMessageI18nAdvice 拦截
  -> I18nMessageService.resolve(...)
  -> LangContext 决定语言
  -> 返回给前端最终 message
```

## `R<T>` 结构

`R<T>` 当前不只是一个简单三元组，而是包含：

- `code`
- `message`
- `data`
- `messageCode`
- `i18n`

### 设计意义

- `code`
  对外暴露业务状态
- `message`
  最终显示给前端的文案
- `messageCode`
  内部暂存 `I18nPrompt`
- `i18n`
  保存 `module/code/args` 元信息，便于跨服务或网关二次翻译

## 异常体系

当前异常体系至少包含：

- `BusinessException`
- `I18nBusinessException`
- `GlobalExceptionHandler`

### 实现特点

- 国际化业务异常优先保留提示码与参数
- 普通业务异常统一折算为 `R.fail(...)`
- 参数校验异常统一映射到 `BAD_REQUEST`
- 登录与权限异常映射到标准状态码

## 语言上下文实现

`LangContext` 基于 `ThreadLocal` 存储当前语言。

### 语言来源顺序

1. `X-Lang`
2. `Accept-Language`
3. 默认 `zh-CN`

### 标准化

会把：

- `zh`
- `zh-cn`
- `zh-tw`
- `en`
- `en-us`

这类输入标准化为统一格式，避免数据库和配置层因语言编码不一致导致匹配失败。

## 出参国际化实现

`RMessageI18nAdvice` 是响应国际化的核心拦截器。

### 处理方式

如果 `R.messageCode` 不为空：

1. 解析 `message` 中的参数
2. 调用 `I18nMessageService.resolve(prompt, args)`
3. 把翻译后的文本写回 `message`

如果没有 `messageCode`，但 `i18n` 不为空且 `message` 为空：

1. 使用 `module + code + args`
2. 继续走国际化解析

最后会清空 `messageCode` 与 `i18n`，只把最终文本返回给前端。

## 存量硬编码提示兜底（历史兼容，禁止新增）

`I18nMessageServiceImpl` 仍然优先读取 `fx_i18n_message` 中的正式多语言文案。只有在数据库记录缺失、语言文本缺失或需要使用 Prompt 默认模板时，才会调用 `LegacyMessageTranslator` 做历史兼容翻译。

`LegacyMessageTranslator` 不是新的国际化数据源。新增业务代码禁止直接调用、封装调用或继续向该类新增词条；任何缺失翻译都应回到 `I18nPrompt` / 模块 Prompt 枚举与 `fx_i18n_message` 配置中处理。

`GlobalExceptionHandler` 对参数校验异常也会先处理 DTO 注解里的默认提示，再进入 `CommonPrompt.BAD_REQUEST` 模板。这样类似 `@NotBlank(message = "模板名称不能为空")` 的存量注解，在非中文语言下不会直接把中文原文返回给前端。该能力只用于消化历史注解；新增 Bean Validation / DTO 校验提示不得继续写硬编码中文，应转换为 Prompt 或统一异常提示，并通过数据库维护多语言文案。

这层是兜底机制，不替代正式接入流程。新业务必须定义模块 Prompt 枚举，并在 `fx_i18n_message` 中维护完整多语言文案；如果触发兜底，应视为配置缺口或存量迁移项，而不是继续写死代码的理由。

## 多语言数据来源

### 1. `fx_i18n_message`

主用于：

- `module + prompt_code` 维度的多语言消息

### 2. `fx_i18n_language_type`

主用于：

- 维护系统支持语言
- 默认语言
- 启用状态

### 3. `sys_response_message_template`

主用于：

- 租户级接口返回模板覆盖

### 覆盖优先级

```text
租户模板
  -> 公共模板 tenant_id = 0
  -> Prompt 默认模板
```

## 网关与跨服务补翻译

`R.i18n` 的存在，说明系统已经考虑了“返回对象跨服务传递后，原始 `messageCode` 可能不可直接用”的情况。

这类场景下，网关或其他服务仍可通过：

- `module`
- `code`
- `args`

重新定位并翻译文案，避免前端收到半成品提示码。

## 与前端的联动

前端 `src/api/http.ts` 已验证有这些行为：

- 自动写入 `X-Lang`
- 自动写入 `Accept-Language`
- 成功写操作自动显示后端 `message`
- 失败统一显示后端 `message`
- `silentHttp` 支持静默模式

因此后端统一返回和国际化不是“为后端自己设计”，而是前端交互体验的基础设施。

## 常见问题排查

### 前端拿到的还是默认中文

优先检查：

1. 请求头是否带 `X-Lang`
2. `LangContext` 是否解析正确
3. `fx_i18n_message` 是否有对应语言记录

### 异常信息没走国际化

检查是否直接抛了普通 `Exception`，绕过了 `I18nBusinessException` 或标准 `R.fail(prompt)`。

### 同一个提示不同租户文案不一致

这是可能的，因为 `sys_response_message_template` 支持租户覆盖。排查时要先看是不是覆盖生效，而不是先怀疑代码错误。
