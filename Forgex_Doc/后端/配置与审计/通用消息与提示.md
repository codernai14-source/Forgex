# 通用消息与提示

> 分类：后端 / 配置与审计  
> 版本：**V0.5.0**  
> 更新时间：**2026-04-13**

## 一、概述

Forgex 后端的“通用消息”不是简单的字符串常量，而是一整条可配置链路：

```text
Controller / Service
  → CommonPrompt / 模块 Prompt 枚举
  → R.ok(...) / R.fail(...) / I18nBusinessException
  → RMessageI18nAdvice
  → I18nMessageService
  → fx_i18n_message / sys_response_message_template
  → 前端 http.ts 自动成功 / 失败提示
```

这套设计解决了 4 个问题：

1. **后端不再到处硬编码中文提示**；
2. **同一条提示可以做多语言翻译**；
3. **租户可以按需覆盖接口返回模板**；
4. **前端成功/失败提示可统一托管，不必每页重复写 `message.success`。**

---

## 二、核心类与表

| 类型 | 名称 | 路径 / 表 | 作用 |
|---|---|---|---|
| 提示枚举 | `CommonPrompt` | `com.forgex.common.i18n.CommonPrompt` | 平台级通用提示码 |
| 提示接口 | `I18nPrompt` | `com.forgex.common.i18n.I18nPrompt` | 统一定义 `module/code/template` |
| 统一返回 | `R<T>` | `com.forgex.common.web.R` | 承载 `code/message/data` 与 i18n 元信息 |
| 响应翻译 | `RMessageI18nAdvice` | `com.forgex.common.web.RMessageI18nAdvice` | 在返回前翻译 `message` |
| 文案解析 | `I18nMessageService` | `com.forgex.common.service.i18n.*` | 根据语言解析消息模板 |
| 响应模板服务 | `TemplateConfigServiceImpl` | `com.forgex.common.service.template.TemplateConfigServiceImpl` | 租户/公共模板查询与同步 |
| 多语言消息表 | `fx_i18n_message` | 数据库表 | 存储 `module + prompt_code` 多语言文案 |
| 接口返回模板表 | `sys_response_message_template` | 数据库表 | 存储租户级返回消息模板 |

## 三、`CommonPrompt`：系统级通用消息枚举

`CommonPrompt` 适合承载所有模块都会复用的提示，如：

- 保存成功 / 修改成功 / 删除成功
- 未登录 / 无权限 / 参数错误
- 文件上传失败 / 数据不存在
- 账号已锁定 / 邀请码无效 / 注册成功

### 3.1 典型枚举值

| 枚举 | 默认模板 | 适用场景 |
|---|---|---|
| `SAVE_SUCCESS` | `保存成功：{0}` | 通用保存类接口 |
| `UPDATE_SUCCESS` | `修改成功：{0}` | 更新操作 |
| `DELETE_SUCCESS` | `删除成功：{0}` | 删除操作 |
| `NOT_LOGIN` | `未登录或登录过期` | 统一登录失效提示 |
| `NO_PERMISSION` | `无权限` | 权限拦截 |
| `BAD_REQUEST` | `请求参数错误：{0}` | 参数校验失败 |
| `NOT_FOUND` | `数据不存在` | 详情不存在 |
| `OPERATION_FAILED` | `操作失败` | 默认兜底失败 |
| `REGISTER_SUCCESS` | `注册成功` | 注册完成 |
| `ACCOUNT_LOCKED` | `账号已被锁定，请{0}分钟后再试` | 登录安全控制 |

### 3.2 什么时候用 `CommonPrompt`

优先用于：

- 跨模块共用提示；
- CRUD 类标准消息；
- 鉴权、登录、参数、上传等平台级提示；
- 需要前端自动弹窗但不想在页面硬编码文案的场景。

### 3.3 什么时候新建模块 Prompt

如果消息**明显只属于某个业务域**，建议单独建枚举，例如：

- 工作流审批状态提示；
- 报表模板发布提示；
- 某业务单据生命周期提示。

建议命名方式：

```java
public enum WorkflowPrompt implements I18nPrompt {
    APPROVAL_SUBMITTED("APPROVAL_SUBMITTED", "审批单已提交"),
    APPROVAL_REJECTED("APPROVAL_REJECTED", "审批单已驳回");

    @Override
    public String getModule() {
        return "workflow";
    }
}
```

## 四、统一返回如何承载消息

### 4.1 推荐写法

| 场景 | 推荐写法 |
|---|---|
| 仅返回数据 | `R.ok(data)` |
| 写操作成功并弹消息 | `R.ok(CommonPrompt.SAVE_SUCCESS)` |
| 带参数模板消息 | `R.okWithArgs(CommonPrompt.UPDATE_SUCCESS, "用户")` |
| 普通失败 | `R.fail(CommonPrompt.OPERATION_FAILED)` |
| 业务异常 | `throw new I18nBusinessException(CommonPrompt.NOT_FOUND)` |

### 4.2 参数化消息

`CommonPrompt` 中很多模板带 `{0}` 占位符，例如：

- `保存成功：{0}`
- `账号已被锁定，请{0}分钟后再试`

此时应使用：

```java
return R.okWithArgs(CommonPrompt.SAVE_SUCCESS, "租户配置");
throw new I18nBusinessException(CommonPrompt.ACCOUNT_LOCKED, lockMinutes);
```

### 4.3 不推荐写法

以下写法建议尽量避免：

- `return R.ok("保存成功")`
- `throw new BusinessException("删除失败")`
- Controller 中大量硬编码中文字符串

原因：

1. 无法统一多语言；
2. 难以被租户模板覆盖；
3. 前端难以形成统一提示体验；
4. 后续审计、批量排查成本高。

## 五、响应翻译链路

`RMessageI18nAdvice` 会拦截所有 `R<?>` 返回体，并按以下规则处理：

1. 如果 `R.messageCode` 不为空，说明当前返回使用了 Prompt 枚举；
2. 从 `R.message` 中解析模板参数；
3. 调用 `I18nMessageService.resolve(prompt, args)`；
4. 按当前请求语言生成最终 `message`；
5. 清空内部 `messageCode/i18n` 元信息，只把最终文本返回给前端。

### 5.1 请求语言来源

最终语言通常来自：

1. 请求头 `X-Lang`
2. 请求头 `Accept-Language`
3. 默认语言 `zh-CN`

前端 `src/api/http.ts` 已自动写入这两个请求头。

## 六、`fx_i18n_message`：多语言提示文案

这是 Prompt 枚举的主翻译表。

### 6.1 核心字段

| 字段 | 说明 |
|---|---|
| `module` | 模块标识，如 `common`、`auth`、`workflow` |
| `prompt_code` | 提示码，如 `SAVE_SUCCESS` |
| `message_json` | 多语言 JSON 文案 |

### 6.2 示例

```json
{
  "zh-CN": "保存成功：{0}",
  "en-US": "Saved successfully: {0}",
  "zh-TW": "儲存成功：{0}"
}
```

### 6.3 解析顺序

`I18nMessageService` 通常按以下顺序返回文案：

1. 查 `module + prompt_code` 对应记录；
2. 取当前语言对应的文本；
3. 找不到时按语言前缀兜底；
4. 仍找不到则回退到枚举中的 `defaultTemplate`；
5. 使用 `MessageFormat` 填充参数。

## 七、`sys_response_message_template`：租户级接口返回模板

除了 Prompt 枚举 + 国际化消息表外，Forgex 还支持更细粒度的**接口返回消息模板**。

### 7.1 适用场景

适合以下需求：

- 公共平台提供默认提示，但租户希望覆盖文案；
- 同一接口在不同租户中要展示不同描述；
- 希望通过数据库改文案，而不是改代码发版。

### 7.2 查询优先级

`TemplateConfigServiceImpl` 的核心规则是：

```text
当前租户模板 → 公共模板（tenant_id = 0）→ 模板编码本身 / 默认文案
```

### 7.3 典型字段

| 字段 | 说明 |
|---|---|
| `template_code` | 模板编码，通常与提示码关联 |
| `template_content` | 模板正文 |
| `lang` | 语言代码 |
| `tenant_id` | 租户 ID |
| `enabled` | 是否启用 |

### 7.4 模板同步

服务支持把公共模板同步到租户：

- 公共模板为源租户：`tenant_id = 0`
- 目标租户可按需覆盖
- 适合初始化新租户时同步一份默认提示体系

## 八、前后端联动方式

后端的消息设计直接影响前端体验。

### 8.1 前端默认行为

`src/api/http.ts` 已支持：

- 成功请求自动显示后端 `message`
- 失败请求统一显示后端 `message`
- 登录失效统一处理
- 消息防重复显示

### 8.2 最佳配合方式

后端写操作建议：

```java
return R.okWithArgs(CommonPrompt.UPDATE_SUCCESS, "部门信息");
```

前端直接：

```ts
await http.post('/sys/department/update', form)
```

这样前端页面可以不再手写：

```ts
message.success('修改成功')
```

### 8.3 不建议的做法

- 后端成功返回无 `message`，前端页面自己猜测提示；
- 后端返回中文，前端再次手动翻译；
- 前后端都各自弹一次成功消息。

## 九、接入建议

### 9.1 新增接口时

建议优先检查：

- 是否可以复用 `CommonPrompt`
- 是否需要新增业务模块 Prompt
- 是否属于租户可配置提示
- 是否要支持多语言参数占位符

### 9.2 设计建议

1. **共性消息进 `CommonPrompt`**；
2. **业务消息进模块 Prompt**；
3. **面向租户差异的文案放模板表**；
4. **Controller 尽量只返回 `R`，不要直接返回裸字符串**；
5. **写操作尽量返回明确 message，便于前端自动托管**。

---

## 十、排查清单

当出现“前端没提示 / 提示没翻译 / 提示内容不对”时，可按下面顺序排查：

1. Controller 是否返回了 `R.ok(prompt)` / `R.fail(prompt)`；
2. Prompt 的 `module + promptCode` 是否正确；
3. `fx_i18n_message` 中是否有对应多语言记录；
4. 当前请求头是否带了 `X-Lang`；
5. 是否被 `sys_response_message_template` 覆盖；
6. 前端是否使用了 `silentHttp` 或显式关闭提示；
7. 是否因重复消息抑制机制而未重复弹出。
